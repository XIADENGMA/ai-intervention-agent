## How to read this report

Each finding is scored along **two axes**:

1. **Vendor severity** — what the upstream advisory says. Untouched.
2. **Effective risk for `ai-intervention-agent`** — does our code actually
   reach the vulnerable path, and does our deployment posture (loopback by
   default, no public file uploads, no OAuth flows) neutralise it?

The **action column** is what a future maintainer should actually do. In
most cases that is "wait for Dependabot", because every advisory has an
upstream fix already shipped and our `dependabot.yml` is configured to open
weekly PRs.

## Remediation actually applied (2026-05-04)

`fastmcp 3.1.1 → 3.2.4` was pulled in alongside the Starlette /
python-multipart / cryptography / authlib / werkzeug / markdown /
pygments / python-dotenv chain. The release notes show two
fastmcp-side breaking changes; both were validated as not affecting us:

1. **3.2.0** — `Route app tool calls via ___-prefixed names` only
   affects users of `FastMCPApp`. We use plain `FastMCP`.
2. **3.2.4** — `Scope tasks to authorization context, not session` only
   affects users of background tasks behind auth. We don't have any.

Three transitive packages received **major-version bumps**:

| Package        | Before | After  | Risk for us                                                                                                                   |
| -------------- | ------ | ------ | ----------------------------------------------------------------------------------------------------------------------------- |
| `cffi`         | 1.17.1 | 2.0.0  | None — used through cryptography only.                                                                                        |
| `cryptography` | 45.0.5 | 47.0.0 | None — we don't directly call cryptography APIs.                                                                              |
| `starlette`    | 0.46.2 | 1.0.0  | None — fastmcp owns the Starlette adapter; the full test suite (2244 tests) and the red-team guard pass against the 1.0 line. |

One repo-side fix was needed: `scripts/test_mcp_client.py` directly
imported a private fastmcp helper (`fastmcp.tools.tool._convert_to_content`)
that moved to `fastmcp.tools.base` in 3.2. The script now does a
`try/except ImportError` fallback so it stays compatible with both
pre-3.2 and 3.2+ installations.

The exact recipe that was run:

```bash
cp uv.lock uv.lock.backup   # rollback safety
uv lock --upgrade-package fastmcp \
        --upgrade-package markdown \
        --upgrade-package pygments \
        --upgrade-package python-dotenv \
        --upgrade-package python-multipart \
        --upgrade-package starlette \
        --upgrade-package cryptography \
        --upgrade-package authlib \
        --upgrade-package werkzeug
uv sync
uv run python scripts/ci_gate.py    # 2244 passed, 0 warning, 0 error
uv run pip-audit                    # 17 → 1 (only pytest dev-only remains)
```

The remaining `pytest 8.4.0` → `9.0.3` jump is **deferred** because:

- pytest is dev-only — not in the runtime wheel, no end-user exposure.
- pytest 8 → 9 is a major version bump with documented behavioural
  changes (deprecated APIs removed). Conflating it with the runtime
  security wave would muddy the bisect surface if anything broke.
- Dependabot is configured to keep raising it; let the next round take
  it under a separate PR.
