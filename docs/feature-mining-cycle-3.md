## 0. Methodology (cycle-2 hardened — quote-for-quote)

For every candidate feature in §3:

```bash
# 1) filesystem evidence (what's currently in tree)
rg '<feature-keyword>' src/ packages/

# 2) history evidence (what was already shipped / removed / discussed)
git log --grep '<feature-keyword>' --since='2 years ago' --oneline
git log --grep '<keyword-alt>' --since='2 years ago' --oneline
```

Both outputs must be **pasted verbatim** in each §3 candidate
section, even if empty. Empty grep = positive evidence that
nothing exists; presence of matches = strong hint that the
feature is shipped / partially done / consciously rejected.

**Why mandatory**: cycle-2 had 3 survey misses
(§3.3 input-height-memory R137, §3.4 project_directory R partial,
§3.1 Session Export R125/R125c/R135). All were caught **after**
the survey, during implementation grep. New §0 forces the grep
**before** classifying as "not started".

### 1.3 Anti-pattern surfaced — CHANGELOG drift

The most interesting cycle-3 Track A finding is **NOT a
feature**. It's an **anti-pattern**: competitor's
`CHANGELOG.en.md` HEAD only documents up to **v2.5.6**, while
GitHub Releases page goes through **v2.6.0**, and PR #207
references **v2.6.1**. The release notes live in 3 separate
places that drift apart:

1. `RELEASE_NOTES/CHANGELOG.en.md` (markdown, mostly hand-
   maintained, last entry v2.5.6)
2. GitHub Releases UI (auto-generated by github-actions[bot],
   has v2.6.0)
3. PR titles / commits (has v2.6.1 in chore release bump)

**Lesson for us**:
- Our `CHANGELOG.md` should auto-derive from a single source of
  truth (commits + releases), not be hand-maintained in
  parallel.
- Add a CI check that asserts `CHANGELOG.md` HEAD == latest
  release tag.
- **Routed to**: cycle-3 forward log (§6) as "low priority but
  defensible" follow-up. Not auto-shipping; ROI is "avoids
  doc-drift embarrassment" which is non-quantifiable.

## 2. Track B — interaction-pattern survey (gemini-cli + claude-code)

Track B targets **idioms**, not features:

- Q: How do these CLIs handle the "AI claims completion but
  user wants follow-up" problem? (our `interactive_feedback`
  niche)
- Q: How do they surface task state to the user during long
  loops? (compare with our task tab + countdown)
- Q: How do they format AI-to-user pre-defined option prompts?

### 2.1 `gemini-cli` (Google) — `ask_user` tool

Source: `docs/tools/ask-user.md` (main branch as of 2026-06-04).

gemini-cli ships **`ask_user`** — a structured tool that lets
the model pause execution and ask 1-4 questions of the user via
an interactive dialog. **Direct counterpart** to our MCP
`interactive_feedback`. Schema:

```text
questions: array of 1-4 items, each:
  question:    string (full question text, required)
  header:      string (≤16 chars, displayed as chip/tag, required)
  type:        'choice' | 'text' | 'yesno'   (default: choice)
  options:     2-4 items (required for type='choice')
    label:       string (1-5 words, display text)
    description: string (brief explanation)
  multiSelect: bool (choice only; auto-adds "All the above")
  placeholder: string (input hint for type='text')

output: JSON {"answers": {"0": "Option A", "1": "Some text"}}
```

**Idioms worth borrowing into AI-intervention-agent**:

1. **`header` chip / tag** (≤16 chars, displayed as small
   prefix label). For our multi-task layout, a header chip
   like "Auth" / "DB" / "Layout" would give users instant
   visual context without reading the full question. **Low
   LoC, high UX**.
2. **`yesno` question type**. We currently force agents to
   construct binary questions as 2-option `choice`. A
   dedicated `yesno` semantic would simplify UX (rendered as
   bigger buttons, not radio).  **Low LoC, semantic clarity**.
3. **`placeholder` for text input**. Trivial CSS / HTML
   change; **free win** for text feedback discoverability.

**Anti-idioms (consciously NOT borrow)**:

1. **Multi-question per call (1-4)**. Our design philosophy is
   one decision point per request — mixing 4 questions in one
   prompt creates cognitive load and forces agents to
   pre-cluster questions that may actually need separate
   reasoning. Our explicit "ask in sequence" gives the agent
   more flexibility.
2. **`multiSelect` + "All the above"**. Complexity-to-benefit
   ratio is poor; multi-select rare in our use case (~99% of
   feedback is single decision).

### 2.2 `claude-code` (Anthropic) — Hooks system

Source: `code.claude.com/docs/en/hooks` (as of 2026-06-04).

claude-code's hooks are a **fundamentally different design
space**: lifecycle control points, not user interactions.
22 events × 4 handler types (`command` / `http` / `prompt` /
`agent`). Examples: `SessionStart`, `UserPromptSubmit`,
`PreToolUse`, `PostToolUse`, `Stop`.

**Not directly comparable to `interactive_feedback`**, but two
adjacent insights:

1. **`UserPromptSubmit` event with `additionalContext` return**:
   hook can inject extra context into the conversation at the
   exact prompt-submit moment. We have a parallel concept in
   our `feedback_suffix` config (appended to feedback before
   sending to agent). Our impl is static-config; claude-code's
   is dynamic per-event. **Could borrow**: make our suffix
   accept a `${task_id}` / `${timestamp}` template placeholder.
2. **`Stop` with `decision: block`**: hook can prevent the
   model from claiming completion if a checklist isn't met.
   **Could borrow**: a "completion gate" config that requires
   the agent to call `interactive_feedback` before declaring
   done. We already document this in CLAUDE.md but it's not
   enforceable. **Defer**: needs MCP-level mechanism we don't
   own (would have to be agent-side enforcement).

**Anti-idiom**: claude-code's 22 events × 4 handler types is
massive surface area. Their use case (general purpose AI
coding agent) justifies it; our use case (interactive
feedback) doesn't. We should resist "let's add a hooks
layer" temptations — we have **one** lifecycle event (task
completion) and **one** decision point per task.

### 2.3 Borrow / not-borrow summary

| Source | Idiom | Decision | ROI | Cycle |
|---|---|---|---|---|
| gemini-cli | `header` chip (≤16 chars) | **borrow** | high (UX context) | cycle-3 ship |
| gemini-cli | `yesno` question type | **borrow** | medium (semantic) | cycle-3 ship if time |
| gemini-cli | `placeholder` text hint | **borrow** | low-LoC freebie | cycle-3 ship |
| gemini-cli | multi-question 1-4 per call | **anti-pattern** | n/a | document only |
| gemini-cli | multiSelect + "All the above" | **anti-pattern** | n/a | document only |
| claude-code | `additionalContext` template vars | **maybe** | medium | cycle-4 (eval needed) |
| claude-code | `Stop` completion gate | **defer** | architectural | needs agent-side |
| claude-code | 22-event hooks lifecycle | **anti-pattern** | n/a | over-engineer for us |

### 2.4 Output recap

Achieved: **3 borrow + 2 anti-idiom + 1 maybe + 1 defer**, vs
budgeted 0-3 borrow + 0-2 anti-idiom. Reality skewed slightly
borrow-heavier because gemini-cli's `ask_user` schema is an
explicit overlap with our `interactive_feedback` — every
schema field becomes a forced comparison.

Time spent: ~45 min total (well under 1-day budget). Source
quality high: both projects have first-party docs maintained
in repo.
