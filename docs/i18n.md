# Internationalization (i18n) Guide

This document is the single source of truth for contributors touching
user-facing strings. It describes the locale layout, the runtime
contract of `t()`, the CI gates that protect parity, and the dev
tooling that keeps feedback loops tight.

## Scope

The project ships two user-facing surfaces, both of which go through
the same JSON-based locale bundles:

- **Web UI** — `static/js/i18n.js` is loaded in the browser. Locale
  JSONs live under `static/locales/<lang>.json`.
- **VSCode webview** — `packages/vscode/i18n.js` is a mirror of the
  Web UI module with identical public API. Locale JSONs live under
  `packages/vscode/locales/<lang>.json`.
- **VSCode extension host** — strings surfaced outside the webview
  (notifications, command palette labels, etc.) go through
  `vscode.l10n.t()`. The bundle lives at `packages/vscode/l10n/`.

Supported languages today: `en` (default), `zh-CN`. Pseudo-locale
(`pseudo`) is a developer tool — see [Pseudo-locale](#pseudo-locale).

## Adding a new user-facing string

1. Add the key to both `static/locales/en.json` **and**
   `packages/vscode/locales/en.json`. Keep them in sync — a CI gate
   (`tests/test_i18n_locale_key_parity.py`) fails if they drift.
2. Add the translation to `zh-CN.json` in both places.
3. Call `t('your.key', { ...params })` from JS; use `data-i18n="..."`
   attributes in HTML for static text.
4. Run `scripts/gen_pseudo_locale.py` to refresh the pseudo locale
   (the CI gate also does this in `--check` mode).
5. Run `uv run python -m pytest tests/ -k "i18n"` locally; everything
   should be green.

### Key naming convention

Keys are dotted paths following the per-surface namespace:

```text
ui.<area>.<element>.<variant>   # webview-specific strings
status.<...>                    # Web UI top-bar/status strings (shared)
aiia.state.<...>                # shared tri-state panel (both halves)
statusBar.<...>                 # VSCode host-side (statusBar.tooltipXxx)
notification.<...>              # notification-center copy
```

Do **not** invent keys that cross these namespaces. The orphan-key
scanner (`scripts/check_i18n_orphan_keys.py`) treats unreferenced keys
as dead and will eventually turn into an error gate.

## ICU plural / select

`t()` supports a small ICU MessageFormat subset (plural/select) so
pluralization lives in the locale file, not in the call site:

```json
{
  "cart.items": "{count, plural, =0 {no items} one {# item} other {# items}}"
}
```

```js
t('cart.items', { count: 1 }) // → "1 item"
t('cart.items', { count: 5 }) // → "5 items"
```

`#` inside a plural option is replaced by the locale-aware number
(`Intl.NumberFormat`). Nested ICU blocks are **not supported** — split
into multiple keys instead. See `tests/test_i18n_icu_plural.py` for
the exhaustive pinned semantics.

## `Intl.*` wrappers

Use the built-in wrappers instead of raw `toFixed` / `toLocaleString`
so the output adapts to the active locale:

| API                           | Backed by                 | Use when               |
| ----------------------------- | ------------------------- | ---------------------- |
| `formatNumber(n, opts?)`      | `Intl.NumberFormat`       | any user-facing number |
| `formatDate(d, opts?)`        | `Intl.DateTimeFormat`     | any user-facing date   |
| `formatRelativeTime(n, unit)` | `Intl.RelativeTimeFormat` | known unit             |
| `formatRelativeFromNow(ms)`   | `Intl.RelativeTimeFormat` | auto-pick unit         |
| `formatList(items, opts?)`    | `Intl.ListFormat`         | comma-separated lists  |

All wrappers cache per-`(locale, options)` tuples so hot paths don't
re-construct a formatter per call.

## Pseudo-locale

The build generates `static/locales/_pseudo/pseudo.json` (and its
VSCode mirror) by running:

```bash
uv run python scripts/gen_pseudo_locale.py
```

The generator wraps every translatable leaf in a visually distinct
envelope like `[!ńëẉẗàśǩ!]`, elongates the string ~30% to test
overflow, and preserves all placeholders / ICU tokens verbatim. To
toggle pseudo at runtime:

- **Web UI**: append `?lang=pseudo` to the URL **or** set
  `localStorage.aiia_i18n_lang = 'pseudo'` from the dev console.
- **VSCode extension**: open Settings → search _pseudo-locale_ →
  toggle **AI Intervention Agent › I18n: Pseudo Locale** (tagged
  _experimental_; never ship this flag as true).

The pseudo locale exists to catch:

- Hardcoded strings (they stay in English while everything around
  them turns into `[!…!]`).
- Concatenation bugs (`'Hello ' + name` breaks because pseudo words
  run into each other).
- Layout overflow (30% longer text exercises wrapping/truncation).
- Missing `dir=rtl` handling (pseudo can be flipped to RTL in
  `gen_pseudo_locale.py --dir=rtl`, not yet default).

## Missing-key observability

Default behavior: `t('not.a.real.key')` returns `"not.a.real.key"` so
the UI never shows a blank. Opt into stricter feedback:

```js
// Dev tooling: log misses to console or telemetry sink.
AIIA_I18N.setMissingKeyHandler((key, lang) => {
  console.warn('[i18n] miss:', key, 'lang=', lang)
})

// Unit tests / dev builds: make misses fatal.
AIIA_I18N.setStrict(true)

// Dashboards: pull a key → count histogram.
AIIA_I18N.getMissingKeyStats()
```

| Gate                                          | Script                                                               | Severity |
| --------------------------------------------- | -------------------------------------------------------------------- | -------- |
| JSON parity across `en` / `zh-CN` / `pseudo`  | `tests/test_i18n_locale_key_parity.py`                               | error    |
| No hardcoded CJK in JS/TS source              | `scripts/check_i18n_js_no_cjk.py`, `scripts/check_i18n_ts_no_cjk.py` | error    |

| Pseudo locale ≥30% length inflation           | `tests/test_i18n_pseudo_inflation.py`                                | error    |
| `t(key, { params })` ↔ placeholder parity     | `scripts/check_i18n_param_signatures.py --strict`                    | error    |
| TypeScript `I18nKey` literal-union up-to-date | `scripts/gen_i18n_types.py --check`                                  | error    |

| Orphan (unreferenced) keys                    | `scripts/check_i18n_orphan_keys.py`                                  | warn     |

## Editor tooling

- Install the [i18n-ally](https://marketplace.visualstudio.com/items?itemName=Lokalise.i18n-ally)
  VSCode extension. `.vscode/settings.json` points it at both locale
  trees and teaches it the custom `t()` / `hostT()` / `data-i18n=...`
  patterns so inline hover + key navigation works out of the box.
- The pseudo tree is explicitly ignored (`i18n-ally.ignoredLocales`)
  so translators never accidentally edit generated output.

## Runtime locale switching

- **Web UI**: the top-right Settings panel exposes a language
  dropdown (`en` / `zh-CN`). Add `pseudo` there manually for QA.
- **VSCode webview**: language follows the extension host
  (`vscode.env.language`) unless the pseudo-locale setting is on.
- **VSCode extension host**: `vscode.l10n.t()` picks up the active
  display language from VSCode itself — no override needed.

## TypeScript compile-time key safety

`packages/vscode/extension.ts` calls into the extension host's
localized strings through `hostT(key: I18nKey)`. `I18nKey` is a
literal-union type generated from `packages/vscode/locales/en.json`
by `scripts/gen_i18n_types.py`, so a typo like
`hostT('statusBar.unkown')` fails at `tsc` time.

Refresh the type bundle whenever you add a new key:

```bash
uv run python scripts/gen_i18n_types.py
```

The CI gate runs the same script with `--check` and refuses to merge
a tree where `i18n-keys.d.ts` drifts from `en.json`.

## When in doubt

- Don't hardcode English or Chinese strings directly into `.js` /
  `.ts`. The CJK gate will reject Chinese; ESLint will reject
  unknown i18n keys; the orphan scanner will flag leftover keys.
- If you need literal braces (`{`, `}`) inside a translation, use
  ICU's apostrophe-escape: `'{literal}'`.
- If a translation value uses `{{version}}` (Mustache) or
  `{count, plural, ...}` (ICU), the param-signature gate
  (`scripts/check_i18n_param_signatures.py --strict`) ensures every
  call site passes exactly the expected params — no missing, no
  extras. Run it locally before opening a PR if you touched either
  a locale value or a `t()` call site.
- If you're unsure whether a string is translatable (console log,
  error string, etc.), err on the side of _translatable_ — it's much
  cheaper to add a key now than to retrofit later.
