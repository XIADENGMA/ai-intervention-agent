# Internationalization (i18n) Guide

This document is the single source of truth for contributors touching
user-facing strings. It describes the locale layout, the runtime
contract of `t()`, the CI gates that protect parity, and the dev
tooling that keeps feedback loops tight.

## Scope

The project ships two user-facing surfaces, both of which go through
the same JSON-based locale bundles:

- **Web UI** — `static/js/i18n.js` is loaded in the browser. Locale
  JSONs live under `static/locales/<lang>.json`.
- **VSCode webview** — `packages/vscode/i18n.js` is a mirror of the
  Web UI module with identical public API. Locale JSONs live under
  `packages/vscode/locales/<lang>.json`.
- **VSCode extension host** — strings surfaced outside the webview
  (notifications, command palette labels, etc.) go through
  `vscode.l10n.t()`. The bundle lives at `packages/vscode/l10n/`.

Supported languages today: `en` (default), `zh-CN`. Pseudo-locale
(`pseudo`) is a developer tool — see [Pseudo-locale](#pseudo-locale).

## Adding a new user-facing string

1. Add the key to both `static/locales/en.json` **and**
   `packages/vscode/locales/en.json`. Keep them in sync — a CI gate
   (`tests/test_i18n_locale_key_parity.py`) fails if they drift.
2. Add the translation to `zh-CN.json` in both places.
3. Call `t('your.key', { ...params })` from JS; use `data-i18n="..."`
   attributes in HTML for static text.
4. Run `scripts/gen_pseudo_locale.py` to refresh the pseudo locale
   (the CI gate also does this in `--check` mode).
5. Run `uv run python -m pytest tests/ -k "i18n"` locally; everything
   should be green.

### Key naming convention

Keys are dotted paths following the per-surface namespace:

```text
ui.<area>.<element>.<variant>   # webview-specific strings
status.<...>                    # Web UI top-bar/status strings (shared)
aiia.state.<...>                # shared tri-state panel (both halves)
statusBar.<...>                 # VSCode host-side (statusBar.tooltipXxx)
notification.<...>              # notification-center copy
```

Do **not** invent keys that cross these namespaces. The orphan-key
scanner (`scripts/check_i18n_orphan_keys.py`) treats unreferenced keys
as dead and will eventually turn into an error gate.

## ICU plural / select

`t()` supports a small ICU MessageFormat subset (plural/select) so
pluralization lives in the locale file, not in the call site:

```json
{
  "cart.items": "{count, plural, =0 {no items} one {# item} other {# items}}"
}
```

```js
t('cart.items', { count: 1 }) // → "1 item"
t('cart.items', { count: 5 }) // → "5 items"
```

`#` inside a plural option is replaced by the locale-aware number
(`Intl.NumberFormat`). See `tests/test_i18n_icu_plural.py` for the
exhaustive pinned semantics.

### Supported ICU features

| Feature                                 | Supported | Notes                                                                                                                       |
| --------------------------------------- | --------- | --------------------------------------------------------------------------------------------------------------------------- |
| `{arg, plural, =N {…} one {…} other {…}}` | yes     | `=N` exact match wins over the CLDR category.                                                                               |
| `{arg, selectordinal, =N {…} one {…} two {…} few {…} other {…}}` | yes | Uses `Intl.PluralRules({ type: 'ordinal' })`. Same `=N`-beats-category rule as cardinal plural. See [Ordinal plural](#ordinal-plural-selectordinal). |
| `{arg, select, KEY {…} other {…}}`       | yes      | String equality; `other` is the required fallback.                                                                          |
| `#` inside plural option                 | yes      | Bound to the **innermost enclosing plural / selectordinal** argument only — nested plurals never leak their `#` outwards.    |
| Nested plural/select inside any option   | yes      | Arbitrary depth. See the three-level case in `test_i18n_icu_plural.py::test_three_level_nesting_plural_select_plural`.     |
| `{{mustache}}` interpolation             | yes      | Processed **after** ICU selection, so placeholders inside the non-chosen branch are never exposed.                           |
| `{arg, number, …}` / `{arg, date, …}`    | no        | Skeletons aren't parsed. Pre-format the value with `formatNumber` / `formatDate` and pass the string as a `{{mustache}}` param. |

### Apostrophe escaping

`t()` implements ICU `MessagePattern.ApostropheMode.DOUBLE_OPTIONAL`
(the ICU4J / FormatJS default) so messages can contain literal `{`,
`}`, `|`, `#` or `'`:

```jsonc
{
  // Two apostrophes render as one literal apostrophe.
  "status.contraction": "Didn''t finish",           // → "Didn't finish"
  // 'X' starts a quoted span (X ∈ {, }, |, #) that runs until the next
  // unpaired '. Inside a quoted span, special chars are literal.
  "status.literal_braces": "Render '{literal}' verbatim", // → "Render {literal} verbatim"
  // Lone ' that is NOT followed by a special char is itself literal.
  "status.lone_apos": "I don't know"                // → "I don't know"
}
```

Within a plural body, `'#'` suppresses numeric substitution:

```jsonc
{
  "tags.plural": "{n, plural, one {# item (tag '#')} other {# items (tag '#')}}"
}
// n=1 → "1 item (tag #)"   (leading # is formatted, second is literal)
// n=3 → "3 items (tag #)"
```

Internally the apostrophe-escape runs once at `t()` entry via a PUA
(Private Use Area) tokenizer, so nested ICU blocks and mustache
placeholders are unaffected by the rewrite.

### Ordinal plural (`selectordinal`)

English, Welsh, and a handful of other locales have an *ordinal*
plural axis ("1st", "2nd", "3rd", …) whose categories differ from the
cardinal axis ("1 item", "2 items", …). ICU MessageFormat exposes
this as `selectordinal`, and `t()` supports it with its own
`Intl.PluralRules({ type: 'ordinal' })` cache so the two axes never
steal capacity from each other:

```jsonc
{
  "rank.suffix": "{n, selectordinal, one {#st} two {#nd} few {#rd} other {#th}}"
}
// en  n=1  → "1st"    zh-CN n=1 → "1th"  (zh-CN has only the "other" bucket)
// en  n=22 → "22nd"   zh-CN n=22 → "22th"
```

Rules:

- `=N` exact match still beats the CLDR category, identical to
  cardinal plural.
- Locales without a distinct ordinal axis (e.g. `zh-CN`) collapse
  every number onto `other`, matching the behaviour of the browser's
  `Intl.PluralRules('zh-CN', { type: 'ordinal' })`.
- The cardinal `{arg, plural, …}` and ordinal `{arg, selectordinal, …}`
  caches are partitioned so exercising one never evicts the other —
  see [Intl instance cache](#intl-instance-cache).

Pinned by `tests/test_i18n_selectordinal.py` and the `E12`
cross-feature case in `scripts/red_team_i18n_runtime.mjs`.

## `Intl.*` wrappers

Use the built-in wrappers instead of raw `toFixed` / `toLocaleString`
so the output adapts to the active locale:

| API                           | Backed by                 | Use when               |
| ----------------------------- | ------------------------- | ---------------------- |
| `formatNumber(n, opts?)`      | `Intl.NumberFormat`       | any user-facing number |
| `formatDate(d, opts?)`        | `Intl.DateTimeFormat`     | any user-facing date   |
| `formatRelativeTime(n, unit)` | `Intl.RelativeTimeFormat` | known unit             |
| `formatRelativeFromNow(ms)`   | `Intl.RelativeTimeFormat` | auto-pick unit         |
| `formatList(items, opts?)`    | `Intl.ListFormat`         | comma-separated lists  |

### `formatRelativeFromNow` unit buckets

The unit auto-pick uses the same thresholds as moment.js's
`relativeTimeThreshold` defaults (`s=45 / m=45 / h=22 / d=26 / M=11`),
which day.js, luxon and date-fns-tz all inherit. Picking this table
avoids the `in 60 seconds` / `in 24 hours` UX bug where a naive
`< 60 ? 'second' : 'minute'` cutoff keeps counting seconds past a
user-perceived minute boundary:

| Absolute delta                                | Bucket   | Divisor            |
| --------------------------------------------- | -------- | ------------------ |
| `< 45 s`                                      | `second` | `1`                |
| `< 45 × 60 s = 2700 s`                        | `minute` | `60`               |
| `< 22 × 3600 s = 79200 s`                     | `hour`   | `3600`             |
| `< 26 × 86400 s = 2246400 s`                  | `day`    | `86400`            |
| `< 11 × 30 × 86400 s = 28512000 s`            | `month`  | `2592000`          |
| else                                          | `year`   | `31536000`         |

Pinned by `tests/test_i18n_relative_time_thresholds.py` against both
`i18n.js` copies; a byte-parity assertion forbids the two halves from
drifting on the same input.

#### Edge-case inputs

`formatRelativeFromNow` never throws — the contract for junk input is
"degrade to `in 0 seconds`" (for `en`, `numeric: 'always'`), matching
moment.js's `humanize(0)` fallback for malformed durations. Covered
inputs:

| Input                       | Output                 | Rationale                                                   |
| --------------------------- | ---------------------- | ----------------------------------------------------------- |
| `Infinity` / `-Infinity`    | `in 0 seconds`         | `new Date(Infinity)` is `Invalid Date`; graceful fallback   |
| `NaN`                       | `in 0 seconds`         | same as above                                               |
| `Number.MAX_SAFE_INTEGER`   | `in 0 seconds`         | adding to `FAKE_NOW` overflows `Date`'s ±8.64e15 ms range   |
| `new Date("")`              | `in 0 seconds`         | invalid date input                                          |
| `undefined` / `null`        | `in 0 seconds`         | `_toDate` returns `now`, diff is 0                          |

Pinned by `tests/test_i18n_relative_time_thresholds.py::TestRelativeTimeEdge*`.
Byte-parity is enforced across both halves.

### Intl instance cache

Constructing `new Intl.NumberFormat / DateTimeFormat / …` is a
millisecond-level operation on low-end devices and the VSCode
extension host, so `_getIntl` memoises per `(ctor, locale,
stableStringify(options))`. The cache is LRU-bounded to keep long
sessions from leaking formatter instances:

| Cache                    | Cap | Promotion on hit | Eviction |
| ------------------------ | --- | ---------------- | -------- |
| `Intl.{NumberFormat,DateTimeFormat,RelativeTimeFormat,ListFormat}` | `50` per ctor | `delete` → `set` moves key to tail | Drop head when `size > 50` |
| `Intl.PluralRules` (cardinal) | `16` | same | same |
| `Intl.PluralRules` (ordinal, `{ type: 'ordinal' }`) | `16` | same | same |

These caps are pinned by `tests/test_i18n_intl_cache_lru.py`. If you
need to nudge them, update both `i18n.js` copies **and** the test
constants in the same commit.

#### Stable-sorted cache key

The cache key canonicalises options via a recursive `_stableStringify`
that sorts object keys before serialising — same idiom FormatJS's
`intl-format-cache` uses. Without it, `Object.assign({}, base,
override)` and plain-literal option objects would produce distinct
keys for semantically-identical options, silently doubling the LRU
footprint. Array order is preserved (all `Intl.*` options with
positional semantics). Behaviour is pinned by
`tests/test_i18n_intl_cache_stable_key.py`.

##### Cycle-safe serialisation (Batch-2 H8)

`_stableStringify` walks arbitrary shapes. Three historical foot-guns
are now closed:

1. **Circular references** — a naive recursive `JSON.stringify` blows
   the stack on `const a = {}; a.self = a`. `_stableStringifyInner`
   threads a `WeakSet` through each call, throws a tagged
   `{ __aiiaCircular: true }` error on re-entry, and `_intlKey` catches
   the tag to fall back to a **shape signature** (`lang + '|?cycle|' +
   sortedTopLevelKeys`). Two cyclic option bags with different
   top-level shapes therefore map to *distinct* cache entries instead
   of colliding into one poisoned slot.
2. **`toJSON()` honouring** — the public contract of `JSON.stringify`
   is that `Date`, `Moment`, and any object with a `.toJSON()` method
   gets serialised through that method. `_stableStringifyInner` does
   the same before recursing, so two `Date` instances at the same
   instant collapse to a single cache key.
3. **`BigInt`** — vanilla `JSON.stringify` throws on `BigInt`.
   `_stableStringifyInner` serialises `123n` as `"123n"` so options
   bags that embed BigInts remain cacheable. Distinct BigInts land in
   distinct cache entries.

Pinned end-to-end by `tests/test_i18n_stable_key_cycle.py` plus the
`E13` cluster in `scripts/red_team_i18n_runtime.mjs`.

## Pseudo-locale

The build generates `static/locales/_pseudo/pseudo.json` (and its
VSCode mirror) by running:

```bash
uv run python scripts/gen_pseudo_locale.py
```

The generator wraps every translatable leaf in a visually distinct
envelope like `[!ńëẉẗàśǩ!]`, elongates the string ~30% to test
overflow, and preserves all placeholders / ICU tokens verbatim. To
toggle pseudo at runtime:

- **Web UI**: append `?lang=pseudo` to the URL **or** set
  `localStorage.aiia_i18n_lang = 'pseudo'` from the dev console.
- **VSCode extension**: open Settings → search _pseudo-locale_ →
  toggle **AI Intervention Agent › I18n: Pseudo Locale** (tagged
  _experimental_; never ship this flag as true).

The pseudo locale exists to catch:

- Hardcoded strings (they stay in English while everything around
  them turns into `[!…!]`).
- Concatenation bugs (`'Hello ' + name` breaks because pseudo words
  run into each other).
- Layout overflow (30% longer text exercises wrapping/truncation).
- Missing `dir=rtl` handling (pseudo can be flipped to RTL in
  `gen_pseudo_locale.py --dir=rtl`, not yet default).

## Missing-key observability

Default behavior: `t('not.a.real.key')` returns `"not.a.real.key"` so
the UI never shows a blank. Opt into stricter feedback:

```js
// Dev tooling: log misses to console or telemetry sink.
AIIA_I18N.setMissingKeyHandler((key, lang) => {
  console.warn('[i18n] miss:', key, 'lang=', lang)
})

// Unit tests / dev builds: make misses fatal.
AIIA_I18N.setStrict(true)

// Dashboards: pull a key → count histogram.
AIIA_I18N.getMissingKeyStats()
```

Contract when a missing-key handler throws:

- **Strict mode on** (`setStrict(true)`): the exception bubbles up so
  the caller sees the failure — this is the mode dev/CI builds should
  use.
- **Strict mode off** (default): `t()` still returns the raw key so the
  UI stays alive, and the handler's exception is surfaced via
  `console.warn('[i18n] missing-key handler threw:', e)` so the bug is
  visible in DevTools / the extension-host Output Channel instead of
  being silently swallowed. Both the Web UI and VSCode copies behave
  identically; `tests/test_i18n_missing_key_handler.py` pins the
  parity.

## Security: prototype-pollution hardening

`t()` and the key lookup pipeline refuse to surface prototype-chain
values. This closes the class of bug tracked by Snyk
SNYK-JS-I18NEXT-1065979 and preempts any downstream merge-function
regression from leaking `Object.prototype` methods through the
translation layer.

- `_interpolateMustache` requires `{{name}}` to be an **own** property
  of `params` (`Object.prototype.hasOwnProperty.call`) **and** rejects
  the three canonical pollution keywords (`__proto__`, `constructor`,
  `prototype`) outright — templates containing them render as literal
  placeholders, never as `function Object() { [native code] }` /
  `[object Object]` / similar runtime leaks.
- `resolve(key, lang)` applies the same two-layer guard at every
  dotted-key segment, so `t('ui.__proto__.leak')` always misses even
  if the locale bundle was hand-crafted to include an attacker-
  controlled `__proto__` key.

Pinned by `tests/test_i18n_interp_security.py` and
`tests/test_i18n_resolve_security.py`. Behaviour is byte-parallel
across Web UI / VSCode copies.

## Security: XSS contract for `t()` output

i18next shipped two disclosed XSS CVEs (CVE-2017-16010 and
AIKIDO-2024-10543) that collapse to the same root cause: callers
treat translation output as HTML-safe, write it to `.innerHTML`, and
a malicious translation or parameter value injects arbitrary markup.
Our `t()` intentionally does **not** HTML-escape (same contract
`react-intl`'s `FormattedMessage` uses when rendered through JSX)
because:

- The default sink is `.textContent`, which auto-escapes.
- ICU `#` / `{{mustache}}` templates are developer-authored and
  escaping them would double-encode the visible output.

So every caller that writes `t(…)` / `_t(…)` output into
`.innerHTML` (or a framework path that forwards to it) must carry an
explicit **`AIIA-XSS-SAFE: <reason>`** comment on the assignment line
or in the contiguous comment block directly above it. The reason
should answer *why* the payload is safe — typically one of:

- "Dev-authored locale key + no user-controlled params" — the
  translation value lives in `locales/*.json` and `t()` is called
  without params (or with numeric-only params from ICU `plural` /
  `selectordinal`).
- "Runs through `sanitizePromptHtml`" / "explicit `DOMPurify.sanitize`
  boundary upstream" — the value was already filtered.

A pre-commit + CI gate (`tests/test_i18n_innerhtml_safety_comment.py`)
scans `static/js/**` and `packages/vscode/**` with a ripgrep-style
regex and fails if any `.innerHTML = …t(…)` assignment is missing
the marker. The `data-i18n-html` DOM binding is an explicit opt-in
for locale values that *must* be HTML — the binding points carry the
`AIIA-XSS-SAFE:` marker in both `static/js/i18n.js` and
`packages/vscode/i18n.js`.

## Resolve: non-string leaf warn-once

`resolve(key, lang)` refuses to surface non-string leaves (an object,
a number, a boolean, `null`, …). That's the right default — writing
`"[object Object]"` into the UI would be worse than falling back to
the raw key. But it collapses two genuinely different developer
mistakes into the same "missing key" diagnostic:

- `t('aiia.foo.bar')` where the locale file never defined
  `aiia.foo.bar` — the right fix is to add the key.
- `t('aiia.foo')` where the locale file has
  `{ aiia: { foo: { bar: '…' } } }` — the right fix is to call
  `t('aiia.foo.bar')`. Adding a key named `aiia.foo` would shadow the
  namespace and break every call site that used the deeper key.

`_resolvePath` now returns a `{ value, shape, nodeType }` tuple so
`t()` can emit a **separate** diagnostic per genuinely different
mistake:

- On the first `(lang, key)` hit that lands on a non-string,
  `_reportNonString` logs
  `[i18n] resolved non-string: aiia.foo (lang=en, type=object,
  hint: try a deeper key)`.
- Subsequent hits for the same pair are silenced (once-set keyed by
  `lang + '\u0001' + key`) — no log spam.
- Strict mode (`setStrict(true)`) throws
  `[i18n] non-string resolve: aiia.foo (lang=en, type=object)`
  instead of the generic missing-key message, so CI / unit tests
  pinpoint the namespace vs. leaf confusion immediately.
- Plain missing keys keep their existing `[i18n] missing-key handler
  threw: …` / `[i18n] missing key: …` signatures so dashboards and
  log scrapers that already distinguish the two never see one
  masquerading as the other.

Pinned by `tests/test_i18n_non_string_resolve_warn.py` and the `E14`
cluster in `scripts/red_team_i18n_runtime.mjs`. Test-only hooks
`AIIA_I18N__test.getNonStringHits()` and
`AIIA_I18N__test.resetNonStringHits()` expose the dedupe set for
assertions without parsing console output.

## ICU AST compile cache

`_findIcuBlock` + `_parseIcuOptions` are pure functions of the
(apostrophe-escaped) template string. FormatJS's
`intl-messageformat` shipped AST caching for exactly the same
reason: hot re-render paths should not re-walk the template every
time. Batch-3 H12 adds an LRU cache keyed by raw template:

- Entries store the full top-level block layout
  (`{ blocks: [...], trivial: boolean }`) so `_renderIcu` becomes a
  `for` over the cached array instead of a while-loop over
  `_findIcuBlock`.
- Cap: **256 entries per module** (matches FormatJS's default
  `maxCacheSize`). Hits refresh insertion order (`delete` → `set`)
  so MRU templates survive LRU eviction.
- **Fast-path anti-pollution**: templates with no `{` at all skip
  the cache entirely. Post-hash-replacement literals such as
  `"3 items"` recurse through `_renderIcu` once per distinct count,
  and without this short-circuit they would each evict a real
  template. The cache now holds only strings that could plausibly
  be re-seen.
- `AIIA_I18N__test.getIcuCompileCacheSize()` and
  `peekIcuCompileKeys()` expose the cache to pytest
  (`tests/test_i18n_icu_compile_cache.py`). The red-team script's
  **E17** case pins the geometry across Web ↔ VSCode halves.
- `AIIA_I18N__test.clearIntlCaches()` empties the ICU bucket
  alongside the `Intl.*` and `Intl.PluralRules` caches, so one call
  resets the entire formatter state for deterministic tests.

## Bidirectional text isolation (UAX #9 §3.1)

`AIIA_I18N.wrapBidi(value)` wraps a fragment with `U+2068 FIRST
STRONG ISOLATE` and `U+2069 POP DIRECTIONAL ISOLATE`. The W3C i18n
WG's `qa-bidi-unicode-controls` guide, Mozilla's Project Fluent
and ICU4J 74's `MessageFormatter.formatWithBidiIsolate` all
recommend this pairing as the default for non-HTML sinks (plain
text, VSCode webview Output Channel, `title` attributes, log
lines), because inline bidirectional embeddings can otherwise leak
directional runs into their neighbours.

Contract:

- `wrapBidi('Ada')` → `'\u2068Ada\u2069'`.
- `wrapBidi(null)` / `wrapBidi(undefined)` / `wrapBidi()` /
  `wrapBidi('')` → `''` (callers expecting a no-op for empty
  inputs get it).
- `wrapBidi(42)` → `'\u206842\u2069'` (non-string coerced first).
- **Idempotent**: an already-wrapped input passes through
  unchanged, so nested call sites never produce `FSI·FSI·…·PDI·PDI`.

Today the project does not ship an RTL locale, so the wrapper sits
on the public API for callers that stitch user-provided fragments
(filenames, usernames, paths) manually. When Arabic / Hebrew land
the wrapper is already tested, byte-parallel across halves
(`tests/test_i18n_bidi_isolation.py` and red-team **E16**), and
documented as the canonical remedy.

## Locale JSON shape validation

`scripts/check_i18n_locale_shape.py` is the lint-time backstop for
the `resolve()` runtime contract. It refuses to exit `0` if any
locale file under `static/locales/` or `packages/vscode/locales/`
has a non-string leaf (number, boolean, `null`, array), a non-object
interior, or malformed JSON.

- Prints a structured diagnostic per violation:
  `path › dotted.key :: reason (got type)`.
- Accepts `--locales-dir` (repeatable) so tests can point it at a
  synthetic fixture without mutating the real bundles.
- `--json` emits a machine-readable array for tooling integration.
- Empty objects are allowed — they're legitimate namespace
  placeholders.
- Wired into `scripts/ci_gate.py` so PRs that smuggle
  `{ "count": 3 }` into a locale are caught before the Batch-2 H11
  runtime warn-once ever fires.

Batch-2 H11 still stands: if somehow a non-string leaf slips past
this gate (e.g. via a rogue `registerLocale` call injecting raw
JavaScript), the runtime logs a once-per-`(lang, key)` console
warning and throws in strict mode. The two layers are defence-in-
depth, not duplicates.

## Property-based fuzz parity

`tests/test_i18n_fuzz_parity.py` drives a deterministic random
generator (`random.Random(seed=0xA11ABADE)`, 200 iterations) over
a grammar that produces literals, mustaches, plurals,
`selectordinal`, selects, apostrophes and up to two levels of
nesting. Each template is rendered through both
`static/js/i18n.js` and `packages/vscode/i18n.js` in a single
Node process per half, and the test asserts:

- every call returns a string without throwing;
- Web ↔ VSCode outputs are byte-identical for the full corpus;
- no PUA marker (`U+F0000`–`U+F0FFF`) leaks into the output — the
  apostrophe tokenizer must always round-trip.

The seed is pinned so CI reproduces the exact 200-sample matrix on
every run. When a regression hits, the failing `id` + template +
params are printed so the case can be copied into the permanent
pin set. The red-team script's **E18** case carries a compact
landmine list (three-level nesting + apostrophe + `selectordinal`
exact-match) as an always-on smoke check.

## Web UI ↔ VSCode webview parity

`static/js/i18n.js` and `packages/vscode/i18n.js` are byte-parallel
mirrors: every runtime contract (ICU subset, apostrophe escape,
nested `#` scope, `formatRelativeFromNow` thresholds, missing-key
handler, Intl LRU caps) must produce identical output for identical
input. Several tests run against both files from the same harness:

- `tests/test_i18n_icu_plural.py::TestIcuPluralVSCode` re-runs every
  ICU parser case on the VSCode copy.
- `tests/test_i18n_relative_time_thresholds.py::TestByteParityForRelativeTimeBuckets`
  fails if the two halves disagree on any bucket boundary.
- `tests/test_i18n_intl_cache_lru.py::TestIntlLruByteParity` fails if
  the LRU eviction order drifts.
- `tests/test_i18n_missing_key_handler.py` runs the full suite under
  both `WEBUI_I18N` and `VSCODE_I18N` paths.
- `tests/test_i18n_intl_cache_stable_key.py::TestStableKeyByteParity`
  pins identical canonical cache-key bytes across halves.
- `tests/test_i18n_stable_key_cycle.py` exercises cyclic / `toJSON` /
  `BigInt` option bags on both halves — byte-parity is implicit
  (same fallback cache-key rules).
- `tests/test_i18n_selectordinal.py` runs the CLDR ordinal matrix
  against both copies so an ordinal-cache divergence fails the gate.
- `tests/test_i18n_non_string_resolve_warn.py` pins the warn-once
  diagnostic (`[i18n] resolved non-string: …`) on both copies.
- `tests/test_i18n_innerhtml_safety_comment.py` enforces the
  `AIIA-XSS-SAFE:` marker across both trees statically.

## Editor tooling

- Install the [i18n-ally](https://marketplace.visualstudio.com/items?itemName=Lokalise.i18n-ally)
  VSCode extension. `.vscode/settings.json` points it at both locale
  trees and teaches it the custom `t()` / `hostT()` / `data-i18n=...`
  patterns so inline hover + key navigation works out of the box.
- The pseudo tree is explicitly ignored (`i18n-ally.ignoredLocales`)
  so translators never accidentally edit generated output.

## Runtime locale switching

- **Web UI**: the top-right Settings panel exposes a language
  dropdown (`en` / `zh-CN`). Add `pseudo` there manually for QA.
- **VSCode webview**: language follows the extension host
  (`vscode.env.language`) unless the pseudo-locale setting is on.
- **VSCode extension host**: `vscode.l10n.t()` picks up the active
  display language from VSCode itself — no override needed.

## TypeScript compile-time key safety

`packages/vscode/extension.ts` calls into the extension host's
localized strings through `hostT(key: I18nKey)`. `I18nKey` is a
literal-union type generated from `packages/vscode/locales/en.json`
by `scripts/gen_i18n_types.py`, so a typo like
`hostT('statusBar.unkown')` fails at `tsc` time.

Refresh the type bundle whenever you add a new key:

```bash
uv run python scripts/gen_i18n_types.py
```

The CI gate runs the same script with `--check` and refuses to merge
a tree where `i18n-keys.d.ts` drifts from `en.json`.

## Known limitations & roadmap

These are deliberate non-goals for the current iteration — documented
so contributors don't accidentally re-open the same investigation.

- **Skeleton-based `{arg, number, …}` / `{arg, date, …}` ICU**:
  omitted to keep the parser tiny. Pre-format via `formatNumber` /
  `formatDate` and pass the string through `{{mustache}}` instead.
- **Bidirectional text isolation (UAX #9 FSI / PDI)**: the helper
  is shipped (`AIIA_I18N.wrapBidi`) and byte-parallel across both
  halves — see the _Bidirectional text isolation_ section — but
  `t()` does **not** auto-wrap every interpolated parameter. Adding
  that would make the output bit-different from i18next for
  LTR-only locales and break pseudo-locale inflation checks. When
  the first RTL locale lands, the call sites that need auto-wrap
  should flip to `wrapBidi()` explicitly.
- **Locale negotiation beyond a single tag**: `navigator.languages`
  BCP47 negotiation (e.g. `en-GB` falling back to `en`) is not wired
  up; we rely on `DEFAULT_LANG = 'en'` as the sole fallback. Adding a
  proper negotiation tree is straightforward but gated on shipping a
  third language.
- **Hypothesis / fast-check property-based fuzzing**: the Batch-3
  fuzz test uses a pinned `random.Random` seed rather than a
  hypothesis-driven shrinker. That keeps the dependency footprint
  lean (no new `hypothesis` package in `pyproject.toml`) at the
  cost of shrinker-guided counterexamples. Upgrade path is
  additive — swap the seeded generator for `hypothesis.strategies`
  without touching the oracle.

## When in doubt

- Don't hardcode English or Chinese strings directly into `.js` /
  `.ts`. The CJK gate will reject Chinese; ESLint will reject
  unknown i18n keys; the orphan scanner will flag leftover keys.
- If you need literal braces (`{`, `}`, `|`, `#`) inside a translation,
  wrap them with ICU's apostrophe-escape `'…'` (e.g. `'{literal}'`) or
  use `''` for a standalone apostrophe. See
  [Apostrophe escaping](#apostrophe-escaping) for the full rule set.
- If a translation value uses `{{version}}` (Mustache) or
  `{count, plural, ...}` (ICU), the param-signature gate
  (`scripts/check_i18n_param_signatures.py --strict`) ensures every
  call site passes exactly the expected params — no missing, no
  extras. Run it locally before opening a PR if you touched either
  a locale value or a `t()` call site.
- If you're unsure whether a string is translatable (console log,
  error string, etc.), err on the side of _translatable_ — it's much
  cheaper to add a key now than to retrofit later.
