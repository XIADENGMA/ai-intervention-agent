### 2.2 `ec70124` — placeholder_truncated warning

cr36 §8 #2 [low] asked for surfacing the silent server-side
clamp. The fix:

1. Adds `placeholder_truncated: true` +
   `placeholder_original_length: int` +
   `placeholder_max_length: 200` to `POST /api/tasks`
   response **only when** clamp activates.
2. **Boundary discipline**: tested 200 (boundary — no warning),
   201 (boundary +1 — warning), 300 (clear truncation),
   missing/empty (no warning), and **whitespace-padded long**
   (strip first; if effective length ≤ 200, no warning).
3. **Backward compat**: old agents that don't read these fields
   see no difference. New agents can surface to user.

The boundary tests are particularly valuable: `200 == 200` is
the boundary where engineers most commonly screw up (off-by-
one direction errors). Test asserts the documented behavior
(`>200 triggers, ≤200 doesn't`), preventing future "I changed
the clamp to 199 but forgot to update the response check"
drift.

### 2.3 `565d6e2` — yesno type as a full-stack idiom port

This is the cycle's largest commit (+421/-5) and the most
disciplined full-stack port to date.

Notable design choices:

1. **hide/show not destroy/rebuild**: when `question_type`
   toggles between None and `'yesno'`, the helper hides
   `#feedback-text` (display=none) and inserts a sibling
   button group; on toggle back, removes the button group
   and restores display=''. This preserves `taskTextarea
   Contents` mapping (task-scoped textarea content saved by
   `switchTask`) — if we destroyed the textarea, switching
   from a yesno task back to a textarea task would silently
   lose any draft.
2. **Forward-compat normalization**: only `"yesno"` (after
   `.strip()`) is whitelisted; *any* other string —
   `"YESNO"`, `"yes-no"`, `"rating"`, `"choice"` — silently
   becomes `None`. Test `test_unknown_type_silently_none`
   exercises 6 unknown values to lock this in. The forward-
   compat rationale (future types like `"choice"` /
   `"rating"` can be added without schema changes; old
   frontends ignore unknowns) is the right architectural
   stance for an MCP tool surface that we want to evolve
   without breaking deployed clients.
3. **44px touch target** in CSS (`min-height: 44px` on
   `.yesno-btn`) — meets WCAG 2.1 AA recommended minimum.
   Tested by `test_a11y_touch_target_44px`. This is the kind
   of a11y discipline most "binary button" implementations
   skip.
4. **Submit literal "yes" or "no"**: button click writes
   `feedbackTextarea.value = "yes"` (or `"no"`) and triggers
   form submit. The agent on the receiving side gets the
   exact literal string — no JSON envelope, no schema
   complexity. Test `test_submit_writes_literal_yes_or_no`
   asserts both branches.
5. **Submit path graceful degradation**: `form.requestSubmit()`
   first (modern browsers, validates form before submit),
   then `form.submit()` (legacy), then `submit-btn.click()`
   (fallback if form element missing). Three-tier defense.
6. **i18n + English fallback**: `t("page.yesnoYes") || "Yes"`
   ensures the buttons render even if i18n loading fails.
   Same pattern as cr35 §8 #1 tooltip suffix.

The cycle ships **two** Track B borrow items in 5 commits
(placeholder hardening + yesno). The third borrow (header
chip) is correctly deferred — it requires tab-UI rethink that
doesn't fit a 5-commit cycle.

### 3.2 `ec70124` [LOW]

- **Response field `placeholder_max_length` is hardcoded 200**
  in `task.py` route handler, while the actual clamp is in
  `add_task` in `task_queue.py`. If we ever change the clamp
  (e.g. to 300), there are **2 places** that need to update.
  **Recommendation**: extract a module-level constant
  `PLACEHOLDER_MAX_LENGTH = 200` in `task_queue.py` and
  import it where needed. ~10 LoC. **Action**: cr38 cycle
  if not picked up sooner.

- **Response field naming `placeholder_max_length` is verbose
  but consistent with `placeholder_original_length`**. Could
  shorten to `max_length` / `original_length` (already in
  the `placeholder_*` prefix scope), but that breaks
  symmetry with potential future fields like
  `prompt_max_length`. **No action**; consistent.

### 3.3 `565d6e2` [LOW + INFO]

- **[LOW]** **No test for the actual button click triggering
  form submit**. The test asserts the source-code patterns
  (`_submit("yes")` literal present, button.addEventListener
  set), but doesn't simulate a click + verify form
  submission. A small Selenium/Playwright e2e would close
  this gap. **Defer**: we don't currently have an e2e harness
  for multi_task.js interactions; would be a larger
  investment than this fix justifies in isolation.

- **[LOW]** **Yesno button group doesn't persist user
  intermediate state** (textarea content). On a yesno
  task, `taskTextareaContents[taskId]` may be empty (because
  the textarea was hidden so user never typed); switching to
  another yesno task and back preserves nothing visible.
  This is *intended* (yesno is binary, no intermediate state
  meaningful), but worth noting as a documented design
  decision rather than an undiscovered bug. **No action**;
  add to mining-3 borrow #2 doc as design note.

- **[INFO]** **`hide` strategy creates a perpetual hidden
  `<textarea>`** in DOM. Accessibility tools may surface it
  inappropriately. We could add `aria-hidden="true"` and
  `tabindex="-1"` when hidden. **Defer**: needs a11y audit
  to confirm; ROI low.

- **[INFO]** **Default literal `"yes"` / `"no"` is English**.
  i18n labels are translated (是/否 for zh-CN), but the
  submitted feedback string is always English. This is
  *intended* — the agent that consumes the feedback should
  not need to understand zh-CN. But agents using their own
  parsers might miss this nuance. **Document only**.
