1. **Non-intrusive default**: `connected` state CSS is `display: none`.

   `data-sse-state`, CSS reads it. Three benefits:
   - Test surface is `getAttribute` ŌåÆ readable string assertions, no
     DOM-style snapshot needed.

   - Cypress / Playwright / browser-use can wait for state transitions
     via `[data-sse-state="reconnecting"]` selector without polling
     internal JS state.
3. **Threshold consistency invariant**: the test
   `test_disconnect_threshold_constant_matches_backoff_cap` asserts
   `SSE_STATUS_DISCONNECTED_DELAY_MS === 30000` AND that the existing
   `Math.min(30000, ŌĆ”)` cap still exists. Future bump to either side
   without synchronising both will hard-fail. This kind of cross-line
   invariant is the right way to lock these dual constants.

## 3. Issues found

### 3.1 [low] `_t` truthy-fallback footgun in `_resolveSseStatusLabel`

In `multi_task.js` after `24ae44e`:

```javascript
var v;
if (key === "page.sseStatus.connected") {
  v = _t("page.sseStatus.connected");
} else if (...) ...
return typeof v === "string" && v.length > 0 && v !== key ? v : fallback;
```

`_t` in this module returns the `key` itself when i18n is unavailable
(line 116). The `v !== key` ternary guards against that case correctly,
but the **structure** is non-obvious ŌĆö first-time reader may assume
"truthy v ŌćÆ use v" pattern and miss the key-equality check.

**Mitigation**: comment already explains it; future refactor could
introduce a `_tStrict(key, fallback)` helper that internally does the
sentinel check, applied module-wide. Not urgent ŌĆö this is a single
function with a clear comment.

### 3.2 [low] `_resolveSseStatusLabel` triple-branch repetition

The function spells out three branches with literal-key calls (for
the i18n scanner). This is verbose but **necessary** given the
scanner's `JS_T_CALL_RE` regex shape. A higher-leverage fix would be
to extend `scripts/check_i18n_orphan_keys.py::JS_T_CALL_RE` to
recognize a "lookup-table" pattern (`var KEYS = { connected: "...",
reconnecting: "...", disconnected: "..." }; ... _t(KEYS[state])`), but
that's scanner-side work spanning all consumers ŌĆö not justifiable for
3 keys.

Verdict: accept the repetition as the price of keeping the scanner
simple.

The SSE indicator's actual visual states (yellow pulse, red badge)
