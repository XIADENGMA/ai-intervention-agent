### §2.4 Production ship review: Track A iOS Safari A2HS hint (commit 2)

**Scope**: JS module (~220 LoC) + ~115 LoC CSS (dark +
light + reduced-motion + safe-area-inset) + i18n (4 keys ×
3 locales) + Flask asset version wiring + template script
+ 22 regression tests + pseudo locale + minified JS +
brotli/gzip precompressed artifacts.

**Quality signals**:

| Aspect | Evidence | Verdict |
|---|---|---|
| Architecture isolation | Pure IIFE module; no existing UI dependency | ✅ zero-intrusion |
| UA detection robustness | 4 dimensions: iPhone/iPad/iPod UA + iPad desktop-mode + in-app browser exclusion + Safari UA verification | ✅ comprehensive |
| Standalone detection | 2 methods (webkit `navigator.standalone` + matchMedia `display-mode: standalone`) | ✅ legacy + modern coverage |
| Race condition protection | 1.5s show-delay + setTimeout callback double-check (dismiss/standalone/banner-exists) | ✅ defensive |
| Persistence schema | `aiia.iosA2hsDismissed.v1` envelope; permanent dismiss vs R247's 30-day TTL | ✅ correct UX distinction |
| a11y | `role="dialog"` + `aria-labelledby` + `aria-hidden` on icons + i18n aria-labels | ✅ screen-reader friendly |
| Mobile-specific UX | `env(safe-area-inset-*)` for iPhone notch + home-bar; bottom positioning | ✅ iOS-native feel |
| Reduced-motion compliance | `@media (prefers-reduced-motion: reduce)` disables 300ms slide-up | ✅ a11y standard |
| Theme awareness | Dark (default) + light theme overrides; brand purple accent both | ✅ system-design consistency |
| Test coverage | 22 tests / 5 layers (JS / CSS / i18n / asset wiring / template) | ✅ comprehensive |

**Design decision review — "permanent dismiss vs 30-day TTL"**:
explicit UX contrast with R247 PWA install. Rationale: iOS
A2HS is a **one-shot instructional hint** (user learns
Share→A2HS once, applies forever); re-prompting would
patronize. R247's 30-day TTL handles "user might change
mind about installing Chrome PWA later", which doesn't
apply once the user understands iOS Share flow. **Defensible**.

**Design decision review — "1.5s show-delay"**:
explicit anti-noise timing. Surfacing banner immediately
on page load would compete with main UI rendering and feel
intrusive. 1.5s gives content first paint time. Defensible
empirical choice (no a/b data, but reasonable).

**Concerns / follow-ups** (none blocking):

- **[low]** Banner has no auto-dismiss timer. User might
  ignore it indefinitely if they don't engage. Could
  auto-dismiss after 30s or auto-dismiss when user
  scrolls. Defer pending UX research; could create user
  fatigue if too aggressive.
- **[low]** No analytics on dismiss-rate / engagement.
  Would need to design opt-in privacy-respecting metrics
  (out of scope for current project positioning).
- **[info]** UA-sniffing is fundamentally fragile. As
  browsers evolve, detection might break. Mitigation:
  conservative detection (4 dimensions) + graceful no-op
  fallback (banner doesn't show if uncertain). Acceptable.

## §4 Lessons learned (cycle-level, not §5.1)

1. **Template-driven cycle kickoff is the new normal**.
   Cycle-8 validates Track F template; future cycles
   should default to copy-template-and-edit. Estimated
   time-savings: ~30 mins/cycle (no re-derivation of v3.2
   spec).
2. **"No-gap" not-borrows are a maturity signal, not a
   process failure**. Project's existing baselines (form
   UX, a11y) are at parity with industry sources. Future
   cycles should track "what fraction of surveys produce
   no-gap findings" as a maturity metric.
3. **3rd consecutive single-cycle execution = default
   expectation**. Document this in next cycle's kickoff:
   "Multi-cycle execution requires explicit justification
   (e.g., > 4 substantive ship tracks)."
4. **PWA install complements (R247 desktop, R248 iOS) are
   architectural pair**: any future PWA-install work
   should consider Android Chrome (covered by R247),
   iOS Safari (covered by R248), and possibly
   PWABuilder-style enhancement (TWA / packaged-app
   wrappers) — but only if user demand emerges.
5. **`feature-mining-cycle-kickoff-template.md`'s "Usage
   notes" section should be deleted on copy**. cycle-8
   doc inadvertently kept it; minor doc-hygiene gap.
   Worth a `<!-- DELETE ON COPY -->` marker or init
   script.
