### 2.3 R218 — multi_task.js → _debugLog migration (e7826b7)

**Files changed**: 1 source + 1 test-update + 1 Node harness fix
+ 1 CHANGELOG entry.

**Surface**: `multi_task.js`'s 46 `console.log` calls live in the
deep link / polling / SSE / task add/remove/sync / retry /
countdown / tab bar render paths — extremely high volume in
multi-task mode (one user reported "browser console scroll
flooded in 2 seconds").

**Diff intent**: NOT the same `console.log → console.debug`
rename used by R216/R217. Instead, migrate to the file's
pre-existing `_debugLog()` helper:

```js
function _debugLog() {
  if (!window.AIIA_DEBUG || typeof console === "undefined"
      || typeof console.debug !== "function") return;
  try { console.debug.apply(console, arguments); } catch (_) {}
}
```

**Why different strategy**: `console.debug` is suppressed in
default DevTools view but appears under "Verbose"; `_debugLog`
is **fully silent** until `window.AIIA_DEBUG = true` is set in
the console. This is the right discipline for hot-path logging
in deep features (multi-task is opt-in, but when used emits
events at ~10Hz; `console.debug` alone would still hurt
DevTools performance under "Verbose").

**Test coverage**: R217's invariant test was extended in the
same commit:
- `MULTI_TASK_BUDGET` tightened: `50` → `0`. Any reintroduced
  `console.log` in `multi_task.js` now fails CI immediately.
- New `test_multi_task_js_uses_debug_log_helper` invariant
  asserting `_debugLog(` call count ≥ 45 (one less than 46 to
  allow for one renamed call site, but proves the migration
  was a migration not a deletion).

**Cross-cycle dependency fixed**: `tests/test_init_parallel_
fetch_r22_3.py` uses a Node.js subprocess to drive
`initMultiTaskSupport`. The harness was injecting only the
init function's body. R218's migration changed `console.log`
inside the body to `_debugLog`, but `_debugLog` is defined
outside the body (at file top), so the harness emitted
`ReferenceError: _debugLog is not defined`. **Fix**: harness
now injects `globalThis._debugLog = () => {};` no-op stub
before running the body. This is the right level of fix —
the harness's purpose is to test parallel-fetch semantics,
not debug-log behavior, so stubbing is semantically equivalent.

**Audit findings**: ✅ all good. The strategy switch (different
helper for hottest file) is well-reasoned and documented in
the CHANGELOG entry. The Node harness fix is exactly the
right scope. The `MULTI_TASK_BUDGET = 0` tightening converts
R217's "temporary carve-out" into a "ratchet" — the budget
can only go down, never up, without a deliberate test edit.

**Risk surface**: minor — the `_debugLog` helper's silent-when-
`AIIA_DEBUG = false` semantics could theoretically swallow a
genuine alert. Mitigated by the fact that **none of the 46
demoted calls were warn/error severity** (verified by reading
each call site in the original diff).

## 3. Cross-cutting observations

### 3.1 Invariant test pattern compounded

CR#23 identified the invariant test pattern as a Cycle 10
maturation. Cycle 11 is **the first cycle where ≥ 4/5
commits ship a dedicated invariant test file** (R216, R217,
R219, R220 each have their own; only R218 piggy-backs on
R217's test file). The pattern is now the **default first
question** when authoring a change: "what's the silent
decay vector here, and how do I lock it?"

Three Cycle 11 invariants particularly worth highlighting:

- **R217's forward-compat invariant** (`test_new_files_
  default_to_strict`): future-proofs the rule against new
  JS files being added under `static/js/` without
  classification.
- **R220's metric-name parity invariant**: cross-tool
  binding between dashboard JSON and Python source —
  protects against the most likely future drift (refactor
  rename in `system.py` not propagated to dashboard).
- **R218's `_debugLog(` count ≥ 45**: proves migration
  actually happened, not "all calls deleted". This is
  the **"proof of work"** style — guards against the
  silent decay where someone "fixes" a noise complaint
  by deleting all log lines instead of demoting them.

### 3.2 Bilingual sync discipline held

R220's docs are double-language (EN + zh-CN README). No
cycle 11 commit broke bilingual lockstep — `docs/
configuration.{md,zh-CN.md}` was not touched, `docs/api/`
+ `docs/api.zh-CN/` not touched. The zh-CN README for
R220 was written from scratch in same commit, not
machine-translated post-hoc.

### 3.3 No VSCode extension changes — intentional

`packages/vscode/` was touched **only** in R217 (byte-
parity fix for `webview-state.js`). No other Cycle 11
commit needed extension updates because:
- R216/R218: extension doesn't use `notification-manager.
  js` or `multi_task.js`.
- R219: lint hook is repo-only.
- R220: dashboard is /metrics-side, not webview-side.

This is correct minimal-touch discipline.

### 3.4 Pre-commit hook surface area grew safely

Cycle 11 added 1 new pre-commit hook (R219). Combined
with R209 (Cycle 10), the project now has **5 custom
hooks** beyond ruff/format/etc:
- `check-changelog-diff-scope` (CR#16 F-4 era)
- `check_tag_push_safety` (R209)
- CSS brand-color drift (R66 era)
- CSS string-quote drift (R174 era)
- CHANGELOG inline-code style (R219)

All are file-scoped (`files: ^...$` regex), so cold-start
cost on unrelated PRs is zero. R219's hook is the first
hook scoped to a **single** file (`^CHANGELOG\.md$`) —
sets the pattern for future single-file lint hooks

## 5. Versioning / release decision

**Context — three-cycle release accumulation**:

Cycle 9 (CR#22) recommended `v1.7.3` — **never tagged**.
Cycle 10 (CR#23) recommended `v1.7.4` stacked on `v1.7.3` —
**never tagged**. Cycle 11 (this CR) is the **third
consecutive cycle without a release**. The `v1.7.2` baseline
tag is now 17 commits behind `main` (R203 through R220 +
CR#22 + CR#23 + CR#24-to-be).

**This is a release-cadence problem, not a commit-quality
problem**. The project's typical 4-5-commit release cadence
broke down because each cycle's CR scheduled "release after
CR lands" but the release step itself was never executed.

**Recommended**: pack Cycles 9+10+11 into a single catch-up
`v1.7.3` release.

Rationale for **using `v1.7.3` (not `v1.7.4` or `v1.8.0`)**:

- `v1.7.3` was the next scheduled patch version; using it as
  the catch-up tag keeps the version stream gap-free and
  avoids "what happened to 1.7.3?" reader confusion forever.
- All 17 commits are individually backward-compatible: R203
  defensive cap (no-op for normal load), R204/R207 new
  Prometheus metrics (additive), R205 opt-in env-var toggle
  (default off), R206 docs only, R208 pure refactor verified
  by tests, R209 opt-in pre-push hook, R210/R211 docs/CHANGELOG,
  R212-R215 test-only, R214 UI bug fix improving behavior,
  R216-R218 console-noise cosmetic, R219 lint guard, R220 docs
  + tests.
- Cumulative CHANGELOG `[Unreleased]` content is large but
  **none of it is a major-version-worthy breaking change**.
  Per project precedent (R199 / R204 historically shipped as
  patches despite being ops-significant), patch is correct.
- 17-commit release is unusual but **CHANGELOG entries are all
  well-formatted and individually self-describing**, so a
  single rollup release notes will read coherently.

**Alternative**: `v1.8.0` if the cumulative new ops surface
(R205 env var + R204+R207 new metrics + R209 pre-push hook +
R220 dashboard) is judged "minor-worthy". Defensible per
Semantic Versioning §7, but project history strongly
precedents patch-level releases for ops-additive surface.

**My recommendation**: `v1.7.3`. Schedule release immediately
after CR#24 lands on `main`.

**Release prerequisites checklist** (per `docs/release-recovery.
md` R206 pre-flight):
- [ ] CR#24 commit landed on `main`.
- [ ] `pyproject.toml` version bumped: `1.7.2` → `1.7.3`.
