## 2. Highlights

### 2.1 `8c5bef1` custom-sound duration cap — pre-flight design is right

Issue identified by cr33 §8 #1 [medium]: 700 KB filesize cap could
still admit 30-minute lo-bitrate audio; decoded PCM at 44.1kHz
stereo = ~150 MB persistent `audioBuffers['custom']` memory. Real
foot-gun for legitimate users.

The fix design is correct:

1. **Pre-flight `decodeAudioData` before `localStorage.setItem`.**
   The author rejected the simpler "post-storage check + rollback"
   path. Pre-flight means: if duration > 30s, localStorage is
   **never** touched, so the user's previously-saved custom sound
   is preserved and no clean-up race exists.
2. **`audioContext` null-check.** If the AudioContext failed to
   initialize at all, the pre-flight check is skipped (falls
   through to old path). This preserves test environments and
   browsers where AudioContext is unavailable but localStorage
   still works.
3. **Source-order invariant test** (`test_duration_check_constant_
   and_callsite`): asserts that within the `saveCustomSoundFromFile`
   body, `.duration` reference appears **before**
   `localStorage.setItem`. This is the right discipline — it
   catches a future refactor that accidentally moves the duration
   check after the storage write (which would re-introduce the
   "pollute then rollback" anti-pattern).
4. **6 → 7 error codes.** The new `duration_too_long` integrates
   cleanly into the existing error-code dispatch in
   `settings-manager.js`. i18n keys added across all 4 locales
   (en/zh-CN/zh-TW auto-regenerated/pseudo).

Minor scope-creep that **didn't** happen:

- No "show duration in upload UI before confirm" preview — would
  require another decode pass + UI surface. Right call: error
  message tells the user what happened, retry is one click.
- No per-file `volume` / `playback_rate` adjustment despite being
  a natural next step — would expand the API surface
  significantly; deferred until user request.

### 2.4 `184c935` mining-2 §3.4 ship — the minimum-viable feature done right

mining-cycle-2 §3.4 was scoped as "LOW ROI, ~50 LoC". Actual
ship: **~70 LoC application code + ~135 LoC test = 205 LoC**;
test:code ratio is ~2:1, very high for a low-ROI feature, but
appropriate because the **invariant** (single-click reserved for
task-switch) is more important than the **behavior** (dblclick
copies).

Design highlights:

1. **Dual-path clipboard with proper fallback.** Primary:
   `navigator.clipboard.writeText` (async). Fallback:
   `document.execCommand("copy")` via off-screen hidden
   `<textarea>` (sync, deprecated but still works on Firefox / iOS
   Safari / loopback HTTP). This covers ~99% of users without
   adding a "your browser doesn't support copy" UI surface.
2. **Reuses existing `status.copied` / `status.copyFailed` i18n
   keys.** No new strings added across 4 locales. The author's
   commit message explicitly calls this out as "不引入新 i18n
   key (现有 status.copied / status.copyFailed)" — this is the
   right discipline for low-ROI features where i18n debt is the
   real ongoing cost.
3. **`dblclick` not single-click.** Single-click reserved for
   task-switch (existing core interaction). The commit message
   explains: a new icon button would clutter the task tab UI;
   dblclick is the universal "alternate action" idiom; the
   existing `title` tooltip ("hover to see full task_id")
   already provides the discovery affordance.
4. **`stopPropagation` on the dblclick handler.** Without this,
   the double-click would fire two `click` events first which
   would switch task twice. The handler is doing exactly the
   right thing: `e.preventDefault()` + `e.stopPropagation()` to
   contain the event entirely.
5. **`data-copyable-task-id` attribute on `textSpan`.** Provides
   a CSS-selector hook for future styling (e.g. cursor change
   on hover) and a test hook for browser-automation regression.
6. **`window.copyTaskIdToClipboard` exposed.** Makes the helper
   externally callable from console / E2E tests / hypothetical
   future "right-click menu" features.

Test design highlights:

1. **`TestAntiRegression::test_no_single_click_hijack`** —
   asserts that the count of `textSpan.addEventListener("click"`
   in `multi_task.js` is exactly **0**. This is the cycle's most
   important invariant: a future PR can't quietly add a single-
   click copy without this test failing. It's a **structural**
   guarantee that the design decision ("single-click stays
   task-switch") will survive code churn.

Minor scope-creep that **didn't** happen (and was correctly
documented):

- `project_directory` copy: the doc honestly says "project_
  directory is not surfaced in the UI at all; would need a
  separate 'show project info' UI feature first". Correct
  scope: don't ship a copy affordance for content that has no
  visible source.

## 3. Issues found

### 3.1 `8c5bef1` custom-sound duration cap [INFO]

- **`decode_failed` and `duration_too_long` error codes are emitted
  from the same `audioContext.decodeAudioData` path.** If the
  decode itself fails for unrelated reasons (e.g. corrupted
  WebM), the user sees `decode_failed`; if it succeeds but is
  too long, they see `duration_too_long`. This is correct
  semantics. **No action**.

- **Two decodes per save** (once for pre-flight, once via the
  `loadCustomSoundFromStorage()` call at the end). For a 700 KB
  upload this is ~20-60 ms total on a modern CPU; negligible
  for an explicit-user-action path. **No action**.

- **i18n key naming inconsistency**: existing error keys use
  `tooLarge` (camelCase tail), new key is `durationTooLong`.
  Consistent with the pattern. ✅

### 3.2 `4700703` CONTRIBUTING + race comment [LOW]

  rewritten), this reference will become stale and confusing.
  **Action**: add a small comment in the markdown source telling

### 3.5 `184c935` one-click copy [LOW]

- **`document.execCommand("copy")` is deprecated.** All major
  browsers still implement it, but spec-wise it's been moved to
  "legacy non-standard" since 2024. Long-term, the
  `navigator.clipboard.writeText` path will be the only one,
  and the fallback path becomes dead code. **No action** for
  now — keep both for the next 12-24 months while old browser
  share is non-trivial.

- **No visual feedback on the source element.** The toast tells
  the user "Copied", but the `textSpan` itself doesn't flash /
  briefly change color / show a check icon. For a power-user
  feature like this, more visual confirmation that **this exact
  element** is what got copied (vs. some other concurrent UI
  action) would help. **Priority: low** — cosmetic improvement.

- **No keyboard shortcut.** Power users who navigate via tab
  switching might want `Ctrl+C` on a focused task tab to copy
  the ID. We don't have keyboard focus on task tabs currently.
  **Priority: low**, defer until we have keyboard nav for
  tabs.
