## 1. R221 — packages/vscode/ console.log invariant ✅

**Status**: ✅ Pass. The cycle's opening commit closed the
F-cycle11-1 follow-up from CR#24 — "extend the console.log
demotion discipline (R216–R218) from `static/js/` to
`packages/vscode/`". Discovery surprised: the VSCode extension
codebase was **already at zero `console.log`** in project-owned
files. The actually-shipped deliverable shifted to **locking that
healthy state in place** via a forward-compat invariant test:

```python
# Conceptually:
project_owned_js = scan_packages_vscode_for_js()
for file in project_owned_js:
    assert count_console_log(file) == 0, file
# Plus: a strict default for unclassified new files
```

**Strengths**:

- **Inverted-finding handling**. The commit message and CHANGELOG
  entry both correctly frame the discovery as "no debt found, so
  the deliverable is the invariant to prevent regression". This
  is the right post-mortem framing — F-cycle11-1's original premise
  (assumed `console.log` debt) was wrong, but the resulting invariant
  is still high-ROI because the next contributor adding noisy
  logging will be caught at CI time.
- **Robust skip-list with whole-segment matching**. After a mid-test
  failure flagged `dist/applescript-executor.js` etc. as
  "unclassified" (TS compilation output), the skip logic was
  refined to wrap matches in `/`-bounded fragments
  (`"/" + rel_str + "/"` against `"/dist/"` / `"/out/"`) so partial
  filename matches (e.g. a real source file named `distinct.js`)
  cannot be silently skipped.
- **Ruff PIE810 cleanup mid-loop**. The fix for chained
  `p.startswith()` checks (`p.startswith(("dist/", "out/"))`) shows
  the cycle's normal practice of treating lint warnings as
  must-fix, not "we'll get to it later".

**Concerns**:

- _None of critical severity_. One nice-to-have: the test's
  forward-compat default (`assert_no_unclassified_js_files`) could
  be stricter — currently it only fails when there's a brand-new
  unclassified file under `packages/vscode/`, but if a contributor
  adds an unclassified file to a deeply-nested directory not yet
  scanned, the test silently passes. Mitigation: the
  `unclassified` definition transitively walks
  `packages/vscode/**/*.{js,mjs,cjs}`, so this is mostly theoretical.

**Verdict**: clean +1. Stencils a useful "lock in zero" pattern for
future quiet-state milestones.

## 3. R223 — Web UI keyboard-shortcut discoverability hint ✅

**Status**: ✅ Pass. Closed a discoverability gap surfaced by
cross-reading `web_ui.html`'s settings panel ("Common shortcuts"
listing 5 input-workflow shortcuts) against
`keyboard-shortcuts.js`'s `showHelp()` (registering 12 shortcuts
total — 7 navigation shortcuts were invisible). R223 adds a
one-line hint beneath the settings list pointing users at
`Ctrl+/` (`Cmd+/` on macOS) for the full reference, noting the
output lands in the browser DevTools console.

**Strengths**:

- **Tiny surface change, high UX payoff**. One new i18n key
  (`settings.shortcutsFullHelpHint`) in 2 locales, one new `<div>`
  with `data-i18n` attribute, one pseudo-locale regeneration.
  Zero code changes — uses the existing `Ctrl+/` binding already
  registered in `keyboard-shortcuts.js`.
- **Hint copy mentions BOTH the binding AND the output location**.
  Without the "browser console" qualifier, users would press the
  key, see no visible UI change, and conclude the shortcut is
  broken. The invariant test
  (`test_settings_shortcuts_full_help_hint_invariant_r223.py`)
  asserts both phrases appear in both locales — locking in this
  copywriting insight against future translators.
- **i18n discipline followed end-to-end**. New key registered in
  `en.json` + `zh-CN.json`; pseudo locale regenerated;
  `data-i18n` attribute uses the key; precompressed mirrors will
  be regenerated next time `ci_gate.py` runs (was bundled with
  R224's commit a few commits later).

**Concerns**:

- _One small drift identified mid-cycle and fixed in R224's commit_:
  R223 did not regenerate `en.json.br` / `zh-CN.json.gz` (the
  precompressed mirrors). The pre-commit hook does not currently
  enforce mirror freshness — `ci_gate.py` catches it. R225/R224's
  commit bundled the refresh as a tagged side-effect rather than
  amending R223, which is the right call (R223 was already pushed).
  → Follow-up F-cycle12-2: consider promoting the precompress
  freshness check from `ci_gate.py` to a pre-commit hook to avoid
  the lag between commit and detection.

**Verdict**: clean +1. Pattern worth repeating: when adding a new
UI binding, check both the discoverability surface (settings panel,
help dialog) and the output surface (where the user looks for the
effect).

## 4. R224 — Grafana per-provider notification drill-down ✅

**Status**: ✅ Pass. Closed CR#24's F-cycle11-3 follow-up. R220
shipped the overview dashboard with one aggregate Notification panel
(`aiia_notification_delivery_success_rate` + `_queue_size`); the
overview answers "is the notification subsystem healthy?" but not
"_which_ provider is broken right now?". R224 adds the drill-down
counterpart:
`docs/observability/grafana-dashboard-notification-providers.json`
(6 panels, schemaVersion 38, uid `aiia-notification-providers-r224`)
covering per-provider attempts rate / success rate / avg latency /
P95 send duration (R191 histogram) / failure streak (R145) /
success streak.

**Strengths**:

- **Drill-down vs overview symmetry**. R220 and R224 are designed
  to be imported together: ops on a degraded overview alert clicks
  through to the drill-down without context-switching tools. Both
  use `schemaVersion 38`, the same `${DS_PROMETHEUS}` template
  variable, the same panel-grid conventions (2-col layout, table
  legends, thresholds-on-line), making the two feel like one
  family.
- **Smart metric-parity invariant for f-string-assembled metrics**.
  The R220 invariant pattern (every `aiia_*` in panel exprs must
  substring-exist in `system.py`) had to be extended for R224
  because per-provider notification metrics are emitted via
  `f"aiia_notification_{metric_suffix}"` and the full literal never
  appears in source. The R224 test's solution — fall back to
  checking `f'"{suffix}"' in source AND "aiia_notification_" in
  source` — captures the contract correctly: any future contributor
  renaming the suffix tuple in `_per_provider_field_specs` must
  also update the dashboard. Documented inline so the next person
  understands why the fallback exists.
- **Per-panel `provider` breakdown invariant**. Without this, a
  future "simplification" could remove the `sum by (provider)`
  aggregation from a panel, silently turning the drill-down into
  a duplicate of the overview's aggregate view. The invariant
  test asserts every panel either uses `sum by (provider` or
  carries `{{provider}}` in its legendFormat.
- **Bundled README updates in both languages with the same per-
  panel rationale table**. Mirrors R220's approach. Cross-link
  back to R220's README + cross-link back to CR#23 (F-cycle10-4
  origin) and CR#24 (F-cycle11-3 origin) makes the provenance
  chain readable from any document in the chain.

**Concerns**:

- _None blocking_. The drill-down doesn't ship its own alert rules
  — the same deliberate choice as R220 (ops conventions differ
  too much to ship a one-size-fits-all alert YAML). Documented in
  the README's `## Companion dashboards` section.

**Verdict**: clean +1. Now the observability story has both
"big-picture health" (R220) and "per-provider drill-down" (R224)
covered. Future R-cycle can add the SSE-bus per-event-type
drill-down using the same template.

## 5. R225 — SSH / WSL remote-environment detector ✅

**Status**: ✅ Pass. Closed a real UX gap surfaced by the very
README table R222 just added: `mcp-feedback-enhanced` advertises
"intelligent SSH Remote / WSL detection" while AIIA's startup
just printed `请在浏览器中打开: http://127.0.0.1:8080` on default
bind — unreachable from a user's local browser when running on
SSH. New module `src/ai_intervention_agent/remote_environment.py`
provides `detect_remote_environment()` returning a
`RemoteEnvironment` TypedDict; the `web_ui.py` startup banner
now emits actionable hints (`ssh -L PORT:127.0.0.1:PORT
user@remote_host` recipe, plus `AI_INTERVENTION_AGENT_WEB_UI_HOST=0.0.0.0`
alternative) when SSH is detected with loopback bind. WSL gets a
note about WSL2 automatic localhost forwarding.

**Strengths**:

- **Pure-function detector, fully testable**. Probes
  `SSH_CONNECTION` / `SSH_CLIENT` / `WSL_DISTRO_NAME` /
  `WSL_INTEROP` env vars and at most one well-known file
  (`/proc/version` for WSL1 fallback). No socket, no subprocess.
  All edge cases (empty strings, whitespace-only values,
  FileNotFoundError, PermissionError, UnicodeDecodeError) are
  explicitly tested in 19 cases.
- **Purely additive integration**. The new startup hint never
  auto-rewrites `host`, never auto-forwards ports — those would
  be footguns for LAN-only and mDNS-enabled paths. We _tell_
  the user what to do; the user decides. Behavior preservation
  is total.
- **Integration guard tests**. Three of the 19 cases assert the
  `web_ui.py` integration didn't silently disappear: import line
  present; detector called; hint mentions the `ssh -L` recipe.
  Without these, a future refactor could remove the hint block
  and nothing would scream at CI.
- **Direct competitive-feature absorption from the same cycle's
  R222**. R222 documents the gap, R225 closes it. Clean cycle
  arc.

**Concerns**:

- _Low-severity_: the existing banner copy is Chinese-only
  (`请在浏览器中打开: ...`), and R225's new hints follow the same
  Chinese-first convention to match. This is consistent with the
  existing CLI/server-log style (Web UI is i18n'd, CLI/banner is
  Chinese) but it does mean an English-only user on SSH still gets
  Chinese hints. → Follow-up F-cycle12-3: bilingual startup banner
  with `--lang` / `AI_INTERVENTION_AGENT_LOG_LANGUAGE` env var.
  Not blocking for v1.7.4 — defer to v1.8.0 unless a user files
  an issue.
- _Style-only_: the detector module is registered in
  `MODULES_TO_DOCUMENT` + `QUICK_NAV_UTILITY` and auto-renders
  `docs/api/remote_environment.md`. Render is correct, but the
  module's docstring section that ends up in the API doc could be
  slightly more user-facing — it's currently written for
  contributors. → Polish item, not a follow-up.

**Verdict**: clean +1. Sets a template for future "absorb a
competitor feature mentioned in README" cycles: feature lives as a
module with its own invariant tests + bilingual docs entry, never
as ad-hoc inline code in `web_ui.py`.
