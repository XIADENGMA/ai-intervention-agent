## 2. 维度突破

### 2.1 API contract — 完成 request+response 双向锁定 (R378)

R355 → R358 → R364 → R368 → R378 形成完整的 5 应用 lineage, **api
contract 维度从启动 (1st) → 4 应用工业化 → 5 应用完全深化**:

| Pass | R#    | 维度                             | 锁定方向        |
| ---- | ----- | -------------------------------- | --------------- |
| 1st  | R355  | 必有 tags/responses/summary 三件套 | -               |
| 2nd  | R358  | POST 必有 ≥1 个 4xx/5xx 文档     | error path      |
| 3rd  | R364  | tag ∈ 封闭集合 + exactly 1       | taxonomy        |
| 4th  | R368  | Pydantic field 曝光分类           | response schema |
| 5th  | R378  | OpenAPI parameters ↔ handler 消费 | **request schema** |

**完成 OpenAPI 文档质量完整 5 维**:
- coverage (1st) — 文档存在性
- error path (2nd) — 失败路径覆盖
- taxonomy (3rd) — 分类一致性
- response schema (4th) — 出参完整性
- **request schema (5th)** — 入参一致性

R378 在实施中验证现状: 所有 POST endpoint 的 OpenAPI parameters 都被
handler 内引用 (无虚假文档 — 文档化了 client 实际不能用的字段), 也无
silent acceptance (handler 接受了 OpenAPI 未文档化的字段)。

### 2.3 Security header strict mode — 2nd 进入巩固期 (R386)

R376 (cycle-42 1st) 锁了 7 个 top-level header **外壳**, 但 CSP header
是单一 header **内含 10 个 directive**, R376 只能锁 header 出现 + nonce
模式, 不能锁内部 directive 完整性。R386 补齐这层:

| 层级       | R#    | 锁定内容                                            |
| ---------- | ----- | --------------------------------------------------- |
| 外壳       | R376  | 7 个 top-level header 出现 + 关键值                 |
| 内核       | R386  | CSP 内部 10 个 directive + 关键值 + 禁用 source-expr |
| nonce 同步 | R306  | response ↔ template ↔ JS reader 三层一致           |

**禁用 source-expression 锁定示例**:
- `frame-ancestors 'none'` 禁止放宽到 `'self'` (clickjacking 核心)
- `object-src 'none'` 禁止放宽到 `'self'` (Flash/object XSS 防御)
- `script-src` 禁止含 `'unsafe-inline'` / `'unsafe-eval'`
- `default-src` 禁止 `*`

**Security header 三层完整覆盖**形成, 进入巩固期标志。

## 5. 跨 cycle 风险与机会

### 风险 A: Pydantic validator coverage 当前 whitelist 总量

R380 + R384 共 6 个 whitelist 字段 (NotificationConfig 3 +
Task 3), 全部有 rationale + future plan 说明 "clamp 当前在 route /
persistence layer 实施"。这是**短期可接受的负债**, 但长期需要把 clamp
逻辑迁移到 model layer `@field_validator(mode="before")` 消除"back door"
(直接构造 Pydantic 模型时绕过约束)。

**缓解措施**: 已用 invariant 强制 whitelist 条目必须含 rationale + future
plan (R384 layer 3 测试 rationale ≥50 字符), 防止 silent acceptance。

**优先级**: P2 (中期, 不影响当前运行时正确性, 但消除最后一层 silent
bypass surface)。

### 风险 B: cr72 §4 提到的 ATEXIT_GRACE_PERIOD 实测验证仍未补

R372 锁定 1.5s 为决策值 + inline rationale ≥100 字符, 但没有 micro-
benchmark 实测 (P95 / P99 / 极端 case)。建议 future cycle 加 R-bench
微测量, 决定是否调整为 2.0s 或保持 1.5s。**未变化**, 优先级 P3。

### 风险 C: cycle-43 第 1 commit 是 R378 (而非 R377)

cycle-42 之后下一个 R 编号本应是 R377, 但 cycle-43 跳到 R378 (per
TODO.md plan; R377 编号被保留给未来 patch)。**这是 plan 设计选择,
非缺陷**, 但需要在 future cr 中说明 R-number gap 含义, 避免被误读成
spillover。

### 机会 D: Pydantic validator coverage 3rd app 候选

- **server_config.WebUIConfig**: 含 port (int) / host (str) / 等字段,
  validator 现状不明
- **server_config.FeedbackConfig**: 含 auto_resubmit_timeout 等

  TOML 端配置, Annotated 兜底, 但仍可补显式 @field_validator
- 3rd app 可在 cycle-44 推进, 完成"3 应用工业化"阈值, 与
