## §4 Architectural decisions worth highlighting

### §4.1 R250 anti-FOUC is a "lifetime fix"

The pattern (head inline + body defer + idempotent writes)
applies to **any** future `<html data-*>` JS attribute.
Codify in CONTRIBUTING:

> **FOUC checklist for `<html data-*>` writers**: every
> module that writes a `data-*` attribute to `<html>` from
> JS must be paired with a synchronous inline `<head>`
> pre-write. Test pattern: see
> `tests/test_feat_perf_audit_cycle3_anti_fouc.py`.

**Follow-up #5 [medium]**: Add this checklist to
`CONTRIBUTING.md` (anchor for future PR review).

### §4.2 PWA offline 3-tier graceful degradation

```
navigation request → SW intercepts
  ├─ network success → respond + (no cache)
  ├─ network fail + cached offline.html → respond cached
  └─ network fail + cache miss → makeOfflineResponse 503
```

Each tier preserves user experience proportional to what's
available. **Reusable pattern** for future SW navigation
strategies (e.g., chat history offline display).

### §4.3 Permanent defer with concrete unlock = better than silent drop

Voice input has been on the TODO since cycle-6 (4 months
ago). Two outcomes were possible:
- Silent drop (just stop carrying it; user might re-
  propose, repeat survey work)
- Formal defer with unlock (current choice)

**Formal defer wins** because:
1. New agents can grep "voice input" → find unlock criteria
   → know not to re-propose without evidence
2. Audit trail preserved (cycle-6 → cycle-9 lineage)
3. Unlock criteria are **falsifiable** — if a GitHub issue
   appears tomorrow, we can immediately reopen with §5.1
   as the spec
