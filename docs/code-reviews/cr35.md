## 2. Highlights

### 2.1 `9131241` session-link copy — the right kind of refactor in a feature commit

mining-cycle-2 §3.2 needed `copyTaskLinkToClipboard`, which
would naturally have inlined the same dual-path clipboard logic
that `copyTaskIdToClipboard` already had. Instead, the author:

1. **Extracted `_writeToClipboard(text)` first** as a private
   low-level helper. Both `copyTaskIdToClipboard` and the new
   `copyTaskLinkToClipboard` call it.
2. **Added `buildTaskDeepLink(taskId, base?)`** using the `URL`
   API instead of string concatenation. This avoids stale
   `task_id` query params (URL handles `searchParams.set`
   correctly) and handles hash fragments / port / etc.
3. **`Shift+dblclick` modifier** in the existing dblclick
   handler. The branch is one `if (e.shiftKey)` — minimal
   complexity addition, and the modifier follows the universal
   "alternate variant" idiom from IDE workflows.
4. **Anti-duplication invariant** `test_no_duplicated_
   execcommand` asserts `document.execCommand("copy")` appears
   exactly **1** time in `multi_task.js`. This is the **right**
   refactor lock-in test: a future "let me revert the helper
   extraction for clarity" PR fails the invariant before
   reviewers see it.

This is the discipline that prevents code duplication from
becoming a maintenance burden: the moment a third caller would
need the same logic, the refactor *already happened* in the
second-caller commit.

### 2.4 `0c8aa7f` export-ui — closing the only real cycle-2 §3.1 gap

After correcting §3.1 to "already shipped (backend); UI button
missing", the author immediately shipped the missing UI in the
same cycle. This is good cycle hygiene — don't let a corrected
gap sit waiting for cycle-3.

Design highlights:

1. **`<select>` + `<a download>` instead of JS-driven fetch**.
   The `download` attribute lets the browser do the heavy
   lifting (progress bar, large file streaming, file-save
   dialog). A JS-driven `fetch` → `Blob` → `URL.createObjectURL`
   path would put the entire JSON in memory before the user
   sees a save dialog, which is exactly the kind of memory
   hazard cr33 §8 #1 just closed for custom-sound.
2. **Format whitelist + `encodeURIComponent` double defense**.
   `fmt === "markdown" ? "markdown" : "json"` ensures `?format=`
   value is one of two known strings; `encodeURIComponent`
   adds belt-and-suspenders against future format additions
   that forget to escape. `test_wire_only_allows_known_formats`
   and `test_wire_uses_encode_uri_component` lock both
   defenses.
3. **`rel="noopener"`** on the anchor. Download-attribute
   anchors don't usually trigger window.opener-style attacks,
   but loaders that misconfigure this have leaked tab opener
   references in the past. `noopener` is free defense.
4. **5 new i18n keys × 4 locales**. Generated, not hand-written
   for zh-TW / pseudo.

The HTML lives next to the existing "Reset settings" button in
the Notifications & system tab, which is the right home: low-
frequency feature, settings is where users go to find
"administrative" actions.

### 2.5 `299e619` zhconv-eval — the right way to reject a popular suggestion

cr33 §8 #2 asked for an evaluation of `zhconv` / `opencc-python`
as a future replacement for the inline `CHAR_MAP_v2` pipeline
when it grows to ~600 entries (currently 445). The investigation:

1. **Reproducible methodology** in §1: 7-line Python script that
   anyone can re-run after `uv pip install zhconv`.
2. **Quantitative result**: 102 / 297 strings match (34.3%).
   The remaining 195 are not "wrong vs right" — both are
   legitimate Traditional Chinese — but **different style
   choices**. zhconv chooses character-faithful (e.g. "反饋"),
   inline chooses Taiwan UI-conventional (e.g. "回饋").
3. **Failure-mode analysis**: zhconv's failure mode (missing
   localization, but characters always Traditional) is
   **invisible** (users read it and feel awkward but don't
   report bugs). Our current failure mode (missing character
   maps → Simplified leaks through) is **visible** (users see
   Simplified glyphs immediately and can report). Visible
   failure is preferred for a maintained pipeline.
4. **Hybrid design proposal** in §5 (inline phrase first +
   zhconv char fallback) is documented as a **future track**
   with explicit trigger conditions (CHAR_MAP_v2 ≥ 600 AND ≥
   3 observed Simplified-residue bugs). Not adopted now.
5. **Rejection rationale** in §6 maps cleanly to the four
   reasons in §4: style control beats character coverage,
   growth curve is sublinear-not-exponential, dev-only build
   maintenance is acceptable, and inline gives 100% predictable
   adopt-rate.

This is **why investigation artifacts matter**: the cr33
follow-up could have been a one-line "evaluated zhconv, sticking
with inline" but that gives a future reviewer no way to
re-derive the decision. The doc lets the next person re-run the
exact comparison and either confirm or update based on whether
the gap closed.

### 3.4 `0c8aa7f` export-ui [LOW]

- **No "Export progress" indicator**. Browser handles the
  download progress, but if the dataset is large and the
  user's browser doesn't show a per-domain progress UI, they
  may click multiple times. The endpoint is rate-limited
  (30/min), so accidental triple-clicks won't hammer the
  server, but the UX is suboptimal.
  **Priority: low**; competitor has the same gap.

- **`download` filename is determined by server's Content-
  Disposition header**, which uses an ISO timestamp. A user
  doing a 5-minute-apart export will get two files with
  identical timestamps (server uses second resolution).
  Cosmetic; not worth fixing unless reported.

- **No "Open after download" affordance** for Markdown.
  Browser default behavior; we don't control it.

### 3.5 `299e619` zhconv-eval [INFO]

  3 cycles (i.e. zero growth this period because the conversion
  is now stable). The threshold may never be hit. **No
  action**; the doc itself acknowledges this in §4.

- **The hybrid design (§5)** is the most interesting future
  work in this doc. If we ever hit the trigger, the design is
