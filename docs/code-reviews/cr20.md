The two most thematically important shifts in this cycle:

1. **Schema-driven event flow** (R198) — the SSE bus was historically free-form `(event_type: str, data: dict | None)`, with frontend / VSCode subscribers having no source-of-truth. 2. **Observability extension** (R199) — R195 made API token rotation possible without restart, but R199 closes the loop by making rotation _queryable_. R197 is pure test infrastructure, but among the strongest invariant tests in the repo (it uses `ast` to enforce that two metric updates remain co-located in the same `_stats_lock` block, preventing future refactors from accidentally separating them).

## 2. Detailed Review

### 2.1 R196 — Notification-specific latency buckets

**Commit**: 3cda85d
**Files**:
- `src/ai_intervention_agent/notification_manager.py` (+8 / -8): replaced bucket tuple
- `tests/test_notification_latency_histogram_r191.py` (+2 / -2): refer to bucket template via name, not hardcoded values

**What it does**:

R191 originally reused `mcp_tool_call_metrics._DEFAULT_LATENCY_BUCKETS` (`0.1, 0.5, 1, 5, 30, 120, 300, 600`) for notification latency — the same "human-interaction" semantic family. But empirical distribution differs sharply:

- MCP tool calls are dominated by *"read question → type feedback"* (~10–300s typical, up to 600s `auto_resubmit_timeout`).
- Notifications are dominated by *"network RTT + provider processing"* (~50ms–500ms typical, ≤ 10s tail).

R196 replaces the notification bucket tuple with `(0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0)` — dense in the 50ms–10s range where notification senders live.

**What's good**:

- The bucket choice is rooted in empirical reality, not arbitrary. The CR#19 §4.1 follow-up note explicitly recommended this exact tuple.
- `_DEFAULT_LATENCY_BUCKETS_SECONDS` lives as a class-level constant on `NotificationManager` (not a global), keeping namespace clean.
- The R191 test that asserts cumulative bucket logic was refactored to read the template by name (`notification_manager._DEFAULT_LATENCY_BUCKETS_SECONDS`) rather than hardcoded values — so future bucket retuning won't silently break the bucket-correctness invariant test. This is a small but **structurally important** decision: the test now tests *the invariant*, not *the specific bucket choice*.

**What could be better** (none critical):

- The `notification_manager.py` source comment around line 511 explains the *why* of R196 but doesn't link back to CR#19 §4.1 — adding `(CR#19 §4.1)` would help future readers trace the design choice. **Minor docs follow-up.**
- The bucket tuple is "opinionated" but **not** runtime-configurable. If a deployment has unusually slow notification provider (e.g., a self-hosted gotify behind a flaky VPN), the dashboard buckets won't capture the > 10s tail well. Acceptable trade-off for now — runtime override would mean per-deployment Prometheus scraper config divergence which is its own complication.

### 2.2 R197 — Latency stats invariant test

**Commit**: 6f73b49
**Files**:
- `tests/test_latency_invariant_r197.py` (NEW, +422)
- `src/ai_intervention_agent/notification_manager.py` (+3 / -4): updated stale R191-era comment block

**What it does**:

`NotificationManager._send_single_notification` updates **two** latency counters in the same `_stats_lock` block:

1. R142 path: `stats["latency_ms_total"] += int(latency_ms)` / `latency_ms_count += 1` (integer milliseconds).
2. R191 path: `self._record_provider_latency_bucket(provider, latency_ms / 1000.0)` (float seconds, into histogram).

The two paths consume the *same* `latency_ms` sample, so their running totals must stay consistent. Specifically:

- `latency_ms_count` (R142) **==** `histogram[provider]["count"]` (R191).
- `latency_ms_total / 1000.0` (R142) **≈** `histogram[provider]["sum_seconds"]` (R191), within float tolerance.

If a future refactor accidentally separates the two updates (async fan-out, conditional execution, different lock scopes), neither R142's per-counter test nor R191's per-counter test would catch the divergence — they each test only their own counter. Cross-counter invariant tests are the missing layer.

R197 fills that layer with **10 test cases** across 4 invariant classes:

1. **Mathematical consistency** (3): single sample, multiple samples, int-to-float unit conversion don't drift.
2. **Multi-provider isolation** (2): different providers track independently; same invariant holds per-provider.
3. **Source-level AST guards** (3): the two metric updates must live in the same lock block; histogram call must use seconds (not ms); no alternate histogram call path may exist outside the lock.
4. **Edge cases** (2): zero samples (baseline), high-volume float accumulation (1000 × 1ms → 1.0 ± 1e-12).

**What's good**:

- The AST guard tests are the standout. `test_latency_ms_total_and_histogram_call_in_same_lock_block` parses `notification_manager.py` with `ast`, locates the two metric updates inside `_send_single_notification`, and asserts they share the same `with self._stats_lock:` ancestor. This is a *source-code-structural* invariant, not a runtime invariant — and structural invariants survive both refactors and conditional execution paths that runtime tests miss.
- `test_high_volume_float_accumulation_tolerance` runs 1000 × 1ms samples and asserts the float accumulation drift stays within 1e-12. This is realistic — many production deployments hit ~100 notifications/hour, accumulating slowly over weeks. Test catches accumulation bugs that would only manifest in long-running deployments.
- The fix for the regex bug in `test_histogram_call_uses_seconds_unit` (mentioned in the commit message) — distinguishing function definition site from function call site — shows attention to test correctness, not just coverage count.

**What could be better** (none critical):

- The test file is 422 lines, but a single class (`TestSourceLevelLatencyPathColocation`) doesn't have a docstring header explaining *why* AST testing is preferred over runtime testing here. A 3-line header in the class would help future maintainers — *"R142/R191 share a sample; if future code separates them, dashboards diverge silently because both counters individually still look correct"*. **Minor docs follow-up.**
- Test file uses module-global `notification_manager._provider_latency_histograms` state. Other tests that run in the same pytest session and don't isolate this state could pollute R197's assertions. The test does `notification_manager.reset_provider_latency_histograms()` in `setUp` — checked, present, correct. ✅

### 2.3 R198 — SSE event schema registry

**Commit**: 2bbb8b8
**Files**:
- `src/ai_intervention_agent/sse_event_schemas.py` (NEW, +137)
- `tests/test_sse_event_schemas_r198.py` (NEW, +445)
- `scripts/generate_docs.py` (+8 / -0): add new module to QUICK_NAV_UTILITY
- `docs/api/sse_event_schemas.md` (NEW)
- `docs/api.zh-CN/sse_event_schemas.md` (NEW)
- (regen of 3 other api.md files for unrelated drifts: `enhanced_logging`, `mcp_tool_call_metrics`, `notification_manager`)

**What it does**:

Defines `EventSchema` frozen dataclass and `EVENT_SCHEMAS` registry covering 4 known event types:

- `task_changed` (R40 / oldest): emitted from `web_ui_routes/task.py`. Required fields: `task_id`, `old_status`, `new_status`. Optional: `stats`.
- `config_changed` (R48): emitted from `web_ui_config_sync.py`. Required: `reason`, `hint`.
- `log_level_changed` (R192): emitted from `web_ui_routes/system.py`. Required: `old_level`, `new_level`, `logger`, `changed_by`.
- `oversize_drop` (R58): emitted from `web_ui_routes/task.py` (bus self-replacement path). Required: `original_event_type`, `size_bytes`, `limit_bytes`.

Public API: `EVENT_SCHEMAS`, `get_known_event_types()`, `get_schema(event_type)`, `validate_payload(event_type, payload) -> list[str]`. No runtime validation — schemas exist purely as the source-of-truth for emit-sites and frontend / VSCode subscribers, validated at test-time / IDE-time.

**What's good**:

- **No runtime cost**: `emit()` doesn't call `validate_payload` — schemas exist as documentation + test contract, not as a check on every fan-out. Smart choice; SSE fan-out is in the hot path of the activity dashboard.
- **AST guard for emit-site coverage**: the test `_find_sse_emit_call_sites_in_source_tree` walks the entire `src/ai_intervention_agent/` tree, finds every `_sse_bus.emit("<literal>", ...)` call, and asserts each literal is registered in `EVENT_SCHEMAS`. Adding a new event type without registering it fails CI. This is the same structural-invariant style as R197.
- **`emitted_by` field**: each schema records the file path that emits it, and the AST guard cross-checks that emit-sites are listed. Prevents emit-site relocation from drifting silently.
- **Disjoint required ∪ optional** invariant test: catches schema-author mistakes like accidentally listing a field as both required and optional.
- Schema dataclass uses `frozenset` for `required_fields` / `optional_fields` and `tuple` for `emitted_by`, making the whole structure hashable + immutable. Could be cached / interned trivially.

**What could be better**:

- **F-198-1 / nice-to-have**: schema doesn't carry **type info** for fields (just names). `task_changed`'s `stats` field is documented in `description` text as `dict[str, int]`, but a subscriber consuming `EVENT_SCHEMAS` programmatically can't distinguish `int` from `str` from `dict`. Future enhancement could add `field_types: frozenset[FieldSpec]` where `FieldSpec = (name, py_type)`. Trade-off: more complex schema, runtime cost if validated. **Skip for now** unless a real consumer needs it.
- **F-198-2 / nice-to-have**: `validate_payload` only checks field set membership, not deep value validation (e.g., `size_bytes` should be positive int, not string). Same trade-off applies; intentional.
- **F-198-3 / docs**: 4 events registered today, but the SSE bus has been around for ~80 cycles. Are there any *historical* event types that were emitted, then removed, and might come back? The schema registry doesn't have a deprecated/historical section. Low priority unless we add a 5th event type.

### 2.4 R199 — API token age + last-rotated tracking

**Commit**: 32a3ab4
**Files**:
- `src/ai_intervention_agent/shared_types.py` (+9): `api_token_rotated_at: SafeStr = ""` field
- `src/ai_intervention_agent/config_modules/network_security.py` (+47): validation + whitelist
- `src/ai_intervention_agent/web_ui_routes/system.py` (+143 / -2): new GET endpoint + write timestamp on rotation
- `config.toml.default` (+8)
- `docs/configuration.md` (+1), `docs/configuration.zh-CN.md` (+1)
- `tests/test_api_token_info_r199.py` (NEW, +351)
- `tests/test_network_security_config.py` (+5 / -1): output_structure invariant +1 field
- `CHANGELOG.md` (+72)

**What it does**:

1. Persists `[network_security].api_token_rotated_at: str` (ISO-8601 UTC) when `POST /api/system/rotate-api-token` is invoked.
2. Adds `GET /api/system/api-token-info` (loopback-only, rate-limit 30/min) that returns:
   - `has_token: bool` — non-empty + length ≥ 16
   - `token_length: int | None`
   - `rotated_at: str` — the persisted timestamp or `""`
   - `age_seconds: int | None` — wall-clock delta, `null` for unset / parse-fail / future timestamps
3. **Never** returns the `api_token` plaintext — rotation endpoint remains the sole disclosure point.

**What's good**:

- **Same loopback gate as R195** — token age is metadata, not a secret, but knowing "this token has been used for 89 days" gives an attacker a window-prediction signal. Loopback-only is the right gate. The endpoint docstring explicitly explains this.
- **Future timestamp → null age, not 0**: if NTP jumps or the config is hand-edited to a future timestamp, `age_seconds` becomes `null` rather than `0`. Otherwise a dashboard would interpret `0` as "just rotated" — exactly the wrong signal. Subtle, important call.
- **Timestamp generated before write, not after**: rotation endpoint computes `rotated_at = datetime.now(UTC).isoformat()...` *before* the `update_network_security_config` call. This guarantees disk-state and response-body share the same string — subsequent `GET /api/system/api-token-info` reads back the same timestamp.
- **Validation in `_validate_network_security_config`**: enforces `Z` or `+00:00` suffix + `fromisoformat` parseable. Dirty values are silently dropped to `""` with a warning. Prevents poisoned config from breaking age calculation.
- **Tests cover the security-sensitive invariants**:
  - `test_response_never_leaks_api_token` walks **all** string-valued fields of the response and asserts the actual token string doesn't appear anywhere — defends against `repr()` / accidental string-formatting leaks.
  - Source-level guard asserts `"api_token":` literal is not in the endpoint body — catches future refactors that accidentally `return ..., "api_token": cfg.api_token` thinking they're being helpful.
- **TODO.md schema invariant test fix** (`test_output_structure` +1 field): R189 schema invariant test was extended to include `api_token_rotated_at`. This is the correct way to extend a schema invariant — explicitly opt-in, not silently relax. ✅

**What could be better**:

- **F-199-1 / important / minor inconsistency**: if an admin manually edits `config.toml` to set `api_token = ""` (revoking) but doesn't clear `api_token_rotated_at`, the endpoint returns `has_token=false` + `rotated_at="2026-04-02T..."` + `age_seconds=non-null`. This is "stale ghost" state — dashboard alerts might fire for "token 60 days old" but the token is actually gone. **Fix**: when `_validate_network_security_config` normalizes a config and notices `api_token == "" and api_token_rotated_at != ""`, log a warning. Or `update_network_security_config` could auto-clear `rotated_at` when `api_token` is explicitly set to `""`. R200 candidate.
- **F-199-2 / nice-to-have**: `GET /api/system/api-token-info` doesn't expose a `recommended_rotation_age_days` field. NIST SP 800-63B suggests 30-90 days — emitting it from the endpoint would let admin tools render the alert threshold without hardcoding (could be a future config field). Skip unless a downstream actually needs it.
- **F-199-3 / docs**: the rotation endpoint docstring (R195) wasn't updated to mention "also writes `api_token_rotated_at` to config". Minor — readers can infer from the source, but explicit doc would help.

## 3. Cross-cutting Observations

### 3.1 Test density continues to climb

CR#19 noted ~5300 tests; this cycle added ~50 tests (R197 + R198 + R199) bringing the suite to ~5366. Wall-clock for full run remained < 3 minutes — no degradation. The pattern of "every feature commit lands with 10-20 invariant tests + at least 1 AST guard" is now firmly the cycle-7 norm.

### 3.2 AST guards as the cycle-7 signature

Three of the 4 commits in this cycle (R197 / R198 / R199) include AST-based source-level invariant tests. This is a meaningful evolution from cycle-5 (runtime tests only) — the project is now defending invariants at the structural level (function body shape, lock co-location, schema registration) not just behavioral level (input → output assertions).

The trade-off is brittleness: refactors that move code (even for legitimate reasons) need to also update the AST tests. This is by design — we want refactors to be deliberate, with the developer choosing whether to relax or update the invariant. Discussed in CR#16 §3.5 (structural invariants vs runtime tests).

### 3.3 No regressions caught mid-cycle

Two test failures during the cycle (`test_changelog_diff_scope` from formatter pollution, `test_config_docs_parity` after adding `api_token_rotated_at`) were both **mechanical drift** (formatter touched too many lines / docs lagged config). Neither indicated a functional bug. Both were resolved with `git restore` + 1-line docs edits respectively. No "actual bugs" were fixed in this cycle — uncommon for a 4-commit cycle, indicates code quality is in a healthy state.

### 3.4 Schema-driven evolution beyond SSE

R198 sets a precedent for schema registries. Other places in the project that could benefit from a similar treatment (with similar AST guards):
- **Notification provider types** — currently each provider class self-registers (`bark`, `pushover`, etc.) via `_PROVIDER_REGISTRY` dict. A formal schema with required vs optional config fields per provider would help docs + config validation.
- **CLI command surface** — `aiia` CLI accepts a fixed set of subcommands; documenting them as a registry similar to `EVENT_SCHEMAS` would standardize help text.
- **Metric families exposed via `/metrics`** — currently `_render_prometheus_metrics()` hard-codes the list. A registry would let us assert "every registered metric has HELP + TYPE + at least 1 sample".

These are R200+ candidates, **not** required by this cycle.
