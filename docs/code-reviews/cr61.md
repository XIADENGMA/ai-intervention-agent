## 4. 风险 & 隐患

### 4A. context_processor 全局注入的潜在性能影响

R306 注册 `@app.context_processor` 后, 每次 `render_template()` 都会执行
`getattr(g, "csp_nonce", "")` 一次。

**影响估算**:
- 每请求开销: ~1-3 µs (getattr + dict lookup)
- web_ui.html 每 5s 一次 GET / + 大量 static 不走 render_template
- 总开销: 微不足道 (~< 0.001% 总 CPU)

**风险等级**: 极低
**缓解建议**: 不动。如果未来 templates 数量爆炸 (>100), 可考虑 LRU cache
g.csp_nonce 引用。

### 4B. R310 future-guard 数字 baseline 维护

R310 锁了 `asyncio.create_task` 总数 = 2 (baseline)。未来新增需要:
1. 实现 cancel + await + suppress cleanup
2. 在 R310 测试中显式列入新 task
3. 更新此 baseline 数字

**风险等级**: 低 (这正是 invariant 的设计目的)
**缓解建议**: 在 `docs/contributor-guide-invariant-tests.md` 加一节说明
"新加 asyncio.create_task 时的 3 步流程", 让贡献者知道为什么 R310 测试
会 fail。

### 4C. R308 文档锁定后的更新负担

R308 锁了 `docs/perf-ci-xdist-r308.md` 必须含特定关键词 + benchmark 表格。
未来如果 GitHub Actions runner 升级到 8 vCPU, 文档需要重测 + 重写, 但
invariant 不会自动更新数据 — 需要人工补 benchmark + 改文档。

**风险等级**: 中
**缓解建议**:
- 文档已经有 "Re-benchmark trigger conditions" 一节, 列了 4 个触发条件
- 风险可接受, 因为 GitHub Actions 升 vCPU 不会无声发生 (会有 changelog)
