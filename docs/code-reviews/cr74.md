## 5. 跨 cycle 风险与机会

### 风险 A: Pydantic validator coverage 的 EXEMPT_FIELDS 当前为空

R388 的 `EXEMPT_FIELDS_BY_CLASS` 当前是 `{}` 空字典 — WebUIConfig +
FeedbackConfig 所有数值字段都有 validator, 无需豁免。**这是理想状
态**, 但也意味着如果未来有需要豁免的字段, 当前 invariant 的 layer
3 测试 (whitelist 必须含 rationale ≥ 50 字符) 永远不会 exercise。

**建议**: 在 future cycle, 如果有新的 Pydantic 模型加入, 主动考虑是
否需要在 R388 EXEMPT 列表中预留 placeholder + rationale 模式, 让
layer 3 有真实数据可验证。优先级 P3。

### 风险 B: doc-parity 4 应用都是 EN/zh-CN 二元

cycle-42 R374 已经把 i18n consistency 扩到 zh-TW. doc-parity 4 应用
全部是 EN/zh-CN, 未覆盖 zh-TW. 是否需要 zh-TW 翻译, 取决于项目实际
受众。

**建议**: 在 future cycle, 决定 doc-parity 是否扩到 zh-TW, 或者明确
"项目只维护 EN/zh-CN 双语文档" 的方针, 减少潜在 audit 漏洞。优先级
P3。

### 风险 C: API contract 6th 应用 (R392) 当前只锁了 same-level
required ↔ properties, 未递归到 nested schema

R392 通过递归 walk YAML 树找所有 `required + properties` 同级对, 但
没有验证 nested `$ref` 引用的 schema 是否完整。当前 codebase 不使用
`$ref`, 所以这不是问题, 但 future 如果引入 reusable schema definitions
(via `definitions:` block), 需要扩展 R392 跟踪 `$ref` 引用。

**建议**: 标记 R392 为 "no-$ref-yet 假设", 在 docstring 显式声明,
future 若用 `$ref` 必须扩展 invariant。优先级 P2。

### 机会 D: v3.10 候选维度

API contract 6 应用工业化深化, 已达 v3.10 候选阈值。可在 cycle-45
或 cycle-46 评估是否将 API contract 标记为 v3.10 系列首个 pattern,
启动 v3.10 命名。

**建议**: cycle-45 cr75 中评估 v3.10 升级条件 (是否还有 cross-
language schema, idempotent, async-race 等其他 5+ 应用维度同时达成
里程碑)。
