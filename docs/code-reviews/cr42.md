**Design decision review — "no unfreeze"**: explicit
trade-off documented in commit body ("prevent abuse and
simplify state machine, recommending new task creation for
resuming countdowns"). This is **defensible** because:

1. Auto-resubmit's purpose is preventing forgotten tasks
   from blocking the AI client indefinitely. Freezing once
   should be a deliberate signal.
2. Unfreeze would create round-trip thrash on the watched-
   lock; sequential freeze→unfreeze→freeze cycles would be
   indistinguishable from clock-skew bugs in testing.
3. New-task creation is cheap (~1 click) and forces user
   to reconsider intent.

**Concerns / follow-ups** (none blocking):

- **[low]** Freeze button label "冻结" / "Freeze" is
  technical jargon. Could A/B-test "永不超时" / "Keep
  active" for clearer intent. Defer to UX research.
- **[low]** No e2e test that simulates the full flow
  (active task → click freeze → countdown stops →
  auto-resubmit doesn't fire). Per cr41 §8 #7 defer, e2e
  harness is still out of scope.

### §2.4 Discoverability gap — rule of three confirmed (commits 3, 4)

**Three consecutive cycles** missed already-existing
features at the candidate-logging stage:

| Cycle | Candidate | Already-existed as | Discovery commit |
|---|---|---|---|
| cr40 §8 #5 | `scripts/bump_version.py` helper | comprehensive script with `CITATION.cff` / `uv.lock` / issue-template support | cr41 `03a68b0` |
| cycle-6 Track B | user-defined feedback templates | `quick_phrases.js` R130 (cycle-1 era) | cr42 `a101a48` |
| cycle-6 Track C | cross-task message queue | `multi_task.js:taskTextareaContents` + R139 `feedback_drafts.js` | cr42 `97669ad` |

**Pattern**: each cycle proposes work without `rg` /
`git log --grep` pre-check, then discovers the feature
during ship-prep grep. This is **time-cost wasted in two
places**:

1. Cycle-N: spending 15-30 mins logging the candidate with
   "fake evidence" (`rg` snippets from the **source**
   project, not own codebase)
2. Cycle-N+1: spending 30-60 mins discovering, reverting
   exploration work, writing closeout doc

**v3.2 methodology proposal** (commit 4 §5.1 lesson #2):
make pre-§2.1-add `rg` check **mandatory and blocking** —
log a `methodology-rg-check` step in every future cycle's
kickoff doc §0 with the actual `rg` command output as
evidence. Failing to attach `rg` evidence = closeout
reviewer must reject the candidate row.

**Defensibility assessment**: this is the right gate
because:

- **Cost is trivial**: `rg <keyword> src/ tests/` is 1
  command, < 1s
- **Catch rate is high**: would have caught all 3
  occurrences
- **Failure mode is recoverable**: if mistakenly logged
  without `rg`, reviewer rejects → next cycle re-evaluates
- **Aligns with v3.1**: v3.1 already requires `rg`
  filesystem evidence for the **source** project; v3.2
  extends to **own** codebase as well

**Concerns**:

- **[low]** `rg` could miss heavily-refactored features
  (e.g., feature is named "voice input" in mining cycle
  but lives as `dictation_*` in code). Mitigation:
  reviewer should also `grep` for semantic synonyms.
- **[info]** v3.2 needs explicit codification in a future
  mining-cycle kickoff doc to take effect; this cycle
  proposed it, cycle-7 must adopt it.

2. `security-architecture.md` documenting `Permissions-

3. Privacy modal text drafted in en + zh-CN

**Verdict**: defer is **correctly justified** with concrete
re-evaluation criteria. Not "defer indefinitely" — has
explicit unlock conditions.

## §4 Lessons learned

1. **Single-cycle execution is viable when carry-over stack
   is rich**. Don't artificially extend cycles for
   ceremony if the work is contained.
2. **Discoverability gaps compound across cycles**. Three
   strikes = systematic. Cheap pre-check (`rg`) prevents
   expensive cleanup.
3. **"Discovered-already-shipped" is a valid closeout
   outcome** — not a failure of the cycle, but a finding
   that the methodology must absorb.
4. **Defer rationale must include unlock conditions**. cr39
   §8 #7 "defer e2e harness" lacks unlock conditions and
   has lived 4 cycles. Track D `voice input` has 3 explicit
   prereqs — model for future defers.
5. **Production ship + process improvement can co-exist in
   one cycle** (1 ship + 4 docs commits). Don't separate
   them into different cycles if work is logically related.
