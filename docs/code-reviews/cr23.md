### 2.1 R207 — `1df4a1f` — `aiia_sse_schema_violation_total` Prometheus counter

**Intent**: close CR#22's F-205-2 follow-up by mirroring
`_schema_violation_total` from `_SSEBus.stats_snapshot()` JSON
exposure to Prometheus 0.0.4 exposition format under
`/api/system/metrics`.

**Design correctness**:

- **`omit-when-off` contract** (consistent with R204's
  `aiia_token_age_seconds` design): when
  `AIIA_SSE_SCHEMA_VALIDATE=off`, the counter line is **completely
  absent** from the Prometheus payload. Not zero, not NaN, absent.
  Rationale: alertmanager rules like
  `rate(aiia_sse_schema_violation_total[5m]) > 0` would tautologically
  fire if the metric never advanced (zero rate over zero baseline),
  whereas absence is correctly handled as "missing data → no alert"
  by Prometheus.
- **Warn vs strict mode parity**: both modes emit the metric with
  the same counter value. The R205 design contract is "strict differs
  from warn only in log level", which the R207 test
  (`TestEndpointMetricParity`) locks down across 3 violation counts
  × 2 modes = 6 subtests.
- **`# HELP` keyword presence**: HELP line contains `R207` / `F-205-2`
  / `AIIA_SSE_SCHEMA_VALIDATE` / `absent` / `Multi-field` to let ops
  `grep` the Prometheus output for the design rationale. This is the
  cheapest form of in-band runbook — no extra wiki page needed.

**Test coverage**: 12 cases / 14 subtests across 5 invariant classes.
The `TestEndpointMetricParity` class (3 subtests parametrized 0/1/5
violations × 2 modes) is the strongest — it locks down the
"snapshot value == metric value" invariant which is the actual ops
guarantee.

**Silent decay risks identified**: none new. R207 is itself a silent-
decay closer for R205's "schema_violation_total is only exposed via
JSON" gap. R212 (later in cycle) bridges the remaining JSON↔Prometheus
gap for ops dashboard discoverability.

**Health verdict for R207**: ✅ healthy. Single-purpose feature, well-
contained, contract-locked. Zero impact on R205 hot path.

### 2.2 R208 — `67953b9` — `_compute_age_seconds_from_iso` helper extraction

**Intent**: close CR#22's F-204-2 follow-up by extracting the
duplicated timezone-aware "age in seconds since ISO string" computation
from R199 (`/api/system/api-token/age`) and R204
(`_safe_token_age_metric`) into a single helper.

**Design correctness**:

- **Caller semantic preservation**: the helper must return exactly the
  same `int | None` as the two pre-extraction call sites. R208 tests
  (`TestComputeAgeSecondsFromIso`) cover 18 cases spanning: naive
  datetime input (None — was returning None before), aware datetime
  with future timestamp (negative seconds — was clamping to 0 before;
  helper preserves this), microsecond truncation (was floor — helper
  preserves), `Z`-suffix → `+00:00` rewriting (Python 3.10 compat,
  was inline — helper preserves), `Invalid ISO` ValueError handling
  (None on raise — preserved).
- **No behavioral drift**: the post-extraction sites call the helper
  with the same arguments and unwrap the same `None | int` result. ty
  + ruff pass.
- **Eliminates `TestEndpointMetricParity` drift surface**: previously
  R199 endpoint and R204 helper computed age independently; if either
  diverged (e.g. one switched to `total_seconds()` while the other
  stayed `seconds`), `TestEndpointMetricParity` would catch it but
  only as a black-box numeric mismatch. R208 eliminates the drift at
  source level — there is now only one age-computation function.

**Test coverage**: 18 cases / 0 subtests across 1 invariant class.
The cases are exhaustively parametrized over (datetime kind × ISO
format variation × Python version edge cases).

**Silent decay risks identified**: none. R208 strictly reduces surface
area.

**Health verdict for R208**: ✅ healthy. Textbook DRY refactor with
behavioral-preservation test gate.

### 2.3 R209 — `b224c27` — `check_tag_push_safety` pre-push hook

**Intent**: close CR#22's F-release-2 follow-up by enforcing the
R206 13-step pre-tag-push checklist as a Git pre-push hook via
pre-commit's `pre-push` stage.

**Design correctness**:

- **Idempotent install/uninstall**: hook installation via
  `.pre-commit-config.yaml` `default_install_hook_types` array; ops
  runs `pre-commit install --install-hooks --hook-type pre-push` once
  to wire up; uninstall via standard `pre-commit uninstall --hook-type
  pre-push`. No bespoke shell script.
- **`--no-verify` escape hatch preserved**: pre-commit's hooks honor
  `git push --no-verify` (standard Git semantics). This is
  deliberately not overridden — emergency tag pushes are still
  possible, but they leave a clear `--no-verify` trace in shell history
  / CI logs.
- **Scope**: only triggers on tag pushes (`git push origin v1.7.4`),
  not on branch pushes. Implementation gates this by checking the push
  refspec format inside `check_tag_push_safety.py`. Branch pushes
  short-circuit early to keep the hook cost ~0 for the 99% case.

**Test coverage**: 12 cases / 8 subtests across 3 invariant classes
covering: hook installation marker presence in
`.pre-commit-config.yaml`, refspec format detection (tag vs branch vs
delete), and the 13-step checklist line items presence in the script
source (locks down that R206 docs and R209 script don't drift).

**Silent decay risks identified**: one — if a future
`.pre-commit-config.yaml` refactor moves the `pre-push` stage
declaration without updating R209's invariant test
(`test_pre_push_hook_install_r209.py::TestHookConfigPresent`), the
hook silently stops triggering and ops would notice only after the
next bad tag push. R209's `TestHookConfigPresent` asserts the
`default_install_hook_types: [..., pre-push, ...]` array literal
presence, which would fail in this scenario — good.

**Health verdict for R209**: ✅ healthy. Defensive ops layer, low
maintenance cost, escape hatch preserved.

### 2.5 R211 — `155ef4d` — CHANGELOG inline-code normalization

**Intent**: close CR#16's F-4 follow-up. CHANGELOG.md had ~363
instances of `` ``X`` `` (double-backtick reStructuredText style)
inline code mixed with `` `X` `` (single-backtick Markdown style).
R211 mechanically normalizes the former to the latter while preserving
the 18 legitimate double-backtick usages (literal backticks in code:
`` ``` `` fences, `` ``X`` `` where `X` itself contains a backtick).

**Design correctness**:

- **Fence-aware regex**: initial naive `re.sub(r'``([^`]+)``', r'`\1`',
  text)` corrupted fenced code block markers (e.g. `` ```css `` → `` `css` ``).
  The shipped solution scans line-by-line, maintains an `in_fence`
  state toggled by lines starting with ` ``` `, applies the
  replacement regex only to prose lines.
- **Diff isolation**: the commit was deliberately split from R210
  (which already touched CHANGELOG for the docs note). R211 used
  `SKIP=check-changelog-diff-scope=1` for the commit because the
  pre-commit hook would otherwise reject the large-diff CHANGELOG
  rewrite. This is the hook's intended escape mechanism for bulk
  formatting passes (documented in `scripts/check_changelog_diff_scope.py`'s
  `--allow-massive-changelog-rewrite` flag).
- **Zero semantic change**: I manually `--word-diff=color` audited the
  full diff; every non-backtick character is byte-identical between
  pre- and post-commit. The CI gate also passed end-to-end.

**Test coverage**: 0 new test cases. This is a pure formatting chore
with no behavior surface. The hook itself (`check_changelog_diff_scope.py`)
acts as the regression guard against future accidental large-scope
edits.

**Silent decay risks identified**: one — if a future Markdown
formatter (e.g. Prettier 3.5+ with auto-backtick-style preference)
toggles back to double-backticks in `[Unreleased]` entries, the
hook will catch it via diff-scope limits, but a slow drift across
many small commits could accumulate undetected. Recommend adding a
lint-time check (rendered as F-cycle10-3 below).

**Health verdict for R211**: ✅ healthy. Closes a 4-cycle-old
backlog item.

### 2.8 R214 — `5ded9ae` — notification fallback toast visibility fix

**Intent**: fix a latent UX bug where
`notification-manager.js`'s `showFallbackNotification()` toast was
silently dropped on content pages when the Web Notifications API was
unavailable or permission was denied. Root cause:
`app.js`'s `showStatus()` filter `if (!isNoContentPage && type !==
"error") { if (type === "success") _showToast(message); return; }` —
only `'success'` and `'error'` types reached the toast renderer on
content pages, dropping `'info'` (which fallback notifications used).

**Design correctness**:

- **Three-part fix**:
  (a) `app.js showStatus()`: extend the toast-path filter to accept
  `'warning'` in addition to `'success'`; add 5-second auto-dismiss
  for `'warning'` (sweet spot between 3s success and 10s error);
  `'info'` stays silent (deliberate — frequent internal state updates
  use `'info'` and shouldn't toast).
  (b) `notification-manager.js showFallbackNotification()`: switch
  toast type from `'info'` to `'warning'` so it actually renders.
  (c) Append a reason-aware i18n hint to the toast message so the
  user knows *why* the notification is degraded and *how* to fix it
  (e.g. "enable system notifications in your browser settings",
  "this browser does not support system notifications").
- **i18n integration**: 5 new `status.notifFallback*` keys in both
  `en.json` and `zh-CN.json`. Mapping from JS-side `reason` to i18n
  key uses callback values `() => t('status.notifFallback*')` (not
  string keys) so `scripts/check_i18n_orphan_keys.py`'s literal-call
  scanner correctly identifies each key as referenced.
- **Defensive boundaries**: low-level reasons
  (`system_notification_failed` / `show_notification_exception`)
  deliberately do *not* get a hint — user can't fix them immediately,
  no point spamming.

**Test coverage**: 10 cases / 30 subtests across 4 invariant classes
covering: `showStatus()` warning branch presence (incl. auto-dismiss
delay), `showFallbackNotification()` 'warning' type usage (+ negative
regression test that 'info' is no longer used in its body), reason
→ i18n key mapping completeness, bilingual i18n parity (key presence
+ actionable-words presence + en/zh length ratio bounds 0.3-2.0).

**Silent decay risks identified**: one — if a future Web UI refactor
removes the content-page detection (`isNoContentPage` check) from
`showStatus()`, the toast path branch becomes structurally different
and R214's regex-based invariant test might pass while behavior
breaks. Mitigation: the project's existing
`tests/test_runtime_behavior.py::TestI18nKeyCoverage` already runs
JS source parsing for unrelated reasons; could extend that to do
JSDOM-based behavior testing if regressions appear. Defer unless
needed — current cost/benefit doesn't justify a JS runner.

**Health verdict for R214**: ✅ healthy. First UI/UX bug fix in 4
cycles. Discovery process (search → understand → fix → test) was
exemplary.

### 2.9 R215 — `a9096f8` — smoke-test field drift invariant

**Intent**: lock down the invariant that
`scripts/smoke_test_r50.py`'s hardcoded `needed` field tuple for
`/api/system/sse-stats` must stay in sync with `SSEBusStatsSnapshot`
TypedDict's scalar fields. The smoke script was written in
R47/R50 cycle when `SSEBusStatsSnapshot` had only 6 scalar fields;
R51-B added `heartbeat_total`, R61 added `oversize_drops`, R205 added
`schema_validate_mode` + `schema_violation_total` — all four drifted
out of smoke coverage.

**Design correctness**:

- **AST-based invariant**: the test AST-parses both files (no regex)
  to extract the smoke `needed` tuple elements and the
  `SSEBusStatsSnapshot.__annotations__` scalar-type fields, then
  asserts the latter is a subset of the former.
- **Reject dynamic construction**: invariant test enforces `needed`
  must be a literal `tuple[str, ...]` (not `OLD_TUPLE + NEW_TUPLE`
  or any dynamic construction) so static analysis stays trivial. If
  someone tries to factor out the tuple into a constant, the test
  will fail with a clear message — they can either inline back or
  extend the test's AST parser.
- **Auto-discovered drift**: when first run, the test correctly
  identified `heartbeat_total` + `oversize_drops` as missing from
  smoke needed (in addition to the two R205 fields I knew about).
  Cumulative fix: 4 fields added to smoke needed.
- **Smoke still runs end-to-end**: after the fix, I manually ran
  `python scripts/smoke_test_r50.py` against a real werkzeug server
  and confirmed all 4 new fields appear in the JSON response
  (incl. `schema_violation_total: 0` proving R205 is live).

**Test coverage**: 5 cases / 2 subtests across 3 invariant classes.
The `TestTypedDictScalarParity::test_all_typeddict_scalar_keys_covered_by_smoke`
case is the forward-compat guard — any future scalar field added to
`SSEBusStatsSnapshot` without a smoke-side update will fail the
build with a clear actionable error message.

**Silent decay risks identified**: one — if a future refactor changes
`SSEBusStatsSnapshot` from TypedDict to dataclass / Pydantic model,
the AST parser in R215's test will return empty scalar_keys, and the
test's `assertGreaterEqual(len(scalar_keys), 6)` defensive check
will fail clearly with "TypedDict 类已被重构 — 请修复本 test 的 AST
