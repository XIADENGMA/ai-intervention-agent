## 1. Cycle theme — three-wave WCAG 4.1.2 sweep + meta-guard layer

CR#26 ended Cycle 13 with R230 hiding 29 decorative SVGs from
screen readers. CR#26's F-cycle13 backlog asked: "are the *icon-only
buttons* that wrap those now-hidden SVGs themselves still named?"
That question chained into a three-stage sweep, each stage closing a
different a11y gap and each one **locking the pattern with an
invariant test**:

| Wave  | R   | Gap closed                                                              | Lock invariant                                                    |
| ----- | --- | ----------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Wave 1 (CR#26) | 230 | Decorative SVGs visible to AT → spam                                  | every `<svg>` w/o role=img has `aria-hidden + focusable=false`    |
| Wave 2 (R232)  | 232 | Icon-only buttons name relied on SVG (just hidden) → no name           | every icon-only button has `aria-label` or `aria-labelledby`      |
| Wave 3 (R235)  | 235 | Form inputs/textareas had only placeholder → no name when typing       | every input/textarea has accessible name or `aria-hidden+tabindex`|

**为什么这个 wave 结构值得记录**：每个阶段都被前一阶段触发。R230 直接
导致 R232（"按钮的 visible name 现在没了"），R232 的审计模式又被 R235

### R231 — invariant test catalogue staleness guard + backfill

**File**: `tests/test_invariant_test_guide_catalogue_r227.py`,
`docs/contributor-guide-invariant-tests.md` (+ zh-CN sibling).

**问题面**：CR#26 写完之后我突然意识到 R227 的 guide 是
**纯文档**，没有任何机制保证 catalogue 表格与实际 `tests/test_*_
invariant_r*.py` 文件同步。果然 R224/R229/R230 已经漏掉。如果只是
回填一次就完事，未来还会再次 drift —— 必须装 trip wire。

**做法**：
1. 扫 `tests/test_*_invariant_r*.py`，提 R-number。
2. 与 catalogue 中提到的 R-number 集合做差。
3. 任何在「最近 MAX_R_LAG = 10」窗口内的 R 如果未列入 catalogue，
   测试失败并指明哪几个漏了。

**为什么选 lag 窗口而非全量**：旧 R 可能合理地从 catalogue 中淘汰
（merge 到更大主题里），强制全量会变成噪音。lag 窗口保证 *最近的
fresh 工作* 一定要进 catalogue，旧的让维护者自己决定。

**测试方法学**：先看到 test fail 报 R224 缺失，再去查为什么 R224
没进 catalogue —— 因为 CR#25 当时没记录所有 R 都需要登记，这本身
就是 cycle 12 漏的文档债。把 R224、R229、R230 都补上，测试通过。

**风险**：`MAX_R_LAG = 10` 是个魔数，没强烈的理由必须是 10 而不是
5 或 20。当前 R# 已到 235，10 个窗口约对应 1.5 个 cycle，能给写
catalogue 的人足够缓冲又不至于拖太久。可以接受。

**Verdict**: GOOD. 装上了从此自动维护 catalogue 同步性的护栏。
唯一可改进点是把 `MAX_R_LAG` 提到 module-level 常量并加 docstring，
当前嵌在测试内。

### Ty fix — `assertIsNotNone` → `assert is not None`（process bug）

**File**: `tests/test_submit_btn_disabled_visible_invariant_r229.py`.

**问题面**：`v1.7.5` Release CI 失败在 `ci_gate.py` 的 `ty` 步骤，
提示 `tests/test_submit_btn_disabled_visible_invariant_r229.py` 中
`re.search()` 结果可能是 `None`。`self.assertIsNotNone(match)` 在
runtime 上完全正确，但 `ty`（与 `mypy`）**不识别它做类型收缩**
（type narrowing）。

**根因**：unittest 的 `assertIsNotNone` 没有 `TypeGuard` 注解。Python
的 `assert x is not None` 是 PEP 484 标准 narrower，所有静态分析器
都认识。

**做法**：把 `self.assertIsNotNone(match)` 全部替换为
`assert match is not None`，然后 `ty` 通过。R232 在写时已经主动
采用了这个写法 —— 这是从一个 release 失败中学到的 lesson 立刻被
应用到下一个 commit 的好例子。

**Verdict**: GOOD（救火 + 立刻 institutionalize），但暴露了一个
**process gap**：`ci_gate.py` 的 `ty` 检查现在只在 release tag CI
里跑，主分支 CI 没跑（否则 v1.7.5 就不会 tag）。建议下一 cycle 考虑
把 `ty` 加进主分支 CI 或 pre-commit（参 R226 同款思路）。**F-cycle14-1**。

### R232 — icon-only buttons aria-label invariant

**File**: `tests/test_icon_only_buttons_aria_label_invariant_r232.py`.

**问题面**：R230 让所有装饰 SVG 对 AT 透明。但 *icon-only button*
（按钮里只有一个 SVG icon、没有文字 child）的 accessible name 之
前可能就是依赖 SVG 内部的 `<title>` 或 SVG 本身。R230 之后这些
按钮变成 "button, no name"。

**Audit 发现**：实际审计后 web_ui.html 里的 icon-only buttons
**已经全部有 aria-label**（团队 a11y discipline 是好的）。R232 没
修任何代码，只**锁住了现状**，防止未来回归。

**测试设计**：
- 扫所有 `<button>` 与 `<a role="button">`。
- 判断是否"icon-only"：只有 SVG child，没有 text content。
- 必须有 non-empty `aria-label` 或 `aria-labelledby`。
- 包括 ty fix（`assert match is not None`）。

**为什么这样的 "现状已 OK" 的 invariant 也值得加**：它把
"团队历史上做对" 的不成文规则变成 *可执行的契约*，新贡献者引入
icon-only button 时立刻得到反馈。这是 R227 guide 里描述的
"prevent silent decay" 模式的标准案例。

**Verdict**: GOOD。零功能改动 + 高 long-term ROI。

### R233 — README factual claims invariant + backfill

**File**: `README.md`, `README.zh-CN.md`,
`tests/test_readme_factual_claims_invariant_r233.py`.

**问题面**：README 的 positioning paragraph 写了三个量化 claim：
"5,500+ tests"、"~700 subtests"、"6-job release pipeline"。这三个数
都来自具体源（`pytest` 输出 + `release.yml`），但没人保证它们与
源保持同步。审计结果：

- 测试数从写 README 时的 5,500 涨到 5,600+
- subtest 数从 ~700 涨到 ~800（详见下面 heuristic 段）
- release pipeline 从 6 job 变成 5 job（合并过）

三个数全过时。

**做法（一次性 backfill）**：直接改 README 到当前数。

**做法（持续护栏）**：写 invariant，把每个数与 *canonical source*
做对照：

- **Job count**：直接 parse `.github/workflows/release.yml` 数 `jobs:`
  顶级 key。精确。
- **Test count**：parse `pytest --collect-only` 输出 → 取 `collected N`
  → 与 README 数字比较，允许 ±400 漂移容差。
- **Subtest count**：subtest 数没法 collect 期统计，因为 subtest 是
  runtime concept（每次 loop 迭代一个新 subtest）。

**Subtest heuristic 难点**：第一版用静态计数 `subTest()` 出现次数，
严重低估（很多 subtests 在 loop 里）。第二版引入
`HEURISTIC_LOOP_FACTOR = 9`，把静态 `subTest()` 调用数乘以经验系数。
基于运行 R235 commit 时 `pytest` 显示的真实 subtest 数（~836）做的
校准。

**风险**：`HEURISTIC_LOOP_FACTOR` 是经验数字。如果之后大量增加非 loop
的 `subTest()` 调用（或反之），会漂移。容差设到 ±150 给了缓冲，但
如果哪天 README 与实际差太多，应该重新校准 factor 而不是改容差。
**F-cycle14-2**: 考虑改成在 pytest 跑完之后直接 grep 测试日志的真实
subtest 数（更准但要 hook 进 CI）。

**Verdict**: ACCEPTABLE — 第一个版本，heuristic 有理论缺陷但实操可用。

### R234 — feedback-textarea disabled visual → CSS

**File**: `src/ai_intervention_agent/static/css/main.css`,
`src/ai_intervention_agent/static/js/app.js`,
`tests/test_feedback_textarea_disabled_css_invariant_r234.py`,
`tests/test_submit_btn_disabled_visible_invariant_r229.py` (test 改向).

**问题面**：CR#26 的 R229 修了 `#submit-btn` + `#insert-code-btn`
的 disabled visual 漏看 bug（CSS `!important` 隐式覆盖 JS inline
color）。R229 **故意没** 改 `feedback-text` textarea，理由当时是
".feedback-textarea CSS 不用 !important，所以 JS inline 实际生效"，
还留了 `TestFeedbackTextareaInlineStyleKept` 测试**保护**这个差异。

R234 反转了这个决定，因为再细读 JS inline 值：

| Variable           | Value                          | Color theme     |
| ------------------ | ------------------------------ | --------------- |
| backgroundColor    | `#2c2c2e`                      | dark gray       |
| color              | `#8e8e93`                      | dark gray       |
| (subtle bg)        | `rgba(255,255,255,0.03)`       | white-on-dark   |
| (text alt)         | `#f5f5f7`                      | light gray      |

**全部是 dark-theme-only hex codes**。浅色模式下 disabled textarea
显示深灰文字配米色背景，对比反向 → 与 R229 修的 button 完全同款的
"theme-incorrect inline override" bug。

**做法（与 R229 完全镜像）**：

1. CSS 加 `.feedback-textarea:disabled`（深色）+
   `[data-theme="light"] .feedback-textarea:disabled`（浅色），
   后者用 `!important` 赢启用规则的 `!important`（R229 完全同款根因）。
2. JS 删掉所有 inline style，只留 `disabled = true/false` 翻转。
3. R229 的 `TestFeedbackTextareaInlineStyleKept` 改名/改向为
   `TestFeedbackTextareaInlineStyleRemovedByR234`，从"锁住 inline 存在"
   变成 "锁住 inline 已删除"。
4. 新写 R234 专属 invariant，7 cases 锁两个主题的 :disabled selector +
   light !important + 4 visual cue 属性（background, color, cursor,
   border-color）+ rgba (避 R66 brand drift)。

**意义**：到 R234 三个 interactive control（submit-btn / insert-btn /
feedback-text）的 disabled visual 全部走 CSS，pattern 统一。"JS 只翻
属性，CSS 负责 visual" 这个准则现在被 4 个独立 invariant 共同守住。

**Verdict**: GOOD。修正了 R229 当时的判断错误（不该信"CSS 没用
`!important` 所以 inline 生效"，因为 inline 的 *值* 本身就是错的）。
诚实地把过往的 defensive lock 反向 — 这是健康的 invariant 演化。

### R235 — form inputs accessible name (WCAG 4.1.2)

**File**: `src/ai_intervention_agent/templates/web_ui.html`,
`src/ai_intervention_agent/static/locales/{en,zh-CN}.json`,
`tests/test_form_inputs_accessible_name_invariant_r235.py`.

**问题面**：R230 + R232 解决了图形 control 的 a11y name。Audit
（在写 R235 时跑）发现 form input/textarea 面还有 3 处：

1. `#feedback-text`（R234 刚动过的主反馈区）只有 placeholder，没有
   aria-label。placeholder 在用户输入时消失 → screen reader 说 "edit,
   blank"。
2. `#code-paste-textarea`（iOS 剪贴板 fallback 粘贴区）同款问题。
3. `#file-upload-input` 用 `class="hidden"` 但缺
   `aria-hidden="true" + tabindex="-1"`。键盘 Tab 用户会 focus 到不
   可见 input。**关键**：紧挨着的 `#quick-phrases-import-file` 已经
   有正确 pattern → inconsistency 本身就是 hazard。

**做法**：
- 两个 textarea：加 `data-i18n-aria-label` + 初始 `aria-label`，新增
  双语 i18n key（`page.feedbackTextareaAriaLabel` /
  `page.codePasteTextareaAriaLabel`），EN/ZH 都补。
- `#file-upload-input`：补 `aria-hidden="true" + tabindex="-1"`，
  代码注释指向 sibling pattern。

**测试**：3 个 class：
- `TestEveryFormInputHasAccessibleName`：扫全部 input/textarea，
  必须有 wrapping `<label>` / `<label for>` / aria-label /
  aria-labelledby / `aria-hidden=true + tabindex=-1` 之一。
  **审计算法的边界**：wrapping label 判断用 "上一个 unbalanced
  `<label>` open"，简单但对当前 HTML 足够（没有嵌套 label，HTML 也
  不允许）。
- `TestHiddenFileInputPatternIsConsistent`：所有
  `<input type=file class=hidden>` 必须用 R235 pattern。Sanity
  ≥ 2 个 input。
- `TestInputCoverageSanity`：总 input/textarea 数在 [18, 40]，
  baseline 21。catches mass deletion/copy-paste explosion。

**bug 链发现意义**：R235 第一版测试 fail 时报告 `#feedback-text` 漏，
直接证明 R234 commit 时没注意到自己改的 element 本身就有更上游的
a11y 债。这是一个 *self-discovery* 的例子，invariant 帮上一个 invariant
找到漏的 case。

**Verdict**: GOOD。三波 a11y sweep 的收尾，pattern 完整闭合
(WCAG 4.1.2 全表面覆盖)。
