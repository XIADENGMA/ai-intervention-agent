## 1. Cycle theme — "shrink the canvas, sharpen the polish"

This cycle is the inverse of most prior cycles: instead of adding
invariants / hooks / new features, every commit either **deletes**
surface area or **polishes** an existing surface.  Three principles
held across all seven commits:

1. **Frontend removal ≠ backend removal**: every `wastebasket` commit
   explicitly preserves the underlying Flask route + view function +
   payload field names.  Rationale: CI smoke-tests, k8s liveness
   probes, monitoring exporters, and ad-hoc curl-based debug scripts
   are silent consumers we can't enumerate, so prune the UI but leave
   the API exposed.
2. **Differential regression contracts**: each new `test_feat_*.py`
   suite explicitly locks **both sides** of the difference — "UI must
   be absent" + "backend must be present" + "stale historical tests
   must be pruned".  Pattern is symmetric: any future revert would
   trip multiple invariants at once.
3. **Cross-surface parity**: `feat-footer-link-{web,plugin}` introduced
   a `TestWebVsPluginFooterParity` class in the plugin test suite
   asserting the web HTML literal `"AI Intervention Agent {{ version }}"`
   matches the plugin locale value `"AI Intervention Agent {{version}}"`
   character-for-character (modulo Jinja2 vs Mustache spacing).  This
   is the first cross-surface invariant in the test suite — earlier
   parity tests (maybe-BUG6 etc) only locked single-axis ones.

## 2. Highlights & solid wins

### 2.1 BUG6: catch the second-translateDOM-races-real-value race

`data-i18n-value="page.loading"` on `<input>` meant the i18n system
treated the field as a translated placeholder and **clobbered the
freshly-fetched real path** when `translateDOM()` re-ran after the
async default-locale prefetch finished.  The fix is a single
`el.removeAttribute("data-i18n-value")` immediately after writing the
real value — surgical, two-line change in two files
(`multi_task.js::fetchFeedbackPromptsFresh`,
`settings-manager.js::openConfigFileInIde`).  The regression suite
(`test_bug6_config_path_input_i18n_override.py`) locks the strip-after-write
**ordering invariant**, not just presence — which is what actually
matters.

### 2.2 maybe-BUG6: align plugin / web / backend `maxlength`

The plugin's `feedbackResubmitPrompt` and `feedbackPromptSuffix`
textareas had `maxlength="500"` hardcoded since R0; backend constant
`PROMPT_MAX_LENGTH = 100_000` and web UI `maxlength="100000"` both
accept 200x that, so any 600-char paste in the plugin silently lost
99% of it.  The fix updates both plugin textareas to `100000` and the
new test (`test_maybe_bug6_plugin_prompt_maxlength_parity.py`) parses
all three surfaces and asserts equality — making it impossible to fix
one without fixing the others.

### 2.3 feat-remove-test: pruning 9 invariant tests in lockstep

This commit deletes 10 historical test files (R146/R147/R148/R150/R151/R152/R153/R154/R155/R156)
that asserted *existence* of the now-removed UI.  Adding
`TestStaleHookFilesPruned` in the new regression file asserts these
historical files **stay deleted** in future — a meta-invariant.
This pattern is borrowed from the inverse-flip done in
`test_quick_phrases_panel_r130.py` where the `test_export_button_still_present`
was rewritten to `test_export_button_removed_per_feat_remove_download`.

### 2.4 feat-reset-confirm: zero-new-component pattern reuse

Reset settings has been the #1 user pain point ("I clicked once and
had to manually re-tune everything").  The fix opts for `window.confirm`
+ i18n key rather than a custom modal, **deliberately** to match
`quick_phrases.js`'s existing delete-confirmation pattern.  No new
component, no new CSS, no new ARIA wiring needed.  Regression
contract locks:
- `window.confirm` call **before** state mutation;
- cancel branch (`!window.confirm(...)`) **immediately returns**;
- post-confirm path still calls `saveSettings/updateUI/applySettings`.

### 2.5 feat-footer-link-{web,plugin}: cross-surface visual parity

Two-line commit goal ("merge two visual tokens into one link") is
small, but the cross-surface invariant (`TestWebVsPluginFooterParity`)
is the real win — for future contributors, renaming "AI Intervention
Agent" or `{{ version }}` on either side will trip the test.

## 3. Issues found during review (none merge-blocking)

### 3.1 [low] Docs drift in `docs/troubleshooting.{md,zh-CN.md}` § 13

Section 13 ("Client/server payload field-name drift") references the
Activity Dashboard's "Recent logs" row staying "—".  After
`feat-remove-test` removed the dashboard UI, this user-facing
symptom is impossible — only `curl /api/system/recent-logs` directly
can repro that bug class now.  Suggest follow-up: shorten § 13 to
"backend-only — curl-driven repro" framing, or move it to a
backend-troubleshooting subsection.

**Severity**: low.  The docs aren't wrong (the field-drift class still
exists for backend consumers), they just over-promise UI-visible
reproducibility that no longer applies.

**Recommendation**: address as part of the next `loop-task` cycle.

### 3.2 [low] `docs/code-reviews/cr9.md` § R150/R151/R152 narrative outdated

cr9.md narrates the lifecycle of the self-test button + activity
dashboard subsystems.  After their removal in `feat-remove-test`,
the narrative now describes "code that no longer exists".  Same
treatment recommended: backfill a "→ removed in feat-remove-test"
note at the top of the affected sections.

**Severity**: low.  Historical CRs are immutable narrative, not
contracts.  But a one-line backref note helps onboarding.

### 3.3 [low] Backend API documentation may need a "no UI" note

`web_ui_routes/notification.py::/api/system/notifications/test` and
the three `web_ui_routes/system.py` routes now have **no** in-app UI
consumer.  Their docstrings and the auto-generated `docs/api*/*.md`
files don't mention "UI removed in feat-remove-test, use curl
directly" — a future maintainer might wonder why these routes are
"never called" and over-eagerly deprecate them.

**Severity**: low.  No correctness impact, just a footgun for future
prune-attempts.

**Recommendation**: add a one-line `# UI removed by feat-remove-test;
external CI / monitoring scripts still consume.` comment above each
of the 4 route registrations.

### 3.4 [info] `window.confirm` pattern is the right choice **here** but isn't a long-term primitive

`feat-reset-confirm` follows `quick_phrases.js`'s `window.confirm`
pattern.  Both are correct local choices, but `window.confirm`:
- isn't styleable (looks Chrome-native, not project-themed);
- in some iframe / Electron embeddings (Cursor's "glass" mode?) may
  be suppressed entirely (we fall through to no-confirm, which is
  documented in the implementation comment);
- can't easily render markdown / icons / long help text.

If the project grows more destructive actions (delete a saved
quick-phrase preset, clear all tasks, etc.) we should consider a
project-themed modal primitive (probably extending
`aiia-kshelp-overlay` from R144).  Not blocking this cycle.

### 3.5 [info] Footer-link parity test compares **literal text**

`TestWebVsPluginFooterParity` checks that the literal string
`"AI Intervention Agent {{ version }}"` (web) matches
`"AI Intervention Agent {{version}}"` (plugin).  This is fragile if
i18n ever wants to localise the product name (e.g. zh-CN saying
"AI 干预代理 x.x.x" instead).  Today we don't localise the product
name, so the test is fine — but should be loosened to "both surfaces
use the same locale key resolution" if name-localisation lands.

**Severity**: info only.  No action needed unless / until product
name localisation is on the roadmap.

## 4. Test discipline — what got locked, what didn't

### 4.1 What got locked (good)

- BUG6: ordering invariant — `removeAttribute` **after** `value =`.
- maybe-BUG6: 3-way equality — backend constant, web maxlength, plugin maxlength.
- feat-remove-download: absence + backend route preservation + button-count baseline.
- feat-remove-test: 22-case differential contract incl. meta-invariant for stale-test pruning.
- feat-reset-confirm: confirm-precedes-mutation ordering + early-return on cancel + i18n key resolution.
- feat-footer-link-web: anchor structure + 3-locale i18n key migration + theme override.
- feat-footer-link-plugin: similar + cross-surface parity vs web UI.

### 4.2 What didn't get locked (defensible)

- No runtime smoke-test for the merged footer link's `target="_blank"`
  actually opening a new tab — covered structurally by HTML attribute
  assertion; runtime requires a browser harness we don't have.
- No screenshot regression for the visual hover / focus states of the
  new `.version-link` / `.settings-footer-link` — covered structurally
  by CSS selector assertions; visual regression infrastructure (Percy /
  Chromatic / Playwright snapshots) is out of scope for this cycle.
- No e2e click-through test for the `resetSettings` confirm dialog —
  same reason; structural call-ordering test stands in.

These gaps are acceptable for the cycle but should be marked as
follow-ups if the project grows e2e infrastructure.
