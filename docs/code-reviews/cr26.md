## 2. Per-commit audit notes

### R226 — precompress pre-commit hook

- **What landed**: `.pre-commit-config.yaml` gains a `local`
  hook `check-static-precompress-fresh` that calls
  `scripts/precompress_static.py --check` whenever a file in
  `static/(css|js|locales)/` changes.
- **Why it matters**: R223 forgot to regenerate `.br` / `.gz`
  mirrors. `ci_gate.py` caught it but only after pushing to
  CI — wasted ~3 minutes round-trip. Moving the check to
  pre-commit makes the feedback < 1s.
- **Invariant test**: `test_precompress_pre_commit_hook_invariant_r226.py`
  (R226 number reuses the same id as the impl, intentional —
  the test IS the spec). Locks: hook id exists; `entry` is
  exactly the script path; `files:` glob matches the 3
  subdirs; `pass_filenames: false`; `language: system`.
- **Risk**: hook adds ~700 ms latency to every commit that
  touches static assets. Tolerable for the dev experience
  improvement (no broken pushes). Mitigation: the script
  already skips files whose `.gz` mtime > source mtime
  (`skipped_fresh` counter).
- **What I caught**: confirmed R229 commit hit this hook and
  required regen, validating the design.
- **Forward compat**: if static-asset directory layout changes
  (e.g. `static/fonts/` added with its own freshness rule),
  glob must be updated. Test locks the exact glob string so
  this won't silently regress.

### R227 — invariant-test contributor guide

- **What landed**: `docs/contributor-guide-invariant-tests.md`
  + `docs/contributor-guide-invariant-tests.zh-CN.md` with
  4 pattern descriptions (A: static-analysis source-string,
  B: JSON / YAML / TOML structural, C: configuration content,
  D: bilingual i18n parity), a decision tree, a catalogue of
  29 existing invariant tests, and an anti-pattern list.
- **Why it matters**: 290 test files in the repo. Without a
  guide, a new contributor has to read 5-10 invariant tests
  before recognizing the pattern. Now they can read ONE doc.
- **Invariant test**: `test_invariant_test_guide_catalogue_r227.py`
  — every catalogue entry points to a real test file (caught
  3 incorrect paths in the original draft); bilingual parity;
  cross-link from `docs/README*.md` exists.
- **R228 / R229 / R230 catalogue refresh**: R229 commit
  refreshed the catalogue to include R227 itself + R228 (R229
  not yet landed at the time). R230 doesn't refresh the
  catalogue because R230's invariant test is sufficiently
  self-explanatory — but a future contributor reading the
  catalogue will miss R229 + R230 entries until R231 refreshes.
  **Followup F-cycle13-1** captures this.

### R228 — notification body completeness

- **What landed**: `shortcuts.notifyBody` i18n key rewritten
  in EN + zh-CN to list every keyboard shortcut compactly
  + explicit "(full table in DevTools console)" qualifier.
- **Why it matters**: R223 added a settings-panel hint
  pointing users at the console for the full reference. But
  pressing Ctrl+/ ALSO sent a web notification, whose body
  only listed 3 of 7 shortcuts. For users with notifications
  enabled but DevTools closed, the notification was the only
  visible cue — and it silently lied by omission.
- **Invariant test**: `test_shortcuts_notification_body_completeness_invariant_r228.py`
  (11 cases). Locks: both locales mention all 7 shortcuts;
  both mention "console"/"控制台"; len ≤ 250 chars; `{{mod}}`
  placeholder preserved; keyboard-shortcuts.js still uses
  this key.
- **250-char cap rationale**: macOS notification banners
  truncate at ~150 chars in the visible preview, expand to
  ~300 chars on hover. Windows toast limit is 200 chars in
  practice. The 250 cap is conservative.
- **What I caught**: pseudo locale auto-regenerated correctly;
  R226's pre-commit hook regenerated the precompressed mirrors
  on first attempt (validating R226 design).

### R229 — submit button disabled visual state

- **What landed**: CSS `#submit-btn:disabled, #insert-code-btn:disabled`
  rules for dark + light themes (with `!important` to override
  the enabled-state brand gradient `!important`); JS
  `disableSubmitButton` / `enableSubmitButton` stripped of dead
  inline color writes (keeps only `disabled = true/false`).
- **Why it matters**: pre-R229 disabled state was visually
  identical to enabled. Per W3C CSS cascade spec, author
  `!important` always wins over inline non-`!important`, so
  the JS `style.backgroundColor = "#3a3a3c"` was dead code.
  Users couldn't tell the button was disabled until they
  clicked and nothing happened. R229 sinks the disabled visual
  to CSS where it belongs.
- **Invariant test**: `test_submit_btn_disabled_visible_invariant_r229.py`
  (13 cases). Locks: both themes have `:disabled` selector;
  the rule uses `!important`; JS no longer writes inline
  background/color/cursor for the two buttons; JS still flips
  `.disabled` (without which CSS `:disabled` doesn't fire);
  `feedback-text` textarea inline styling is intentionally
  kept (defensive lock).
- **Brand-color discipline**: chose `rgba(255,255,255,0.06)`
  for dark and `rgba(0,0,0,0.05)` for light because R66 brand-
  color baseline blocks new iOS-blue hex (`#0a84ff`) and
  Anthropic-orange (`#d97757`) literals. R66 verified
  unchanged: 34 rgba decimal + 9 hex.
- **`feedback-text` left untouched**: its CSS doesn't use
  `!important`, so inline override actually works there.
  Tackling it would be a refactor for consistency, not a
  bug fix. **Followup F-cycle13-2** captures this.

### R230 — decorative SVG a11y

- **What landed**: `aria-hidden="true" focusable="false"` on
  every `<svg>` in `web_ui.html` (29 of 31 were previously
  missing them). Bulk applied via one-shot helper script
  `scripts/_r230_add_svg_aria.py` (kept in-tree as audit
  trail).
- **Why it matters**: WCAG 2.1 SC 1.1.1. Screen reader users
  pressing submit heard "graphic Submit feedback button" —
  the leading "graphic" added zero info but consumed listening
  time. Same noise on every icon button, every section header,
  the theme toggle, the GitHub link, and the logo. R230 is the
  largest a11y win this project has shipped.
- **Invariant test**: `test_decorative_svgs_aria_hidden_invariant_r230.py`
  (4 cases). Locks: every `<svg>` has `aria-hidden="true"` AND
  `focusable="false"` (allowlist `MEANINGFUL_SVG_CLASSES` is
  empty BY DESIGN — meaningful SVGs must be explicitly added
  to the allowlist AND given `role="img"` + `aria-label`);
  ≥ 2 SVGs keep the reference pattern; total SVG count ≥ 25.
- **Belt-and-suspenders pattern**: pairs `aria-hidden="true"`
  with `focusable="false"` because IE / legacy Edge made SVGs
  focusable by default — and once focused, AT reads them even
  through `aria-hidden`. Standard SVG-icon a11y advice.
- **Bulk-edit safety**: validated open/close tag balance
  (`<svg` count == `</svg>` count == 31) post-edit; ran full
  5642-test suite, 0 regressions; visual git-diff inspection
  showed clean per-SVG insertion at the expected indent level.
- **One-shot script**: kept in `scripts/_r230_add_svg_aria.py`
  (underscored prefix marks as helper / not for general use).
  Idempotent — running it again is a no-op. **Followup
  F-cycle13-3** discusses whether to keep, delete, or promote
  to a permanent linter.

## 7. Lessons learned for Cycle 14

1. **The invariant-test discovery chain is real**. R228 was
   found because R227 made me read keyboard-shortcuts.js.
   R229 was found because R228 made me read app.js. R230 was
   found because R229 made me think about decoupled
   functional/visual state. Future cycles should explicitly
   plan time for "audit-after-write" within each commit.
2. **One-shot bulk-edit scripts work**. R230's 29-place edit
   would be tedious with StrReplace. Writing a focused
   `scripts/_r230_*` helper, running it once, and bundling
   the helper in the commit is a good pattern. The helper
   serves as audit trail. Adopt for future bulk-edit needs.
3. **Pre-commit > CI gate for fast loops**. R226 moved the
   precompress check from `ci_gate.py` (3-min round trip) to
   pre-commit (< 1s). Generalize: any check that's deterministic
   and < 1s should be a pre-commit hook, not a CI gate.
4. **a11y debt accumulates silently**. The 29 missing aria-hidden
   were there since v1.0.0. Visual tests don't catch them.
   Without targeted audit (R230) they would have stayed forever.
   Future cycles should occasionally do "audit-by-category"
   passes (a11y, i18n, perf, security) instead of always
   responding to recent discoveries.
