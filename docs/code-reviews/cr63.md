## 4. 风险 & 隐患

### 4A. 4 处其他 ``__new__(NotificationManager)`` 待迁移 (低优先级)

R319 提供了 ``_create_test_instance()``, R145 ``TestStreakInReal
NotificationManagerSendPath`` 已迁移。剩余 4 处:

- ``test_notification_shutdown_race_r114.py:59``
- ``test_notification_manager.py:1101``
- ``test_notification_manager.py:1230``
- 隐式: ``test_silent_failure_debug_logging_r117.py`` (虽然 grep 没匹配
  到 __new__, 但 R114-related fixture 类似模式)

**风险等级**: 低 (这些测试当前是 stable; R145 是因为更复杂的 send 路径
才触发 flake, 其他 4 处仅访问 `_stats` 等浅层 attr)
**建议**: cycle-34+ 渐进迁移, 不强制 (避免风险扩散)

### 4B. v1.8.0 release notes 准确性

v1.8.0 release notes 提到 "cycle-21 → cycle-33 12 cycles" 及 "110+ R
commits"。这是估算, **未严格核对**。如果未来出 ``CHANGELOG.md`` invariant
要求精确数字, 这条会 fail。

**风险等级**: 极低 (release notes 是约数, 多数项目都这么做)
**建议**: 如需精确, 写个 ``scripts/count_r_commits.py`` 工具

### 4C. test-isolation pattern 与其他 ``__new__()`` 使用者交叉污染

R319 _create_test_instance 返回**非 singleton** instance (每次 fresh)。
但 module-level ``notification_manager = NotificationManager()`` (位于
``notification_manager.py:1716+``) 仍是 singleton, 任何 import-time 触
发 singleton 完整初始化的测试都会留下 instance state, R319 fresh instance
不受影响, **但**其他依赖 module-level singleton 的代码 (e.g.
``web_ui_routes/system.py`` 里 ``from .notification_manager import
notification_manager``) 可能跨测试看到污染。

**风险等级**: 中 (但不在 cycle-33 直接观察到)
**缓解建议** (cycle-34 候选):
- 加 ``NotificationManager._reset_singleton()`` API 让测试可以显式重置
  singleton
- 或者: 加 conftest.py 级 fixture 自动 reset (类似 R72-A root logger
  pattern)
