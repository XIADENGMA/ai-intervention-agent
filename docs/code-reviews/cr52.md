## §2 Per-commit analysis

### §2.1 `069b0a5` — R264 focus capture upgrade (cr51 §5 #1)

**File**: `src/.../js/settings-manager.js` + `app.js` (+ minified +
gzip + br).

**Change**: `showSettings()` snapshots `document.activeElement` into
`this._previouslyFocusedElement`; `hideSettings()` calls
`document.contains(prev) && prev.focus()` with try/catch fallback to
hardcoded `#settings-btn`. Same pattern applied to `openCodePasteModal`
/ `closeCodePasteModal` with module-level
`_codePasteModalPreviouslyFocusedElement` state (aligned with
`_imageModalPreviouslyFocusedElement` from R263a).

**Rationale**: hardcoded restore targets (`#settings-btn` / `#feedback-
text`) silently dropped origin focus in 3 documented scenarios:
keyboard-shortcut opens (`Cmd+,`), multi-task tab `#feedback-text`
mismatches, SSE-rebuild element-reference invalidation.

**Risk**: **low**. Pattern is identical to cycle-8 R263a (already
shipped + green for 14 cycles). Fallback to old hardcoded target
preserves pre-R264 semantics. `document.contains()` defense matches
keyboard_shortcut_help.js cycle-1 R255.

**Test surface**: 8 invariants
(`tests/test_feat_a11y_cycle22_focus_capture_upgrade_r264.py`) split
4 settings-panel + 4 code-paste-modal, including:
- `aiiaWired`-style guard naming alignment with R263a
- `document.contains(prev)` stale defense
- `_previouslyFocusedElement = null` clear-on-close (prevent stale DOM
  leak)
- fallback target ID preservation (double-lock, must keep old semantic)

### §2.2 `92976aa` — R265 listener leak fix (cr51 §5 #4 spillover)

**File**: `src/.../js/settings-manager.js`.

**Change**: `_wireCustomSoundControls()` listener bindings wrapped in
`if (!fileInput.dataset.aiiaWired) { ... fileInput.dataset.aiiaWired =
"1"; ...addEventListener... }`. `refresh()` kept outside the guard
(reset must still refresh UI).

**Real bug found**: original implementation re-bound 3 listeners
every time `_wireCustomSoundControls` was called. Called from `init()`
once + from `resetSettings()` on every reset → reset N times produced
N+1 click handlers per button. Linear audio playback amplification:
| reset N | testBtn click → playSound count |
|---------|----------------------------------|
| 0 | 1 |
| 1 | 2 |
| N | N+1 |

Same applies to filePicker change → `saveCustomSoundFromFile` parallel
N invocations + race condition on statusEl.textContent overwriting.

**No existing test caught it** because all settings-manager tests
stub notificationManager and don't exercise reset → re-entry path.

**Risk**: **low**. Pattern aligned with R263a image-modal
`modal.dataset.aiiaInited` (battle-tested across 14 cycles). `refresh()`
outside guard preserves reset functional intent (UI state refresh).

**Test surface**: 5 invariants
(`tests/test_feat_a11y_cycle22_wire_custom_sound_leak_r265.py`):
- guard must exist
- guard must set `"1"` marker (else guard always truthy = no-op)
- 3 listeners must all be inside guard (can't miss one)
- `refresh()` must be outside guard
- guard naming must be in `aiia*` namespace (meta-test)

### §2.3 `e45dea9` — R266 countdown DOM cache perf

**File**: `src/.../js/multi_task.js`.

**Change**: extracted `_getOrCacheCountdownDom(tid, entry)` helper that
caches `ring` + `circle` + `numberSpan` DOM refs on
`taskCountdowns[tid]._domCache`, with `document.contains(cache.ring)`
stale invalidation check. Applied to both `forceUpdateAllTaskCountdowns`
(visibility-sync path) and `startTaskCountdown::setInterval` body
(1Hz tick path).

**Perf impact**: each tick previously did 3 DOM queries per task
(`getElementById('countdown-${tid}')` + 2× `querySelector`). N=10 task
= 30/s = 1800/min DOM lookups. Now reduced to 0 (cache hit) or 3
(cache miss, only on SSE rebuild). Expected steady-state cache-hit
rate ~99%.

**Risk**: **low**. `document.contains()` defense prevents stale-ref
silent fail after SSE incremental tab rebuild (which replaces
`.task-tab` nodes, invalidating cached children). Task removal GCs
the cache automatically (no separate cleanup needed because cache
lives on `taskCountdowns[tid]` entry).

**Test surface**: 6 invariants
(`tests/test_feat_perf_cycle22_countdown_dom_cache_r266.py`):
- helper must exist
- helper must use `document.contains(cache.ring)`
- helper must cache 3 refs (ring/circle/numberSpan)
- helper must store on `entry._domCache`
- `forceUpdateAllTaskCountdowns` must use helper (positive lock)
  AND must NOT inline `getElementById(\`countdown-${tid}\`)` (negative
  lock — prevents revert)
- same double-lock on `startTaskCountdown::setInterval`

### §2.5 `82e9715` — R268 submit-btn null safe (TypeError pollution fix)

**File**: `src/.../js/app.js`.

**Change**: `submitFeedback()` finally block wrapped in
`if (submitBtn) { ... }` null check.

**Real bug found**: original implementation did
`const submitBtn = document.getElementById("submit-btn"); submitBtn.disabled = false;`
without null check. `await fetchWithTimeout(30s)` window allows:
- auto-resubmit timeout → `showNoContentPage()` removes the entire
  feedback form from DOM (empty state has no submit-btn)
- multi-task SSE re-render replaces nodes (race window)

Result: finally throws `TypeError: Cannot set property 'disabled' of
null` → **pollutes the entire submit flow**:
1. User's feedback **actually succeeded** (response.ok=true,
   `showStatus(success)` displayed, `refreshTasksList` switched to
   empty state)
2. But finally TypeError → await caller receives error → triggers
   **duplicate submission** or overwrites catch's
   `status.networkError` toast → user sees red error, thinks
   "failed", retries

Most insidious because **the bug only manifests in the rare race
window**, making it look like an intermittent flake.

**Risk**: **low**. null check is purely defensive; happy-path
behaviour unchanged. If submit-btn is null, the user already cannot
see it, so skipping UI reset is correct.

**Test surface**: 3 invariants
(`tests/test_feat_correctness_cycle22_submit_btn_null_safe_r268.py`):
- finally must have `if (submitBtn)` null check
- finally must NOT naked-write `submitBtn.disabled = false`
  (double-lock prevents simplification revert)
- finally must NOT naked-write `submitBtn.innerHTML` (double-lock)

### §2.6 `bd708c8` — R269 test bare python binary fix

**File**: `tests/test_changelog_freshness_invariant.py`.

**Change**: replaced 3 `subprocess.run(["python", str(SCRIPT), ...])`
with `subprocess.run([sys.executable, str(SCRIPT), ...])`, hoisted
to module-level `PYTHON_BIN = sys.executable` constant.

**Pre-existing bug** (commit `dadeea3` from v1.8.0 cycle, NOT cycle-22
business work): bare `"python"` binary name relies on top-level
`python` entry in PATH. macOS Homebrew default has only `python3`
symlink; fish-shell default PATH has no `python` alias; `uv run` /
`pipx` isolated environments only expose `.venv/bin/python` internally.
Local pytest run produced 3 `FileNotFoundError` failures in the
freshness invariant suite, **invisible on CI Linux** (which has
`/usr/bin/python` symlink) — exactly the platform-asymmetry test-fragility
class.

**Risk**: **none**. `sys.executable` is the canonical cross-platform
solution; also ensures subprocess runs the same Python version as the
test process (avoids "test runs on 3.11 but `python3` resolves to 3.9"
mismatch).

**Test surface**: not adding new invariant — `sys.executable` is the
"obviously right" pattern, no need to lock it. Manual `rg` scan of
`tests/` post-fix returns zero remaining bare `"python"` subprocess
calls.
