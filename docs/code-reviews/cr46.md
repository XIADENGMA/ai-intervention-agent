## §4 Architectural decisions worth highlighting

### §4.1 Composable theme infrastructure (R250 + R253)

The theme subsystem now has **two complementary
mechanisms** sharing one STORAGE_KEY:

```
┌─────────────────────────────────────────────────────┐
│ R250 anti-FOUC                                      │
│   <head> inline script → reads localStorage         │
│   → writes <html data-theme> BEFORE any paint       │
└─────────────────────────────────────────────────────┘
                       │
                       ▼ (same STORAGE_KEY)
┌─────────────────────────────────────────────────────┐
│ theme.js init                                       │
│   → reads same key → confirms data-theme            │
│   → registers storage listener for cross-tab sync   │
└─────────────────────────────────────────────────────┘
                       │
                       ▼ (writes via UI)
┌─────────────────────────────────────────────────────┐
│ User toggles theme in tab A                         │
│   → localStorage.setItem('theme-preference', ...)   │
│   → storage event → tab B handler → applyTheme      │
└─────────────────────────────────────────────────────┘
```

**Pattern**: "head inline pre-write + module init + cross-
tab event listener" is **reusable** for any future
preference that controls visual state (e.g., density,
accent color, font size).

### §4.2 setInterval → recursive setTimeout for variable intervals

R252 demonstrated that **variable-interval polling
patterns** (where each tick may want to adjust its own
next interval) are cleaner with recursive `setTimeout`
than `setInterval`. `setInterval` works only for fixed
intervals; any backoff/throttling logic forces awkward
gymnastics (clear-and-recreate).

**Reusable pattern**: any future periodic-check (status
polling, health pings, retry loops) with variable
intervals should default to recursive `setTimeout`.

4th (cross-tab theme sync). The closeout doc was already
authored at that point. Updating it post-hoc to reflect
the bonus is **healthy**: the closeout is a snapshot of

**Pattern**: when finishing a cycle reveals a small
adjacent improvement that's natural to ship immediately
(low risk, contained scope, immediately testable), do
it as a **bonus** rather than holding for next cycle
"to follow process".
