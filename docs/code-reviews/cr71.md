### 2.3 perf-baseline — 9th app, 完成 HTTP layer 扇区覆盖

R344 (cold start) → R348 (POST hot path) → **R362 (GET hot path)**, 完
成 HTTP layer 的写 + 读双向 perf baseline 覆盖。`/api/tasks` GET 是
浏览器轮询主路径, 锁:

- 空队列 GET < 80ms (基础读锁 + JSON 序列化开销)
- 5-task 队列 GET < 150ms (数据量扩展开销)

这 2 个 budget 比 POST hot path (100ms/200ms) 略紧 — 反映 GET 无
markdown render / sanitize 开销, 应该更快。后续新增 endpoint 时, 同
类 (读/写) 必须证明在已有 budget 同档内, 否则强制 baseline review。

## 3. 失败 / Spillover

**0 spillover** — cr70 §5 推荐的 A1+B1+C1 全部一次通过, 且**主动扩展**
完成 D1 (tag taxonomy) + E1 (zh-TW i18n)。

cr70 §5 中标 "B1 推荐 Error code uniqueness" — cycle-41 R360 选了
**JS classifier i18n forward resolution** 角度而非完整 error code
uniqueness, 因为:

1. Python 端 error 返回的是中文自由文本 (e.g., "服务器内部错误"), 不是
   error code identifier, 无 cross-Python invariant 价值
2. JS 端 `_classifyFetchError` / `_classifyHttpResponse` 输出的是
   i18n key, 这才是真正稳定的 cross-language 锚点
3. 锁 i18n key forward resolution 比锁 error code uniqueness 更直接
   防 user-visible UX bug (orphan key → 用户看字面 key)

未来 cycle-42+ 可推 cross-Python error path 一致性 (e.g., Python 端
统一返回 error_code enum, 然后 JS 端做 reverse mapping)。
