## 2. Detailed Review

### 2.1 R203 — `_SSEBus._emit_by_type` cardinality cap + WARN-once + AST guard

**Commit**: `676ff00`
**Files**:

- `src/ai_intervention_agent/web_ui_routes/task.py` (+30 / -2): 2 class
  constants (`_EMIT_BY_TYPE_MAX_CARDINALITY = 100` + `_EMIT_BY_TYPE_
  OVERFLOW_BUCKET = "__other__"`), 1 instance flag (`_emit_by_type_
  cap_hit_warned`), `emit()` cap-check branch (overflow routing +
  WARN-once + flag set).
- `tests/test_sse_emit_by_type_cardinality_cap_r203.py` (NEW, +412,
  10 cases / 4 invariant classes + 4 subtests).
- `CHANGELOG.md` (+ Cycle 9 entry).

**What it does**:

R202 exposed `_SSEBus._emit_by_type: Counter[str]` as Prometheus
`aiia_sse_emit_by_type_total{event_type="..."}`. The `Counter` itself
has no key-count upper bound — if upstream `emit()` ever uses a dynamic
string as `event_type` (R198 AST guard already locks source-level emit
sites, but `oversize_drop` replacement + future code misuse + test
residue are real attack/bug surfaces), the Counter grows unboundedly,
causing: memory leak / Prometheus exposition payload bloat / Grafana
cardinality explosion / counter pollution (top-N dashboards filled with
noise).

R203 caps cardinality at 100 (4 R198 schema events + ~10x future
expansion margin + ~10x oversize_drop replacement margin). When the cap
hits: new `event_type` strings route to `__other__` overflow bucket
(preserving `sum(by_type) == emit_total` R202 invariant), WARN logs
once globally, old `event_type` strings continue accumulating
normally.

**Lock discipline**:

R203's most subtle bug surface is the "TOCTOU" race window: thread A
reads `len(self._emit_by_type)` and sees `cap`, gets preempted; thread
B inserts a new key making len = cap + 1; thread A wakes up and inserts
its own key making len = cap + 2 — cap broken. R203 prevents this by
**putting both cap-check and overflow-bucket increment inside the
existing `with self._lock:` block in `emit()`**. The
`TestCardinalityCapLockColocation` AST guard (sibling of R197 /
R202 AST guards) locks this structurally at source level — runtime
concurrency tests can't reliably reproduce the race window, but an AST
guard catches any future refactor that moves either check outside the
lock block.

**Why 100, not 1000 / 10000**:

R198 registered 4 schema events; ~10x expansion margin (~40 future
events) + ~10x oversize_drop replacement margin (~40 unique original
types) ≈ 80 + 4 = 84 < 100. Exposition payload at 100 series × ~100
bytes ≈ 10 KB, well under Prometheus default 100 KB/scrape quota.
1000 would be ~100 KB (tight); 10000 would exceed default quota by
10x.

**Health verdict for R203**: ✅ healthy. Test coverage is exemplary —
runtime tests for behavior + AST guard for structural invariant + R202
sum-invariant regression both pass simultaneously.

**Potential follow-up** (S5 — nice-to-have, low priority):

- `_emit_by_type_cap_hit_warned` is per-`_SSEBus` instance state, not
  per-process. Tests new up bus instances freely (which is correct
  isolation). Production uses module-level singleton (`_sse_bus =
  _SSEBus()`). No bug here, just noting the test/prod parity.
- No explicit metric for "how full is the cap currently" (e.g.
  `aiia_sse_emit_by_type_cardinality_used / 100`). Ops would need to
  count series in `aiia_sse_emit_by_type_total` to know. Acceptable
  for now (PromQL `count(aiia_sse_emit_by_type_total)` works), but a
  dedicated gauge could be ½ day of work in a future cycle if alert
  fatigue ever becomes an issue.

### 2.2 R204 — `aiia_token_age_seconds` Prometheus gauge

**Commit**: `fe41077`
**Files**:

- `src/ai_intervention_agent/web_ui_routes/system.py` (+50 / -0): new
  module-level `_safe_token_age_seconds() -> int | None` helper (~30
  lines), new Security section in `_render_prometheus_metrics`
  (~18 lines), endpoint docstring update.
- `tests/test_token_age_seconds_metric_r204.py` (NEW, +274, 11
  cases / 4 invariant classes).
- `CHANGELOG.md` (+ R204 entry).

**What it does**:

R199 (Cycle 7) added `api_token_rotated_at` persisted to config and
exposed `GET /api/system/api-token-info` JSON endpoint with `{has_
token, token_length, rotated_at, age_seconds}`. Alertmanager wanting
to alert on "90-day-old token" had to scrape JSON itself (bypassing
the standard Prometheus scrape pattern), which extends the ops chain
and adds config surface.

R204 mirrors the same data into Prometheus exposition as `aiia_token_
age_seconds` gauge so alertmanager can write standard PromQL rules
(e.g. `aiia_token_age_seconds > 90 * 86400` per NIST SP 800-63B
rotation guidance).

**Critical design decision · endpoint/metric parity**:

R204 deliberately reuses the same age computation logic as R199's
endpoint inline, but in a separate helper (`_safe_token_age_seconds`).
The two share **zero code** — they're verbatim duplicate logic in
different shapes (endpoint returns dict with multiple fields, helper
returns `int | None` scalar).

**This is risky** — bug fixes must be applied to both copies. R204's
`TestEndpointMetricParity` invariant samples both code paths under the
same config + same point in time and verifies they agree within ≤ 2
seconds (covering clock granularity + test execution time). This guard
runs every CI build and catches drift within one commit cycle.

**Why duplicate code instead of refactor**:

The endpoint inline has 5+ existing tests (R199 era) and refactoring
carries backward-compatibility risk; the abstraction layer is
mismatched (dict vs scalar return); the duplicated code is verbatim
copy-paste with no behavioral difference. R204's CHANGELOG entry
explicitly notes "any bug fix must be synced — this cycle's
`TestEndpointMetricParity` invariant verifies parity in source +
runtime sampling. R205+ may consider unified abstraction layer".

**Failure-mode contract · omit metric vs NaN**:

R204 chose **omit metric when data unavailable** (no token / no
`rotated_at` / parse error / future timestamp) rather than emitting
NaN. Prometheus exposition allows NaN for "unavailable", but `omit`
lets Grafana show "no data" + lets alertmanager use `absent(...)`
to trigger two distinct alert classes (no-token vs token-stale).
Aligns with project's `aiia_uptime_seconds` / `aiia_build_info`
existing `_safe_*` helper contract.

**Health verdict for R204**: ✅ healthy. The endpoint/metric duplication
is a calculated debt with explicit parity guard. Test coverage spans
helper logic + Prometheus rendering format + parity invariant +
exposition contract — comprehensive.

**Potential follow-up** (S6 — important, medium priority):

- **F-204-2**: Refactor R199 endpoint + R204 helper into shared
  internal `_compute_token_age_seconds()` to eliminate the verbatim
  duplication risk. Currently relying on `TestEndpointMetricParity`
  to catch drift; refactor would make drift impossible at the source
  level. ~2 hours including new tests. Defer to Cycle 11+ unless a
  parity test catches actual drift.

### 2.4 R205 — SSE schema runtime validation toggle (`AIIA_SSE_SCHEMA_VALIDATE`)

**Commit**: `90bcaa5`
**Files**:

- `src/ai_intervention_agent/web_ui_routes/task.py` (+80): module-level
  3 constants + `_read_sse_schema_validate_mode` helper; `_SSEBus.__
  init__` env-var read + startup INFO/WARN + 2 new instance attrs
  (`_schema_validate_mode`, `_schema_violation_total`); `_SSEBus.emit`
  early validation block; `SSEBusStatsSnapshot` TypedDict + 2 new
  keys; `stats_snapshot` dict + 2 new keys.
- `tests/test_sse_schema_validate_toggle_r205.py` (NEW, +457, 24
  cases / 8 invariant classes + 12 subtests).
- `CHANGELOG.md` (+ R205 entry).

**What it does**:

R198 (Cycle 7) shipped `EVENT_SCHEMAS` + `validate_payload` API but
explicitly documented "not used in production hot path" — the schema
registry was design-time + IDE source-of-truth only. R205 adds an
**opt-in env-var** (`AIIA_SSE_SCHEMA_VALIDATE` ∈ {off, warn, strict})
that lets ops/debugging contexts catch emit-site bugs at runtime
without code changes:

- `off` (default): single attribute compare, 0 overhead, bit-identical
  to R198 historical behavior.
- `warn`: call `validate_payload`, violations → `logger.warning` +
  `_schema_violation_total += 1` per emit (multi-field error = 1
  count, not N counts), but emit still fanouts non-blocking.
- `strict`: same as warn but `logger.error` (alertmanager severity
  routing), still fanouts, **does not raise**.

**Critical design decision · why strict does NOT raise**:

Strict is the most surprising design point — naive expectation is
"strict = throw an exception". R205 deliberately rejects this:
`emit()` is fire-and-forget; most emit sites (`_on_task_status_change`,
`web_ui_config_sync.py`, etc.) have no try/except wrapper. A raised
exception would cascade into a production crash from a schema bug,
violating R198's "bus does not validate" original rationale.

**Strict vs warn differs only in log level**. Alertmanager can route
ERROR severity to on-call paging while WARNING goes to dashboard-only.
This is the operational scaling lever — code path stays simple.

**Env var sticky read (Twelve-Factor)**:

`AIIA_SSE_SCHEMA_VALIDATE` is read **once at process startup** in
`_SSEBus.__init__`. Changing the env var after process start does
**not** take effect (consistent with other `AIIA_*` toggles in the
project). Invalid values (`yes`, `1`, `STRICT_typo`) fall back to
`off` and emit a startup WARN — preventing the silent "ops thinks
it's on, actually off" failure mode.

**Test engineering — `EnhancedLogger` dedup cache**:

R205's most subtle test challenge: `EnhancedLogger` has a 5-second
dedup cache to prevent log-storms. Cross-test reuse caused `assertLogs`
to miss "repeated violation message" patterns from prior tests in the
same suite. R205 tests clear the cache in `setUp` + each `subTest`
iteration. Production behavior unchanged; this is purely test isolation
hygiene.

**The strict-mode no-raise invariant test** (`TestStrictModeNoRaise
UnderConcurrency::test_concurrent_invalid_emits_do_not_crash`) is the
hardest test in the cycle. 4 threads × 20 invalid emits in strict
mode, asserts: no threads crash, exact violation count (80), and no
emit() raises. Fire-and-forget contract locked.

**Health verdict for R205**: ✅ healthy. The env-var toggle is the
right escape hatch — default off preserves R198 zero-overhead, opt-in
enables ops-time investigation, strict-vs-warn log-level split lets
alertmanager handle escalation. Test coverage is the strongest in the
cycle.

**Potential follow-ups** (S4 — nice-to-have, low priority):

- **F-205-1**: Document `AIIA_SSE_SCHEMA_VALIDATE` in `README.md` env
  var section + sample `compose.yaml` (if any). Currently only in
  CHANGELOG + code docstring. ~30 min; trivial.
- **F-205-2**: Add `aiia_sse_schema_violation_total` Prometheus
  counter mirroring `_schema_violation_total` (currently only
  exposed via `stats_snapshot()` + `/api/system/stats` JSON).
  Alertmanager can use `rate(aiia_sse_schema_violation_total[5m]) >
  0` to catch emerging schema drift. Sibling to R204's metric/JSON
  parity pattern. ~1 hour; **moderate priority** — needed if strict
  mode is ever rolled out to production.
- **F-205-3**: Hot reload toggle — read env var on every emit instead
  of `__init__`-sticky. Trade-off: + dynamic ops (flip strict via
  `systemctl set-env` without restart); - per-emit os.environ.get is
  ~1µs which adds up at 1k emit/sec. **Defer to Cycle 11+ unless
  ops needs hot toggle**. Acceptable to keep sticky for now since
  the rest of the project's `AIIA_*` toggles are also sticky.

## 5. Versioning / release decision

**Recommended**: pack Cycle 9 into a single `v1.7.3` patch release.

Rationale:

- All 4 features are backward-compatible (R205 toggle default-off,
  R203 cap default-100 invisible in normal use, R204 new metric
  doesn't break existing, R206 pure docs);
- No CHANGELOG `[Unreleased]` already includes all 4 — release commit
  needs only to move `[Unreleased]` → `[1.7.3]` + version bumps;
- 4-feature cycle is the project's typical release cadence (Cycle 8
  shipped 4-5 commits as v1.7.2 with ty-fix bundled).

**Alternative**: bump to `v1.8.0` if R205's `AIIA_SSE_SCHEMA_VALIDATE`
env var is considered a "new user-facing config surface". This is
defensible per Semantic Versioning §7 ("MINOR version when you add
functionality in a backward compatible manner"), but the project's
v1.7.x history shows multiple minor user-facing additions in patch
releases (e.g. v1.7.1 added R199's `api_token_rotated_at` config
key — also user-facing — and shipped as patch). Pattern consistency
favors `v1.7.3`.

**My recommendation**: `v1.7.3`. Schedule release after CR#22 lands
on `main` so the cycle review itself doesn't trigger a redundant
release.
