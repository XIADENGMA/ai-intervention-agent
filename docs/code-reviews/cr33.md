## 2. Highlights

### 2.1 `d8a7a7c` extend-race fix — the textbook way to close a concurrency review finding

Issue identified by cr32 §3.1 [medium]: `Task.extend_deadline`
performed read-modify-write on `extends_used` and
`auto_resubmit_timeout` **outside** any lock; the route handler
held no lock around it; two concurrent `POST /api/tasks/X/extend`
calls would both see `extends_used = N`, both pass the
`< max_extends` check, both write back `N+1`, and **both** `+=
seconds`. End state: `extends_used` is wrong by 1, timeout is wrong
by `+seconds` × (concurrent_count - 1).

The fix is correct and textbook:

1. **Introduced a queue-level facade** `TaskQueue.extend_task_deadline(
   task_id, seconds, *, max_extends, min_seconds, max_seconds)` that
   takes the queue's existing `_watched_write_lock` *before* calling
   `task.extend_deadline(...)`. Now the read-modify-write is atomic
   relative to other queue mutations.
2. **Route handler switched to the facade.** No lock leak; the route
   never sees the task reference until after the facade returns the
   tuple `(success, error_code, extends_used_after, timeout_after)`.
3. **Threading regression test** (`test_cr32_extend_race.py`):
   `TestExtendDeadlineRaceEliminated::test_concurrent_extends_respect_max_extends`
   spins 10 threads through a `Barrier`, asserts `extends_used <=
   max_extends` **and** `final_timeout == base + extends_used *
   seconds` (the second assertion is the one that would have caught
   the bug — `extends_used` alone could match by coincidence under
   the buggy code).
4. **Anti-regression invariants** (`TestRouteUsesQueueFacade`): two
   AST-style source scans assert (a) the route file mentions
   `task_queue.extend_task_deadline(` and (b) does **not** call
   `task.extend_deadline(...)` outside of test comment strings (the
   test deliberately masks comments before scanning, so a future
   "let me revert the facade for clarity" PR is caught by lint).

This is the right test density for a medium-severity concurrency
fix: a probabilistic threading test alone could be flaky, so the
deterministic AST invariants give us the **structural** guarantee
that the threading test exercises the right code path.

### 2.2 `df97acc` VSCode zh-TW locale — the right way to fix a "shipped but invisible" feature

Issue identified by cr32 §3.2 [medium]: web UI shipped `zh-TW.json`
but the VSCode extension's `i18n.js` only had `normalizeLang`
returning `"zh-TW"` for relevant BCP-47 tags **without** a
corresponding `locales/zh-TW.json` file. End user impact: VSCode
users whose system locale was `zh-Hant-TW` etc. fell through to
**English**, silently.

The fix is correct:

1. **Parameterized the generation script** (`scripts/gen_zhtw_from_zhcn.py`)
   to take `--variant {web,vscode}` and `--all`. Both variants now
   share the same character/phrase mapping, so future zh-TW polish
   only needs to touch one script.
2. **Generated and committed** `packages/vscode/locales/zh-TW.json`
   under `--variant vscode`. The script is deterministic, so this
   is reproducible; the pre-commit hook for build-artifact freshness
   (R246) verifies that downstream.
3. **Updated `packages/vscode/webview.ts`** to eagerly `loadLocale("zh-TW")`
   in `_preloadResources` and include `zh-TW` in the `allLocales`
   fallback list. Without this, the locale file would exist but
   never be loaded.
4. **Fixed two CHAR_MAP_v2 gaps** discovered while generating the
   VSCode file: `测 → 測`, `缀 → 綴`, `内 → 內`. These weren't in
   the web variant's vocabulary but appear in the VSCode strings.
5. **Regression test** (`test_cr32_vscode_zhtw.py`) asserts file
   exists, contains specific Traditional Chinese marker characters
   (`設`, `饋`, `關`) that **do not** appear in zh-CN.json, so the
   test would catch a future regression where the variant accidentally
   emits Simplified again.

Process learning embedded in the script: the `--variant` parameter
means we now have a single source-of-truth tool; future projects in
this monorepo that want zh-TW can ride the same pipeline rather
than copy-pasting yet another conversion table.

### 2.3 `0e6b1fa` custom-sound — the longest commit of the cycle, executed with discipline

Mining cycle-1 §3.4 was scoped as "LOW ROI, ~120 LoC" in the
backlog. This shipped at **~250 LoC application code + 350 LoC
test + 175 LoC i18n** (regenerated) ≈ 775 LoC; the LoC ratio is
**45% test**, which is the right answer for a new persistence-
adjacent feature.

Design discipline highlights:

1. **`playSound(soundName=null, ...)` dispatch redesign.** Default
   parameter changed from `'default'` to `null`. When null, the
   method dispatches to `'custom'` if `audioBuffers.has('custom')`
   else `'default'`. **Explicit** `playSound('default')` (used by
   the Test button) preserves original semantics so the user can
   always preview the built-in sound regardless of custom upload
   state. This is the **right** indirection: a single point of
   policy, two clear escape hatches, and no breaking change for
   any caller that already passed `'default'` explicitly.
2. **Versioned localStorage key** `aiia.notif.customSound.v1`. The
   `.v1` suffix is forward-thinking: when (not if) the schema
   evolves — e.g. to add `volume_db` or `playback_rate` — the
   migration code can check the key version atomically rather than
   relying on heuristic schema sniffing.
3. **6 distinct error codes** from `saveCustomSoundFromFile`:
   `no_file / invalid_mime / too_large / read_failed /
   storage_failed / decode_failed`. Each maps to a separate i18n
   key so the user sees the **actual** problem, not a generic
   "upload failed". This is significantly better than competitor
   UX which dumps stack traces.
4. **CSS uses `--primary-500` instead of `var(--color-primary,
   #007aff)`.** First draft of this commit hit the R66 brand-color
   guardrail; instead of suppressing the lint with a comment, the
   author switched to the project's actual design token. The hex
   `#007aff` was iOS-blue muscle memory; the project's brand is
   purple. This catch + correction is exactly what R66 exists for.
5. **`settings.reset` also clears custom sound.** This is a subtle
   correctness point: "factory reset" must mean factory, not
   "factory + your uploaded MP3". The wiring is in
   `_wireCustomSoundControls` + reset path, and the regression
   test verifies the test ID + i18n key linkage so a future
   `resetSettings` refactor can't silently skip the custom-sound
   cleanup.

Minor scope-creep that **didn't** happen and was correctly
deferred:

- Multi-slot sound management (competitor also single-slot)
- Server-side upload endpoint (would add SSRF / storage attack
  surface)
- Built-in sound pack (needs license-clear audio assets)

The mining doc update (`09656ab`) records all three as explicit
"What was NOT shipped (deferred)" — keeping the rationale near the
implementation rather than scattered across issue tracker.

### 2.4 `5b44d26` cr32 §3.3 fix + perf-audit doc — boxing the audit into a defendable artifact

Two distinct concerns landed in one commit, which would normally
be a smell, but here it's correct: the `extend_deadline` keyword-
only-mandatory change is *part of* the perf-audit findings (it's
the only API-level hardening the audit produced), so splitting
them would put the audit doc into a state where its `§4. cr32 §3.3
fix` section referenced a not-yet-landed change.

Audit doc highlights:

1. **Status-first table in §1.** "All three hot paths (TaskQueue /
   _SSEBus / _upload_helpers) are at their abstraction-layer
   optimum; this cycle ships only 1 API hardening." This is the
   right tone — the audit's value is not "we found 8 things to
   fix" but "we verified 3 subsystems are optimized to their
   architectural ceiling, and here's the evidence."
2. **Defer list with ROI + trigger condition** (§5). Four future
   tracks, each with explicit "when would we revisit this":
   - SSE `task_extended` broadcast: revisit during Feature Mining
     cycle-2 SSE batch.
   - TaskQueue auto-cleanup: revisit if `max_tasks ≥ 500`.
   - Image streaming: revisit if `>5 concurrent uploads × >50MB
     each` telemetry observed.
   - `_persist()` debounce: revisit if `task add freq > 100/s`.
3. **§6 reproducibility recipe.** Three commands (rg / pytest /
   git log) that re-run the audit. This is how you make audits
   non-cargo-culted: next cycle's reviewer can re-derive these
   findings without trusting this author's word.

API hardening highlights:

1. **`Task.extend_deadline` and `TaskQueue.extend_task_deadline`
   now require `max_extends / min_seconds / max_seconds` as
   keyword args without defaults.** The drift risk was: route
   reads from `server_config.COUNTDOWN_*`, but if a future caller
   forgets to pass them, the old hardcoded `3 / 10 / 300` would be
   silently used, diverging from production config after a
   `feedback.toml` change.
2. **New invariant test** `test_extend_keyword_only_enforced` uses
   `cast(Any, t)` to bypass ty's static check, then asserts
   **runtime** `TypeError`. The `cast(Any, ...)` indirection is
   deliberate and documented in the docstring — the test is
   specifically about runtime enforcement, not type-check
   enforcement (the latter is automatic and doesn't need a test).

This is the right hardening: it's catch-only (existing well-formed
callers still work because they all already pass these keywords),
zero behavior change, and prevents an entire class of future
config-drift bug from ever shipping.

## 3. Issues found

Going through each commit looking for problems that should have
been caught earlier or that need follow-up.

### 3.1 `d8a7a7c` extend-race fix [INFO — initially flagged, then corrected]

- **Original concern (during cr33 drafting)**: I thought the test
  used `time.sleep(0.001)` to "give the OS time to schedule." On
  re-reading the code I confirm that the test uses **only**
  `threading.Barrier(N)` and does not call `time.sleep` at all.
  That's the correct design: `Barrier` is the strongest primitive
  for "all threads start the contested section simultaneously",
  and any `sleep`-based amplifier would actually be a regression.
- **Action taken**: added a multi-line comment inside `worker()`
  documenting that Barrier is the only synchronization primitive,
  and that future refactors changing this need to update cr33 §3.1
  too. No production-code change.

### 3.2 `df97acc` VSCode zh-TW [LOW]

- The CHAR_MAP_v2 grew to **445 entries** after the cr32 §3.2 fix.
  At some point this becomes unmaintainable; the project should
  evaluate adopting an established library (e.g. `zhconv` or
  `opencc-python`) before the table exceeds ~600 entries. The
  upfront cost is ~1 dev-day; the saving is "no more whack-a-mole
  on missing characters every time we add a new string."
  **Action**: file under future-mining-cycle, "evaluate zhconv
  dependency vs. continued inline maintenance."

- `test_cr32_vscode_zhtw.py::test_contains_traditional_chinese_markers`
  pins three characters (`設`, `饋`, `關`). This is correct for
  catching "we accidentally regenerated as Simplified", but it's
  fragile: if future zh-CN.json strings drop all occurrences of
  these three sources, the test passes vacuously even though the
  output may have other Simplified-residue bugs. **Action**: switch
  marker selection to "the set of all Traditional codepoints
  emitted, minus any Simplified codepoint that survived" — but
  this is enough complexity that we should only do it if we see
  an actual false-pass.

### 3.3 `0e6b1fa` custom-sound [LOW]

- **localStorage quota error path is generic.** `saveCustomSoundFromFile`
  catches `DOMException` for `storage_failed`, but doesn't distinguish
  "quota exceeded" from "storage disabled by browser policy" or
  "private browsing mode". User sees "storage_failed" for all three.
  In practice these are rare enough that distinguishing them adds
  more code than user benefit, but if user reports start arriving
  along the lines of "I'm in private mode and can't upload" we
  should add a second error code `storage_quota` for the quota
  case (detectable via `e.name === 'QuotaExceededError'`).
  **Action**: defer to user-feedback-driven.

- **No max audio duration check.** A user could upload a 30-minute
  ASMR file that fits under 700KB (lo-bitrate). The notification
  would then play for 30 minutes on every task event. Mitigation:
  the `<audio>` element decode and play is already async; nothing
  stops the next event from cutting it off. **However**, the
  decoded `audioBuffers['custom']` lives in memory; a 30-minute
  decoded PCM buffer at 44.1kHz/2ch is ~150MB. This is a real
  memory hazard. **Action**: add a `duration > 30s` check in
  `saveCustomSoundFromFile` post-decode; reject with new error
  code `duration_too_long`. **Priority: medium** because it's not
  attacker-exploitable (user uploads their own file) but is a
  realistic foot-gun.

- **No way to preview before saving.** UX flow is: upload → save →
  test. There's no "preview without persisting" step. Users who
  upload the wrong file find out only after committing it to
  localStorage. The `clear` button is right there, so the recovery
  cost is one click, but a "Preview" button before save would be
  more polished. **Priority: low** — feature-completeness, not
  bug.

The `d8a7a7c` ratio is the standout. 5:1 for a 5-line race fix is
the right discipline for a medium-severity concurrency bug — the
fix itself is small, but the **invariant** the fix establishes
needs to be regression-proof against future refactors. The two
AST-style tests (`test_route_calls_task_queue_facade` and
`test_route_does_not_call_task_extend_deadline_directly`) are
worth highlighting as a pattern: they enforce a **structural**
property of the code (route uses facade), not a behavioral one
(no two concurrent calls produce wrong count). The structural test
catches the bug-class at PR time even if a future PR rewrites the
behavioral one.
