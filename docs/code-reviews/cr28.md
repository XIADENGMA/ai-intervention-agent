### R236 — ty as pre-commit hook (F-cycle14-1)

**Files**: `.pre-commit-config.yaml`,
`tests/test_ty_precommit_hook_invariant_r236.py`,
bilingual guide §6 catalogue.

**Root cause being closed**: v1.7.5 was tagged + pushed → release
CI ran `ci_gate.py --ci` → `ty check .` step failed with
`unresolved-attribute` on `re.Match | None` (the
`self.assertIsNotNone(match)` pattern). Tag immutability +
release runbook (R206 v1.7.2 hard-won) forced v1.7.5 abandonment
+ v1.7.6 supersession + bundled the ty-fix.

**Why ty wasn't local**: it WAS in `ci_gate.py` (called by
`test.yml` too), but `ci_gate.py` is invoked only in CI, not
during normal `git commit`. Developers ran ruff (pre-commit
hook), brand-color (pre-commit hook), precompress (R226
pre-commit hook) — but not ty.

**Fix**: add `ty-check` hook to `.pre-commit-config.yaml`:

```yaml
- repo: local
  hooks:
    - id: ty-check
      name: ty 静态类型检查（R236 / F-cycle14-1 护栏…）
      entry: uv run ty check .
      language: system
      pass_filenames: false
      files: \.py$
```

**Why `pass_filenames: false`**: Python module interdependencies
make partial type-check unreliable; ty has incremental cache
internally. Same reasoning as ruff's project-wide mode.

**Performance**: 800-1200 ms with cache, same tier as ruff +
brand-color hooks. Acceptable.

**Invariant**: 3 classes / 5 cases. Locks: hook id exists +
entry actually runs `ty check` (defends against silent rename
to `echo skip`) + files filter contains `.py` (defends against
`.md` typo) + stage default (not moved to `manual` / `pre-push`
which would defeat the local-feedback purpose) + `ci_gate.py`
still invokes ty (CI is source of truth, pre-commit is fast
shadow).

**Self-validation during commit**: the new hook ran during R236's
own commit (and every subsequent commit this cycle). Visible in
hook output as `ty 静态类型检查（R236 / F-cycle14-1 护栏…）...Passed`.

**Verdict**: EXCELLENT. Mirrors R226 pattern (CI → pre-commit
migration for fast feedback), proven by 5 subsequent commits
this cycle all triggering the hook successfully.

### R238 — Tab focus trap + lock focus restore (a11y wave 4 imperative)

**Files**: `src/ai_intervention_agent/static/js/app.js`,
`src/ai_intervention_agent/static/js/settings-manager.js`,
`tests/test_modal_focus_trap_invariant_r238.py`.

**Gap closed**: Both modals had Esc-to-close and (legacy)
focus-restore on close. Neither trapped Tab. R238 adds:

- `_modalFocusTrap(panel, event)` helper in `app.js` (~14
  lines). Standard W3C focusable selector (`button:not([disabled])`,
  `[href]`, `input:not([disabled]):not([type="hidden"])`,
  `select:not([disabled])`, `textarea:not([disabled])`,
  `[tabindex]:not([tabindex="-1"])`). Filter visible via
  `offsetParent !== null && !hasAttribute("aria-hidden")`. On
  Shift+Tab at first → focus last; on Tab at last → focus first.
- `handleCodePasteModalKeydown` calls `_modalFocusTrap` after
  Esc check.
- `_settingsFocusTrap` method on `SettingsManager` class.
  Identical logic.
- `_settingsEscHandler` extended to call trap on `Tab`.

**Tradeoff with helper duplication**: I considered creating a
new `static/js/focus-trap.js` module + loading from
`web_ui.html`, but rejected — adding a JS file requires
Flask serve-static config touch + precompress entry + new
test surface. Inline duplication of ~14 lines in 2 callers
is cheaper. If a 3rd modal lands, R241 should extract the
helper.

**Test methodology**: Pattern B (text/regex based) because we
can't run JS at unit-test time. The regex pattern
`function\s+_modalFocusTrap\s*\([^)]*\)\s*\{(.+?)\n\}` was
the trickiest piece — first version used `[^}]+` (greedy
non-`}`) which stopped at the nested `if` block's brace. Fixed
to `.+?` with module-bottom `\n}` sentinel.

**Verdict**: GOOD. Real functional fix + invariant lock. The
helper duplication is intentional and documented.

### R239 — README star-count freshness window (Pattern D)

**File**: `tests/test_star_counts_freshness_invariant_r239.py`.

**Problem**: Both READMEs end "Related projects" with
"Star counts are approximate snapshots (last reviewed
2026-05); …" / "上面的 stars 是粗略快照（最近核对：2026-05），…".
The 5-figure star numbers (`~3.8k`, `~1.4k`, `~310`, `~30`)
above this disclaimer become unverifiable claims as the date
ages — first-time readers infer accuracy from the date.

**Design choices**:

- **12-month threshold**: yearly refresh is the standard for
  "Related projects" sections in mature OSS READMEs. Star
  counts are not security-critical so monthly would be over-
  engineered.
- **Environment-override**: `R239_STAR_COUNT_MAX_AGE_MONTHS=N`
  for emergency / one-off investigation. Intentionally NOT
  bakeable into the test code itself.
- **EN/ZH parity**: same review session should refresh both.

**Date arithmetic edge case**: I used naive
`(later.year - earlier.year) * 12 + (later.month - earlier.month)`
because both dates are anchored at day-1-of-month. This avoids
timezone / DST issues that pure `date.today()` could introduce
in CI runners.

**Recovery path** (documented in test docstring): re-check each
upstream's GitHub star count → update `~Nk` / `~NNN` figures →
bump YYYY-MM in both READMEs.

**Verdict**: GOOD. Pattern D drift-detection in the R231 + R233
spirit. Documented recovery path means the maintainer can fix
without reverse-engineering the test.

### R240 — `inert` background on open modal (a11y wave 4 trilogy complete)

**Files**: `src/ai_intervention_agent/static/js/app.js`,
`src/ai_intervention_agent/static/js/settings-manager.js`,
`tests/test_modal_inert_background_invariant_r240.py`.

**Gap closed**: R238 trapped keyboard Tab but did not block:
- Mouse click on background (R238 only intercepts key events)
- Programmatic `element.focus()` calls from background scripts
- Screen reader swipe gestures landing on background

`inert` is HTML5's canonical answer to "this subtree is not
part of current interaction". Chrome 102+ / Firefox 112+ /
Safari 15.5+ — all stable since 2022.

**Defensive pattern**:

```javascript
try {
  container.inert = true;
} catch (_e) {
  container.setAttribute("inert", "");
}
```

Both forms apply the attribute; the IDL setter is preferred
for type safety. The try/catch protects against polyfills that
shadow the setter or older engine versions without the IDL
property.

**Why `.container` (not `<body>`)**: We need to leave the modal
itself interactive. `.container` is the `role="main"` wrapper
that excludes the panel elements (which live as siblings
under `<body>`). Marking `<body>` would also disable the
modal.

**Test methodology**: Same Pattern B as R238. 5 classes / 6
cases. Locks open+close paths for both modals + defensive
try/catch pattern.

**Verdict**: GOOD. Completes the a11y wave 4 trilogy. With
R237 + R238 + R240 the modal interaction barrier is now
complete in all 3 modalities.
