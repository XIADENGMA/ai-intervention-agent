### 3C. R313 主动质量改进的 ROI

R313 是 cr62 第二个**主动改善源码质量**的 invariant (cr61 R306 是第一个,
修复 R249 mining-9 P0 bug)。R313 的具体改进:

1. **freeze_task_deadline 409 description**: 从 "task 已经被冻结过" 扩
   充到 "**idempotent No-Op**: 第二次 freeze 不破坏状态, 仅返回 409 让
   前端知道按钮无需再点", 明确 contract
2. **activate_task docstring**: 新增 "**幂等性 (idempotent)**" 段落,
   完整解释 P6R-2 修复上下文 (multi-tab race / UX 错觉防御)

**ROI 计算**:
- 直接收益: 后续维护者读 docstring 就理解 contract, 不会修改幂等行为
- 间接收益: invariant 锁文档, 任何企图改 docstring 删除幂等关键词的
  改动都会触发 test fail, 强制 review
- **次生收益**: v3.8 pattern 用 docstring 关键词作为静态 contract
  source, 比 runtime test 更稳 (runtime test 可能因测试污染 flake)

### 3D. R315 数字 baseline 维护

R315 锁了 4 个 task_queue 常量, 引入 3 个 meta-lints:
- REJECT > WARN (语义不变量)
- TIMEOUT / SCAN_INTERVAL >= 5 (扫描频次)
- 必须用 `N * 1024 * 1024` 形式 (人类可读 MB 语义)

这 3 个 meta-lints **不只是值检查**, 而是**语义检查**:
- 即使把 WARN 改成 7 MB / REJECT 改成 8 MB (REJECT < WARN), 字面量
  本身合法, 但语义 invariant 失败, R315 检测到
- 即使把 `6 * 1024 * 1024` 改成 `6291456` (相同值不同表示), R315 检测到
  并提醒保留 MB 单位语义

这是 v3.6 perf-baseline pattern 从 "值精确锁" 进化到 "语义不变量锁" 的
关键演进, 也是 cycle-32 最有价值的方法论收获之一。

## 4. 风险 & 隐患

### 4A. R145 NotificationHealthStreak flake (高优先级 cycle-33 候选)

cycle-32 全 regression 第一次跑出现 3 个 R145 测试 fail, 第二次跑全过。
分析: R145 测试**强依赖**全局 NotificationManager class 状态。即使 R305
使用 `loadfile` 分布, 也只能保证**同文件**测试在同 worker — 但 R145 与
**其他文件**中可能改 NotificationManager 类状态的测试在不同 worker 时,
仍会因为 child process 跨文件 import 顺序不同而出现污染。

**风险等级**: 中 (用户没见过, 但 CI 偶尔 flake 会增加误报噪音)
**缓解建议** (cycle-33 候选):
- 加 R145 测试的 setUp / tearDown 显式重置 NotificationManager class 状态
- 或者: 把 R145 测试改为不依赖全局 NotificationManager (用 dependency
  injection)
- 或者: 给整个 NotificationManager 加上**幂等的 reset** API (与 R313
  guidelines 一致)

### 4B. v3.8 幂等 contract 白名单维护

R313 引入了**测试文件级**白名单 `_IDEMPOTENT_ENDPOINTS_WHITELIST`。当
新增 `/api/tasks/<id>/<new_action>` POST endpoint 时, R313 future-guard
会立即 fail, 强制开发者:
1. 决定新 endpoint 是否幂等
2. 加到白名单或黑名单
3. 在 endpoint docstring 标注幂等性
4. 更新 R313 baseline 数字

**风险等级**: 低 (这正是 invariant 的设计目的, 强制 review)
**缓解建议**: 在 `docs/contributor-guide-invariant-tests.md` (如果存在
或新建) 加一节说明 R313 的 4 步流程, 让贡献者知道为什么 R313 会 fail。

### 4C. perf-baseline 6 次应用后的扩展疲劳

R315 是 perf-baseline 的第 6 次应用, 域已经非常宽 (infra / runtime /
business / network / lifecycle)。**潜在风险**:
- 后续 cycle 找不到新 domain 应用, pattern 进入 "saturation"
- 开发者可能为了凑应用次数, 把不重要的常量也加 perf-baseline 锁定

**风险等级**: 低 (cycle-32 R315 仍有充分价值, 不是凑数)
**缓解建议**: cycle-33+ 应聚焦 v3.7 (2 个 pattern 推 3rd app) 和 v3.8
(推 2nd app + 新 pattern), 让 perf-baseline 自然休息一两个 cycle。
