## 2. Detailed Review

### 2.1 R200 — `api_token_rotated_at` cascade-clear when token revoked

**Commit**: `d407524`
**Files**:

- `src/ai_intervention_agent/config_modules/network_security.py`
  (+27 / -2): `_validate_network_security_config` end-of-function
  sanity-check block; `update_network_security_config` writeback path
  consolidation
- `tests/test_api_token_cascade_clear_r200.py` (NEW, +338, 13 cases /
  4 invariant classes)
- `CHANGELOG.md` (+30 entry under `[Unreleased]`)

**What it does**:

R199 introduced `api_token_rotated_at` as a config field that records
**when** the API token was last set. CR#20 §4.1 flagged a latent
inconsistency: admin clears `api_token` (via `update_network_security
_config({"api_token": ""})`), but the **stale** `api_token_rotated_at`
timestamp remains in config — `GET /api/system/api-token-info` then
reports `{has_token: false, rotated_at: "2026-04-15T12:34:56Z", ...}`,
which is logically impossible (no token, yet rotation timestamp lingers).

R200's fix: at the **end** of `_validate_network_security_config`, if
`api_token == ""` and `api_token_rotated_at` is set, **silently clear**
the timestamp. This runs on every config load and every config write
(both paths reuse `_validate_network_security_config`), making the
constraint structurally enforced regardless of caller.

**What's good**:

- The fix is at the **right architectural layer**: validation, not
  individual update endpoints. Future endpoints that mutate `api_token`
  (or admins editing `config.toml` directly + sending SIGHUP for hot
  reload) automatically benefit.
- Idempotency is explicit: setting `api_token=""` twice doesn't toggle
  any state — `rotated_at` is already `None` after the first clear.
  Tested by `TestApiTokenCascadeClearIdempotency`.
- The implementation chose **silent clear** over **raise ValueError**
  (CR#20's other option). Right call — the inconsistency is a state-
  recovery scenario, not a malicious input; raising would force admin
  tooling to handle the error vs just observing the cleaned-up state.
- 4 invariant classes (Lifecycle / Idempotency / Endpoint consistency
  / Schema integrity) cover the failure modes that matter:
  - Set-token → clear-token → set-token again must yield fresh
    `rotated_at`.
  - `GET /api/system/api-token-info` returns `{has_token: false,
    rotated_at: null}` consistently.
  - Migration from a pre-R200 stale config file (with ghost
    `rotated_at`) is cleaned on first load.

**What could be better** (none critical):

- Cascade-clear is implemented in `_validate_network_security_config`
  but **not logged** (silent transform). If an admin is debugging a
  config and expects `rotated_at` to persist after they manually
  re-edit `api_token`, they might be confused. A single
  `logger.debug("R200: cleared stale api_token_rotated_at because
  api_token is empty")` line would help. **Minor nice-to-have.**

### 2.3 v1.7.2 CI hardening — `ty 0.0.34` 38 diagnostics + docs-sync

**Commits**: `81895d2` (the ty bundle) + `35f9671` (docs-sync follow-up).

**Files** (cumulative across both commits):

- 4 source files (`enhanced_logging.py`, `mcp_tool_call_metrics.py`,
  `notification_manager.py`, `web_ui_routes/system.py`)
- 12 test files
- `docs/api/enhanced_logging.md` + `docs/api.zh-CN/enhanced_logging.md`
  (mechanical regen for the `get_current_log_level` return-type widen)
- `CHANGELOG.md` (+85 lines for the v1.7.2 entry)

**What it does**:

After Dependabot retriggered CI (`83c2bf7` authlib bump), the `Tests`
workflow turned red on `uv run ty check .` — **not** because of the
authlib bump but because v1.7.1's release pinned `ty 0.0.34`, a
stricter version than its predecessor. The strictness exposed **38
latent diagnostics** (31 errors + 7 unused-ignore warnings) spread
across 16 files. v1.7.2's commit `81895d2` fixes them all in one
audit pass; `35f9671` corrects the post-release docs-sync miss that
broke a docs-parity gate.

**Categorization of the 38 diagnostics** (full RCA in CHANGELOG
[1.7.2]):

| Category                          | Count | Treatment                          |
| --------------------------------- | ----- | ---------------------------------- |
| Real return-type bug              | 1     | Fix signature                      |
| Dict-literal type inference quirk | 2     | `cast("dict[str, Any]", ...)`     |
| `isinstance` narrow regression    | 1     | `cast` after the check             |
| `assertIsNotNone` not narrow-form | 3     | Add `assert x is not None`         |
| Subscriptable check on `object`   | 4     | `cast` to expected concrete type   |
| Test-internal typo (real bug)     | 1     | Fix `get_web_ui_config` → `get_section("web_ui")` |
| Deliberate-violation test inputs  | 6     | Add `# ty: ignore[invalid-argument-type]` |
| Generic-method-override (fastmcp) | 1     | Add `# ty: ignore[invalid-method-override]` |
| Already-fixed-upstream warning    | 7     | Remove unused `# ty: ignore` lines |
| `unresolved-import` of `sys.path` injected script | 1     | Add `# ty: ignore[unresolved-import]` |
| Mock framework `side_effect` widening | 1     | Widen `Exception | None` → `... | type[Exception] | None` |
| Test minimal-fake doesn't match Protocol | 4     | Add `# ty: ignore[invalid-argument-type]` |

11 categories, only **2 real bugs** (return type + method-name typo);
the rest are tool-side noise that the ignore policy or `cast` patterns
handle correctly. The 7 "removed unused ignores" indicate `ty 0.0.34`
silently fixed some prior false positives — those ignores were dead
weight and removing them is healthy.

**What's good**:

- **Each ignore has an inline rationale comment** explaining why the
  silencing is correct. The `mcp_tool_call_metrics.ToolCallCounter
  Middleware.on_call_tool` ignore even cites the upstream fastmcp
  source-file docstring example showing this is the documented type-
  narrow pattern. This makes future audits trivial — when `ty` adds
  generic-method-override support, anyone can grep `# ty: ignore` and
  re-evaluate.
- **No wholesale `# ty: ignore-file` directives** — each silencing is
  scoped to one line, so future regressions can't silently smuggle
  themselves through.
- The **docs-sync miss + force-retag recovery** in `35f9671` +
  v1.7.2 tag re-pointing demonstrates the release safety net works:
  CI failed → noticed within 30 seconds → docs regenerated → tag
  re-pointed → all subsequent runs green. The CHANGELOG entry
  explicitly documents this recovery so anyone reading release notes
  for v1.7.2 understands the tag-was-moved event.
- **Process improvement opportunity captured in CHANGELOG**: future
  release scripts could auto-run `scripts/generate_docs.py --check`
  as the last gate before tag-push, catching this class of mistake.
  (Not done in this cycle — see §4.3.)

**What could be better** (none critical):

- The `# ty: ignore[invalid-method-override]` ignore on
  `on_call_tool` will become unused once `ty` lands generic-method-
  override support. There's no follow-up issue / TODO entry to
  re-evaluate when that ships. **Suggest adding a `TODO(ty-issue:
  generic-param-override)` marker** so a future `rg "TODO(ty-issue"`
  finds it. **Minor housekeeping.**
- The categorization table above could live in the source tree (e.g.,
  `docs/tools/ty-ignore-rationale.md`) rather than just CHANGELOG —
  CHANGELOG entries get long and may not be discoverable when
  contributors are deciding whether to add a new ignore. **Suggest
  parking lot for R203.**

### 2.4 R202 — `aiia_sse_emit_by_type_total{event_type="..."}` counter

**Commit**: `f441ba8`
**Files**:

- `src/ai_intervention_agent/web_ui_routes/system.py` (+35 lines in
  `_render_prometheus_metrics`; +7 lines in `SystemRoutesMixin`
  metrics endpoint description docstring)
- `tests/test_sse_emit_by_type_counter_r202.py` (NEW, +375, 12 cases
  / 5 invariant classes + 4 subtests)
- `CHANGELOG.md` (+60-line entry under `[Unreleased]`)

**What it does**:

R198 (Cycle 7) introduced `_SSEBus._emit_by_type: Counter[str]` that
tracks per-event-type emit counts. `stats_snapshot()` exposed it as
`"emit_by_type"` in the JSON snapshot, but the Prometheus exposition
endpoint `/api/system/metrics` rendered **only** the aggregate
`aiia_sse_emit_total`. R202 closes the gap by adding a new metric
family `aiia_sse_emit_by_type_total{event_type="..."}` rendered from
the same `_emit_by_type` snapshot, **preserving** the existing
unlabeled `aiia_sse_emit_total` for backward compatibility.

**Design choice — methodology B (independent new metric) vs A (label
on existing metric)**:

A would have been "add `event_type` label to `aiia_sse_emit_total`"
— but Prometheus exposition format disallows the same metric name
serving both labelled and unlabelled series across scrapes. Strict
parsers (VictoriaMetrics, Cortex, recent Prometheus) reject with
`inconsistent labels for metric family`; Grafana historical lines
would break at the upgrade boundary. B's overhead (one extra family,
N series where N == event_type count) is negligible (~64 bytes per
scrape at current N=4).

**What's good**:

- The decision rationale is documented **in the source** (35-line
  block-comment in `_render_prometheus_metrics`) and **in the
  CHANGELOG** entry (with method-A failure mode explicitly cited).
  Future contributors evaluating "should we add labels to existing
  metric X" can find the precedent immediately.
- **AST guard** is the standout test: `TestSseEmitCounterLockColocation`
  parses `task.py`, locates `_SSEBus.emit`, and asserts that
  `self._emit_total += 1` **and** `self._emit_by_type[event_type]
  += 1` both live in the **same** `with self._lock:` block. This is
  the source-level guarantee behind the runtime invariant
  `sum(by_type series) == emit_total`. Pattern reused from R197's
  `TestSourceLevelLatencyPathColocation` (CR#20 §2.2 highlighted
  this pattern as one of the strongest in the repo).
- **Concurrent stress test** (`test_sum_invariant_holds_under_
  concurrent_emits`): 8 threads × 50 emits with `threading.Barrier`
  for synchronized start. The sum invariant holds (verified) — but
  more importantly, the AST guard ensures it'll **continue** to hold
  even if a future refactor moves either accumulator outside the
  lock, which the runtime test alone can't reliably catch (race
  window is ~5 bytecode instructions wide).
- **Backward compat tests explicitly verify** old `aiia_sse_emit_total`
  remains unlabeled and the two families don't share HELP/TYPE
  lines — R187's "second TYPE for metric" trap (from CR#18 §2.2) is
  guarded against again.
- The R198 4-event subtests use `with self.subTest(event_type=...)`
  so a failure points at the specific event without rerunning the
  whole suite — minor but nice DX detail.

**What could be better** (none critical):

- The `_emit_by_type` accumulator has no upper bound on event-type
  cardinality. If a future bug or test artifact emits 10,000 unique
  event_type strings, the counter family grows unbounded (and so
  does the Prometheus exposition payload). **R198 schema already
  acts as a soft cap** (AST guard fails CI if you emit a non-
  registered type), but `oversize_drop` is the documented exception
  path that allows dynamic types. **Suggest follow-up F-202-1**: add
  a hard limit (e.g., 100 distinct types in `_emit_by_type`) and a
  one-time WARN log when the cap is hit. **Important defensive
  R203+ candidate.**
- The exposition output sorting is alphabetical (`sorted()`), but
  the test only verifies "current output is sorted" — there's no
  test that explicitly fails if a future change switches to insertion
  order. **Minor.**

### 3.2 Process: tag-was-moved during release

The force-retag of `v1.7.2` from `36222a3` to `35f9671` (5 minutes
after initial push, after CI revealed the docs-sync miss) is
unusual but **the right call** under the circumstances:

- The retag happened within 5 minutes of the initial push — no
  downstream consumer had time to fork/clone.
- The retag was documented in the new tag's annotation message,
  explicitly mentioning the docs-sync recovery.
- The CHANGELOG [1.7.2] entry also documents the recovery.
- Alternative (release v1.7.3 with just the docs sync) would have
  doubled release noise and left v1.7.2's CI permanently red — bad
  signal for anyone auditing the release page.

This is the **second** time in project history the project has
needed to retag (cf. CR#15-era retag of v1.x.y for a similar reason).
A formalized "release safety checklist" docs page would prevent the
class of mistakes that motivated the retags both times.

### 3.3 Type-safety as a slow-burn investment

`ty 0.0.34`'s 38 diagnostics surfaced **2 real bugs** out of 38:

- `enhanced_logging.get_current_log_level` had a wrong return-type
  annotation that `mypy` apparently allowed through. Concrete impact
  was nil (callers don't introspect the type), but the annotation
  was misleading anyone reading the source.
- `test_hot_reload_network_security_r193.py` called
  `ConfigManager.get_web_ui_config()` — a method that **never
  existed**. The test would have errored at runtime, but only
  reached an except-clause that swallowed it. Functional damage:
  the test's intended cache-filling path was a no-op (the test
  was effectively testing nothing in that branch). Fixed to
  `get_section("web_ui")`.

The 36 non-bug findings are noise, but the 2 real bugs **would not
have been caught by any other tooling** the project uses (ruff
doesn't check semantic types; mypy 1.x was lenient about both;
pytest only catches behavior that's actually exercised). At a 2/38
yield, the per-diagnostic effort is high — but for a 1-time audit
on a stricter checker version, the cost is amortized over years.

**Lesson for future cycles**: when bumping a static-analysis tool
version, expect 5-10% real-bug yield and budget for full audit.

### 3.4 R202 completes the R198 → R199 → R200 → R202 loop

The Cycle 7 → Cycle 8 arc has a clean narrative:

- **R198** built the SSE event schema registry (typed contracts).
- **R199** built the API-token-info admin endpoint (read-side
  observability).
- **R200** closed the token state-consistency gap (write-side
  hygiene).
- **R202** rendered the per-event-type emit counts to Prometheus
  (closing the R198 observability loop).

What's missing to complete the observability arc:

- **R203 candidate**: `aiia_token_age_seconds` gauge that mirrors
  `GET /api/system/api-token-info`'s `age_seconds` field, allowing
  alertmanager rules like `aiia_token_age_seconds > 90 * 86400`
  without scraping a separate JSON endpoint.
- **R204 candidate**: schema-driven SSE payload validation toggle
  (F-198-1 from CR#20 §4.2). With R198's schema registry already in
  place, adding `validate=True` to `_sse_bus.emit` for dev/test
  environments would catch payload-key drift before prod.

These are **R203+** candidates, not required by this cycle.
