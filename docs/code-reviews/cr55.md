### §2.3 `fe6a616` — R280 submitFeedback entry-side null-guard (R268/R279 spillover)

**Approach**: R268 修了 finally；R279 同 class fix `settings-manager.js`；
R280 进一步推广到 **try 块入口 + success path**。

**触发场景**:

`submitFeedback()` 可由 **Ctrl/Cmd+Enter 键盘快捷键**触发 (`app.js`
line 1527)。这意味着 DOM 不在视图时也能调用：

1. 任务切换动画中 (multi_task.js 渲染中间态)
2. SSE 推送新页面渲染中 (feedback-text / submit-btn 节点重建)
3. auto-resubmit timeout 触发 `showNoContentPage()` 移除编辑器

旧代码 `getElementById("feedback-text").value.trim()` 抛 TypeError 被
catch 翻译成 user-visible "网络错误"。**严重误导**：用户开始检查网络/
防火墙/代理，实际是 stale DOM。

**R280 修复 (3 处)**:

1. `feedback-text` null → silently abort + `console.warn` 留 trace
2. `submit-btn` (try entry) null check → best-effort UI loading state
3. `feedback-text` (success path clear) null check → DOM 切走时无需清空

**Design decisions**:

- feedback-text null → early return: 用户根本不在反馈视图，不发空字符串
  到 `/api/submit` 污染后端日志
- submit-btn null → 不 early return: **反馈不能因 UI loading state 缺失
  就丢失**。fetch 必须执行，UI loading 是 best-effort

**R280 invariant (6 tests)**:

- 4 R280 function-specific
- 1 R280 anchor 注释
- 1 R268 + R279 preserve sanity

**新增 utility**: `_strip_js_comments()` helper 剥离 JS 注释，让 invariant
regex 不误匹配 doc-comment 里的 bad-pattern 示例 (R268 doc comment 提到
了 `getElementById("feedback-text").value.trim()` 作为反例，invariant 不
能误匹配)。

**Pattern evolution**:

| Cycle | R# | Side | Subject |
|-------|-----|------|---------|
| cycle-22 | R268 | finally | submitFeedback() |
| cycle-25 | R279 | finally | openConfigFileInIde() |
| cycle-25 | R280 | try entry + success path | submitFeedback() (再次) |

R268/R279 单点修；R280 是 same function 第二次 fix (扩展到不同 side)。
