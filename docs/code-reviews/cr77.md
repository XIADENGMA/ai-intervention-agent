### 2.3 Release infrastructure 强化 (R416)

历史教训: v1.7.5 release 因 `pyproject.toml` 已 bump 但 `package.json` 未
同步 → release 失败 (`docs/release-checklist.md:71`). 之前的 R341 / R382 /
R410 都是**具体版本锁** (硬编码 "1.8.1"/"1.8.2"/"1.8.3"), 每次 release 必
须修改。

R416 是**结构化 invariant** — 不锁具体版本号, 只锁 "所有来源必须相等且
格式合法", release 时不需要修改 R416 (除非新增 version source):

| 维度    | R#                | 锁定内容             |
| ------- | ----------------- | -------------------- |
| 具体版本号 | R341 / R382 / R410 | "1.8.1" / "1.8.2" / "1.8.3" 硬编码 |
| 结构一致性 | R416              | **5 个来源严格相等 + semver 格式** |

R416 与具体版本锁互补: **release 时做对 + release 间不退** 双层保护。

**R416 锁定的 5 个来源**:
1. `pyproject.toml`
2. `CITATION.cff`
3. `package.json`
4. `packages/vscode/package.json`
5. `package-lock.json` (root + packages."" 自引用 2 处)

## 5. 跨 cycle 风险与机会

### 风险 A: R227 catalogue regex 与 invariant filename convention 不严格匹配

R412/R414/R418 文件名是 `test_feat_<topic>_<r>NNN.py` (省略 `_invariant_`),
但 R227 catalogue 严格要求 `_invariant_rNNN.py` 模式。结果: R412/R414/R418
不被 R231 视为 "需要 catalogue 的 invariant", 静默通过 — 但它们也是真正
的 invariant test。

**建议**: cycle-48 修复 R227 regex 或统一 filename convention. 两个选项:
- 选项 1 (放宽 regex): R227 改用更宽松的 `_r(\d+)\.py$` (任何 `_rNNN.py`)
- 选项 2 (严格 filename): rename 所有 cycle-43+ 的 invariant test 文件加入
  `_invariant_` 前缀

推荐选项 1, 因为 cycle-43+ 已经累积 ~15 个 invariant 用新 convention, rename
成本高且无明显收益。优先级 P1 (战术 hygiene)。

### 风险 B: meta-invariant 模式只有 2 应用, 升级 v3.11 候选未明确

R414 (negative validation) + R418 (ratchet) 共 2 应用, 距离 ≥3 应用工业化
阈值还差 1. 如果 cycle-48 不继续应用 meta-invariant 模式, 第 15 个维度
"invariant meta-protection" 启动可能延迟。

**建议**: cycle-48 实施 doc-parity ratchet (apply ratchet pattern 到 R408
等) 作为 meta-invariant 3rd app, 推进到完全工业化阈值。优先级 P2。

### 风险 C: OpenAPI property description coverage 75% 不能完美锁定 nested schema

R412 的 coverage 计算包括 nested properties (deeply nested object schema),
但实际 75% 主要来自顶层 + 一层 nested. nested 越深, coverage 越难提升。

**建议**: cycle-48 R412 加 path-based exemption — nested >3 layer 的
property 不计入 coverage 分母, 让 ratchet 更准确反映实际可访问字段。
优先级 P3 (细化)。

### 机会 D: R412 ratchet 模式可推广到 doc-parity / i18n / Pydantic 等其他 ratchet 型 invariant

R412 + R418 的 "low water mark + actual improvement + ratchet up" 三段
模式可以推广到所有 ratchet 型 invariant:

- doc-parity coverage ratchet (R335/R340/R346/R394/R400/R408 → 加 ratchet 维度)
- i18n key coverage ratchet (R350/R353/R366/R374 → 加 ratchet 维度)
- Pydantic validator coverage ratchet (R380-R402 → 加 ratchet 维度)
- security header strict 程度 ratchet (R376/R386/R390 → 加 ratchet 维度)

**建议**: cycle-48 / cycle-49 至少应用一次到 doc-parity (R335 等), 推 doc-
parity 从 6 应用 → 6 应用 + ratchet, 形成 "应用数 × 改进强度" 二维度量。
优先级 P2 (战略)。

### 机会 E: v3.10.3 sub-pattern 候选 — error response schema parity

cr75 §6 推荐 v3.10.3 为 "error response schema parity — 4xx/5xx 必须有
`schema` 字段". 当前 codebase 4xx/5xx response 多数只有 `description` 没

**建议**: cycle-48 启动 v3.10.3, 也作为 v3.10 系列第 3 个 sub-pattern. 优
