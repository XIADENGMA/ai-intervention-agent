### §2.2 `06e6992` — R275 resetFeedbackConfig confirm (user TODO t24-3)

**File**: `static/js/settings-manager.js` +
`locales/{en,zh-CN,zh-TW,_pseudo/pseudo}.json` +
`tests/test_feat_reset_feedback_confirm_r275.py` (new) + 6 个 minified/
compressed siblings.

**Change**: `resetFeedbackConfig()` 头部加 `window.confirm` 判断，pattern
与 `resetSettings()` (R231 / cycle-13) 100% 对齐:

```javascript
const confirmMsg = _tl(
  "settings.resetFeedbackConfirm",
  "Reset server-stored feedback config ... All collaborators on this server will see the change via SSE broadcast.",
);
if (
  typeof window !== "undefined" &&
  typeof window.confirm === "function" &&
  !window.confirm(confirmMsg)
) {
  console.debug(_tl("settings.resetFeedbackCancelled", "..."));
  return;
}
// 原 fetch 逻辑
```

4 个 locale 新增 `settings.resetFeedbackConfirm` +
`settings.resetFeedbackCancelled`，强调"同服务器所有协作者通过 SSE
广播收到变更"语义。

**Rationale**: 破坏性等级反转 audit。项目 2 个 reset button:
- `#reset-settings-btn` → `resetSettings()` — 影响当前浏览器 localStorage
- `#reset-feedback-config-btn` → `resetFeedbackConfig()` — 影响 server
  config.toml + SSE 广播到**所有协作者**

(2) 破坏性等级实际**严格 ≥** (1)：影响范围从单浏览器扩展到全服务器。
但 (1) 已有 R231 confirm，(2) 仍是"一键直接重置 + fetch"。用户 TODO
"重置设置点击后应该需要二次确认" 字面是 (1)，但等同强度的 (2) 必须
同步加 confirm 才闭环 UX。

**Methodology v3.4 pattern 首推**: R275 包含 **meta-lint invariant** —
"所有 `reset*()` 方法必须有 `\"settings.*Confirm\"` i18n key + `window.
confirm(` 调用"。未来加新 reset 按钮时自动拦截缺 confirm 的破坏性操作。
这是 cr53 §6 提到的 v3.4 候选模式的第一个生产实现。

**Risk**: **low**. typeof guard 保 backward compat（自动化测试 /
sandbox 环境 fallback 到 "no confirm, direct reset"，与 resetSettings
对齐）。confirm 在 fetch 之前，cancel 路径 return 不调后端。

**Test surface**: 13 new invariants in
`test_feat_reset_feedback_confirm_r275.py`:
- resetFeedbackConfig 必须 (5): _tl key / typeof guard / cancelled debug
  / cancel return / confirm 在 fetch 之前
- resetSettings 原 confirm 不被回归破坏 (1 sanity)
- 4 locale × 2 key (4) + 翻译非空 (2)
- **Meta-lint** (1): 所有 reset*() 方法必须有 "settings.*Confirm" i18n
  key + window.confirm( 调用

### §2.3 `55e3d66` — R276 image upload limits invariant (cr53 §5 mining-14)

**File**: `tests/test_feat_image_upload_limits_consistency_r276.py` (new) +
`CHANGELOG.md`.

**Change**: 不改源码，只加 invariant test 锁定 4 对常量等价:

| Constant | Frontend (image-upload.js) | Backend (_upload_helpers.py) |
|----------|---------------------------|------------------------------|
| 单文件大小 | `MAX_IMAGE_SIZE = 10 * 1024 * 1024` | `MAX_FILE_SIZE_BYTES = 10 * 1024 * 1024` |
| 单请求数量 | `MAX_IMAGE_COUNT = 10` | `MAX_IMAGES_PER_REQUEST = 10` |
| 累计字节预算 | (无前端 cap) | `MAX_TOTAL_UPLOAD_BYTES = 100 * 1024 * 1024` |

只有 `_upload_helpers.py` 注释里 "客户端 (MAX_IMAGE_COUNT / MAX_IMAGE_SIZE)
同值" 作为软约定。任一方向 silent drift 都是糟糕 UX:

- backend 改 20MB 但 frontend 不改 → frontend 误报 "过大"
- frontend 改 20MB 但 backend 不改 → frontend pass / backend 413 Payload
  Too Large (用户填了很久的反馈一键消失)

**Rationale**: R271 揭示 OpenAPI schema 4-way drift 之后，cr53 §5 mining-14
列为 follow-up "distributed source-of-truth audit"。R276 是首个真发现 —
image upload 限制有 2 处 hardcode 同值，但无任何 mechanism 防止 drift。

**Risk**: **zero**. 测试新增（源码零变更），现有值都已对齐，invariant
立即 green。

**Test surface**: 4 new invariants:
- `MAX_IMAGE_SIZE` ≡ `MAX_FILE_SIZE_BYTES` (字节)
- `MAX_IMAGE_COUNT` ≡ `MAX_IMAGES_PER_REQUEST` (个数)
- `MAX_TOTAL_UPLOAD_BYTES` ≥ `MAX_FILE_SIZE_BYTES × MAX_IMAGES_PER_REQUEST`
  (sanity — 累计预算必须 ≥ 单文件最大 × 数量最大)
- `MAX_IMAGE_COUNT` / `MAX_IMAGE_SIZE` 注释锚点保留 (文档查找路径)

跨语言 const parser (JS const + Py annotated assign) 是新 invariant
helper pattern，可复用于未来更多跨端 constant drift detection。

### §2.4 `fc6eea7` — R277 settings-panel backdrop close (cr53 §5 mining-15)

**File**: `static/js/settings-manager.js` +
`tests/test_feat_settings_backdrop_close_r277.py` (new) + 4 个 minified/

**Change**: `settings-manager.js::init()` 添加 backdrop click handler:

```javascript
const settingsPanel = document.getElementById("settings-panel");
if (settingsPanel && !settingsPanel.dataset.aiiaBackdropWired) {
  settingsPanel.addEventListener("click", (e) => {
    if (e.target === settingsPanel) {
      this.hideSettings();
    }
  });
  settingsPanel.dataset.aiiaBackdropWired = "1";
}
```

**Rationale**: 项目 3 个 `role="dialog"` 元素:

| Dialog | ESC | Close button | Backdrop click |
|--------|-----|-------------|----------------|
| `#settings-panel` | hideSettings (R272) | hideSettings | **缺失** ❌ → ✅ R277 |
| `#image-modal` | closeImageModal (R272) | closeImageModal | closeImageModal |
| `#code-paste-panel` | closeCodePasteModal | closeCodePasteModal | closeCodePasteModal |

3 个 dialog 中 1 个 (settings-panel) 缺 backdrop click 关闭，inconsistent
UX：用户在 image-modal / code-paste 学到"点 backdrop 关"，切到 settings
点 backdrop 没反应 → 困惑必须找 X 或按 ESC。

settings 是 auto-save，无 dirty form 概念，backdrop click 关闭符合
Material Design dismissible dialog + iOS sheet swipe-down 等标准 UX。

**3 patterns 复用 in 1 commit**:
- `e.target === panel` 守卫 — 同 image-modal/code-paste 模式
- `dataset.aiiaBackdropWired` guard — R265 N+1 pattern 复用
- delegate to `hideSettings()` — R272 canonical close pattern

**Risk**: **low**. dataset guard 防重复（首次绑定后不再 add listener）；
delegate to hideSettings 走完整 cleanup chain (R272 已验证)。

**Test surface**: 8 new invariants:
- settings-panel addEventListener click + e.target === settingsPanel 守卫
  + delegate to this.hideSettings() + dataset.aiiaBackdropWired guard +
  R277 注释 (5 R277 specific)
- Cross-modal sanity (3): image-modal/code-paste/settings 三 dialog
  backdrop 都 delegate to canonical close (R272 不被回归破坏)
